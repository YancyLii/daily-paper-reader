## 摘要
基于 ReAct 的大语言模型工具调用自主智能体表现出三个局限性：串行延迟、上下文二次方增长，以及易受提示词注入和幻觉的影响。近期的工作趋向于将规划与执行分离，但在每种情况下，模型仍与执行机制耦合。我们为 LLM 智能体引入了一种系统级抽象，将智能体工作流的执行与 LLM 推理层解耦。我们定义了两个一级抽象：（1）意图门控执行（IGX），一种在执行时强制执行意图的安全范式；（2）执行内核，负责管理调度、工具分发、依赖解析、故障处理和安全。在 KAIJU 中，LLM 预先进行规划，通过依赖感知的参数注入，乐观地并行调度工具。工具通过 IGX 进行授权，基于四个独立变量：范围（scope）、意图（intent）、影响（impact）和许可（clearance，即外部审批）。KAIJU 支持三种自适应执行模式（Reflect、nReflect 和 Orchestrator），提供逐渐细粒度的执行控制，适用于复杂的调查、深度分析或研究。针对 ReAct 基准的实证评估表明，由于规划开销，KAIJU 在简单查询上存在延迟惩罚，在适度复杂度下趋于收敛，而在需要并行数据采集的计算查询上具有结构性优势。除了延迟之外，这种分离还强制执行了行为保证，这是 ReAct 仅通过提示词无法实现的。代码可在 https://github.com/compdeep/kaiju 获取。

## Abstract
Tool-calling autonomous agents based on large language models using ReAct exhibit three limitations: serial latency, quadratic context growth, and vulnerability to prompt injection and hallucination. Recent work moves towards separating planning from execution but in each case the model remains coupled to the execution mechanics. We introduce a system-level abstraction for LLM agents which decouples the execution of agent workflows from the LLM reasoning layer. We define two first-class abstractions: (1) Intent-Gated Execution (IGX), a security paradigm that enforces intent at execution, and (2) an Executive Kernel that manages scheduling, tool dispatch, dependency resolution, failures and security. In KAIJU, the LLM plans upfront, optimistically scheduling tools in parallel with dependency-aware parameter injection. Tools are authorised via IGX based on four independent variables: scope, intent, impact, and clearance (external approval). KAIJU supports three adaptive execution modes (Reflect, nReflect, and Orchestrator), providing progressively finer-grained execution control apt for complex investigation and deep analysis or research. Empirical evaluation against a ReAct baseline shows that KAIJU has a latency penalty on simple queries due to planning overhead, convergence at moderate complexity, and a structural advantage on computational queries requiring parallel data gathering. Beyond latency, the separation enforces behavioural guarantees that ReAct cannot match through prompting alone. Code available at https://github.com/compdeep/kaiju

**KAIJU** 的核心意义在于提出了一种**系统级抽象**，将智能体的“推理层”与“执行层”彻底解耦，通过一个“执行内核”来管理复杂的工具调度与安全校验。

#### 2. 论文提出的方法论
KAIJU 引入了两个关键的一级抽象：

*   **执行内核（Executive Kernel）**：
    *   **核心思想**：将任务分解为有向无环图（DAG）。LLM 仅负责初始规划（Planner），内核负责依赖解析、并行调度、参数注入和错误恢复。
    *   **参数引用机制（Param Refs）**：允许在规划阶段定义数据流（如步骤 B 使用步骤 A 的输出字段），在执行时动态注入具体值，实现非阻塞的并行化。
    *   **三种自适应模式**：
        1.  **Reflect**：在依赖波次（Waves）之间进行反思和重新规划。
        2.  **nReflect**：每完成 N 个节点后进行周期性检查。
        3.  **Orchestrator**：为每个节点配备轻量级观察者，实现最高精度的实时控制。

*   **意图门控执行（IGX）安全范式**：
    *   **四变量校验**：工具执行前必须通过四个独立维度的审核：**范围（Scope）**（允许列表）、**意图（Intent）**（调用者权限等级）、**影响（Impact）**（工具危险程度，如只读 vs 删除）、**许可（Clearance）**（外部 HTTP 端点授权）。
    *   **结构化隔离**：安全决策在内核中完成，拒绝信息不回传给 LLM，防止攻击者通过试探性对话绕过安全策略。

*   **性能拐点**：在简单任务中，KAIJU 因规划开销略慢于 ReAct；但在复杂和计算型任务中，KAIJU 的延迟显著降低（如复杂任务 9.5s vs 28.9s），且成功率更高。

*   **行为保障**：KAIJU 强制执行规划好的工具路径，防止了 ReAct 智能体在遇到困难时“偷懒”转而使用模型幻觉（先验知识）回答问题的倾向。
*   **安全性**：IGX 机制在不增加显著延迟的前提下，提供了比提示词工程更可靠的结构化安全保障。

#### 7. 优点
*   **架构解耦**：将 LLM 视为无状态的推理资源，由内核管理状态，这种设计更符合传统操作系统的稳健性原则。
*   **并行效率**：通过 DAG 和参数注入机制，极大提升了需要大量数据采集任务的执行速度。
*   **安全创新**：IGX 的四变量模型和“不回传拒绝原因”的设计，有效封堵了针对智能体的自适应攻击路径。

#### 8. 不足与局限
*   **规划依赖**：系统的上限取决于 Planner 的规划质量。如果初始规划严重失误，虽然有反思循环补偿，但会增加额外的延迟和成本。
*   **简单任务开销**：对于一句话就能回答的简单问题，KAIJU 的规划步骤显得冗余。
*   **动态影响评估**：对于 Shell 等通用工具，其“影响（Impact）”等级的自动分类仍依赖模式匹配，可能存在误判风险。
*   **应用限制**：目前主要针对工具密集型任务，对于纯逻辑推理或创意写作类任务，该架构的优势不明显。
