## 摘要
大型语言模型（LLM）审查者越来越多地用于拉取请求（PR）工作流中，它们的批准有助于决定哪些代码被合并到仓库中。这引发了一个静态漏洞检测或代码生成基准无法回答的问题：当攻击者同时控制代码更改和附带的PR文本时，自动化审查者能否拒绝恶意贡献？我们推出了SEVRA-BENCH（审查代理中漏洞的社会工程学基准），这是一个衡量自动化审查者批准此类对抗性拉取请求频率的基准。SEVRA-BENCH中的每个恶意PR都源自真实的项目提交，该提交之前修复了通用漏洞披露（CVE）数据库中列出的一个漏洞。我们自动反转该修复，恢复原始的易受攻击代码，并将其作为拉取请求提交，包装在15种社会工程框架之一中，这些框架变化了所声称的内容、支持证据、传达的紧迫性、先前批准的信号以及对权威的诉求。SEVRA-BENCH包含1,062个恶意PR，这些PR来自2025年通用弱点枚举（CWE）前25名中前10个条目的CVE关联修复。在现实场景中，我们评估了8个当前的LLM作为代码审查代理，处理引入先前在公开披露中报告的漏洞的PR。我们的结果揭示了闭源和开源模型之间在安全能力方面的显著差距。我们希望SEVRA-BENCH能成为提升开源模型并缩小这一差距的宝贵资源。

## Abstract
Large language model (LLM) reviewers are increasingly used in pull-request (PR) workflows, where their approvals help decide which code is merged into a repository. This raises a question that benchmarks for static vulnerability detection or code generation do not address: can an automated reviewer reject a malicious contribution when the attacker controls both the code change and the accompanying PR text? We introduce SEVRA-BENCH (Social Engineering of Vulnerabilities in Review Agents), a benchmark that measures how often an automated reviewer approves such adversarial pull requests. Each malicious PR in SEVRA-BENCH is built from a real project commit that previously fixed a vulnerability listed in the Common Vulnerabilities and Exposures (CVE) database. We automatically invert that fix to restore the original vulnerable code and submit it as a pull request wrapped in one of 15 social-engineering framings, which vary the claims made, the supporting evidence, the urgency conveyed, signals of prior approval, and appeals to authority. SEVRA-BENCH contains 1,062 malicious PRs drawn from Common Vulnerabilities and Exposures (CVE)-linked fixes across the top 10 entries of the 2025 Common Weakness Enumeration (CWE) Top 25. In a realistic setting, we evaluate 8 current LLMs as code review agents on PRs that introduce vulnerabilities previously reported in public disclosures. Our results reveal a sharp gap in security capabilities between closed- and open-source models. We hope SEVRA-BENCH will serve as a valuable resource for advancing open-source models and narrowing this gap.

### 6. 论文的主要结论与发现

*   **存在显著的“能力鸿沟”**：闭源前沿模型（Claude Opus 4.7, GPT-5.5）展现出近乎完美的拒绝率（平均 >95%），而开源及较弱的闭源模型平均拒绝率仅为 35%-53%。这个差距高达约45个百分点。
*   **开源模型极易受叙事操纵**：同一个恶意代码变更，仅通过更换 PR 描述框架，就能让开源模型的拒绝率产生巨大波动（如 DeepSeek V4-Flash 从最高的100%降至18.2%）。而闭源模型在不同框架下表现稳定。
*   **拒绝的安全性动机不足**：虽然部分模型能拒绝恶意PR，但其拒绝理由（SRR指标）并不总是基于安全考虑。这暗示模型可能因为其他代码规范或随机因素而拒绝，而非真正识别出漏洞。
*   **安全能力与推理效率不完全正相关**：如 Claude Opus 4.7 在实现最高拒绝率的同时，消息交互次数和消耗的 Token 数反而比 GPT-5.5 更少，表明其在安全审查方面更高效。

### 7. 优点

*   **问题定义新颖且现实**：精准定位了“审查代理的社会工程学鲁棒性”这一未被充分研究的高影响力问题，将安全评估从静态检测扩展到动态的、以叙述为背景的决策过程。
*   **基准设计严谨且解耦**：
    *   **真实漏洞来源**：通过反转真实 CVE 补丁，避免了模型生成漏洞代码可能引入的偏差。
    *   **变量控制**：将代码变更（`diff`）和叙事框架（`ϕ`）完全解耦，实现了“控制变量”式的评估，能够清晰测量单一因素（叙事）对决策的影响。
    *   **交互式环境**：部署在真实的Gitea仓库中，审查过程是全交互式的，而非对一段静态文本的问答，更贴近实际工作流程。
*   **评估维度全面**：不仅看是否拒绝（Refusal Rate），还分析了为何拒绝（Security Reason Rate），以及不同漏洞类型和攻击框架下的表现差异，提供了多层次的洞察。

### 8. 不足与局限

*   **数据污染风险**：基准使用的漏洞来自公开CVE，闭源模型在训练时可能已经“见过”这些漏洞，导致其表现被高估，无法完全代表其对零日漏洞的泛化能力。
*   **审查环境过于简化**：评估仅限于孤立的代码审查任务，缺乏现实世界中影响人类审查者的关键上下文信息，如提交者历史、项目信誉、团队讨论等。实际审查是在更丰富的社会技术背景下进行的。
*   **叙事生成的单一性**：虽然设计了15种框架，但具体的 PR 文本是由一个特定的 LLM（Claude Opus 4.6）生成的，这可能引入了该模型特有的语言风格或偏好，未来的攻击者可能使用更具多样性和欺骗性的措辞。
*   **静态、单次攻击模型**：实验将每次 PR 审查作为独立、单轮事件。现实中，攻击者可能根据审查者的反馈进行多轮、自适应的叙事调整，这种动态博弈的威胁模型未被覆盖。
*   **良性PR可能过于简单**：用于对照的良性修复PR可能过于“清晰”，低估了在遇到更复杂、模棱两可的良性变更时模型可能产生的“误拒”率。
