## 摘要
基于大语言模型（LLM）的漏洞检测器在识别内存安全缺陷和可通过既定安全属性表达的漏洞类别方面展现出前景。然而，逻辑漏洞带来了不同挑战，因为其识别需要推断特定于应用的安全不变式以及关于预期行为的隐含假设。即便是前沿的智能体模型也常常力不从心，因为这些不变式往往是隐式的，深藏于无关代码之中。受此差距启发，我们提出了Antaeus，一个将LLM推理立足于仓库级代码上下文中以检测逻辑漏洞的框架。Antaeus采用仓库级流水线，结合了函数优先级排序、基于上下文的推理、比较验证和结构化报告。它利用轻量级的全仓库安全信号对函数进行排序，将昂贵的LLM分析导向相关代码，从而减少调用次数、成本和分诊工作量。对于每个优先排序的函数，Antaeus将局部代码上下文与应用程序功能、安全资源和信任边界的仓库级视图相结合，从而能够推理该函数在更广泛应用中如何执行，而非仅仅作为孤立片段。Antaeus识别安全敏感汇聚点，推导安全执行的安全条件，并检查这些条件在局部是否得到满足。候选发现会经过比较验证，剔除那些反映项目整体规范而非明显违规的问题。最后，Antaeus报告汇聚点、违反的安全条件以及证据，使发现结果可操作且可追溯。我们在28个具有已确认逻辑漏洞的仓库上评估了Antaeus，并将其与函数级模型和智能体模型进行了比较。Antaeus检测并解释了15个漏洞，在相当的令牌用量和成本下表现优于基线方法。

## Abstract
LLM-based vulnerability detectors have shown promising results in identifying memory-safety bugs and vulnerability classes whose violations can often be expressed through established security properties. Logic vulnerabilities, however, pose a different challenge, as their identification requires inferring application-specific security invariants and implicit assumptions about intended behavior. Even frontier agentic models struggle because these invariants are often implicit and buried among unrelated code. Motivated by this gap, we present Antaeus, a framework for detecting logic vulnerabilities that grounds LLM reasoning in repository-level code context. Antaeus follows a repository-scale pipeline combining function prioritization, context-grounded reasoning, comparative validation, and structured reporting. It ranks functions using lightweight repo-wide security signals, directing costly LLM analysis toward relevant code and reducing calls, cost, and triage effort. For each prioritized function, Antaeus combines local code context with a repository-level view of the application's functionality, security resources, and trust boundaries. This enables reasoning about how the function is executed within the broader application rather than as an isolated snippet. Antaeus identifies security-sensitive sinks, derives safety conditions for safe execution, and checks whether they are locally satisfied. Candidate findings undergo comparative validation, pruning concerns that reflect project-wide norms rather than distinctive violations. Finally, Antaeus reports sinks, violated safety conditions, and evidence, making findings actionable and traceable. We evaluate Antaeus on 28 repositories with confirmed logic vulnerabilities and compare it against function-level and agentic models. Antaeus detects and explains 15 vulnerabilities, outperforming baselines with comparable token usage and cost.

## 1. 论文的核心问题与整体含义

- **核心问题**：检测仓库级的逻辑漏洞（Logic Vulnerabilities），如不当访问控制（CWE-284）和敏感信息泄露（CWE-200），是自动漏洞检测领域长期存在的难题。
- **背景与挑战**：
    - 逻辑漏洞源于未能正确执行**应用特定的安全不变式**（Security Invariants），而非通用的内存破坏或数据流污染模式。其检测需要推断程序“本应”的行为，而非仅仅匹配已知的危险代码模式。
    - 现有的 LLM 方法（包括函数级分析和能浏览仓库的智能体模型）在此任务上表现不佳。主要障碍在于：识别漏洞所需的安全不变式和证据往往是**隐含的**、**分布在整个代码库中**的，且埋藏在大量不相关的代码里。
    - 简单的函数级分析上下文不足，而让智能体模型自由探索整个仓库则成本高昂、容易分心，且可能遗漏关键区域。
- **整体含义**：论文提出，仓库级逻辑漏洞检测的关键在于**上下文扎根**——即将模型的推理锚定在关于项目如何定义主体、资源、信任边界和安全敏感操作的具体证据上。这本质上是一个**不变式恢复问题**，而非单纯的检索或推理问题。

## 2. 论文提出的方法论：Antaeus

Antaeus 是一个**五阶段仓库级流水线框架**，旨在通过结构化的上下文扎根和验证，高效利用 LLM 发现逻辑漏洞。其核心思想是通过轻量级步骤筛选和定位，再对高价值目标进行深度的上下文丰富的分析，并最终通过横向比较剔除误报。

- **阶段一：函数优先级排序**：利用**轻量级的仓库级安全信号**对函数进行排序，将昂贵的 LLM 分析聚焦于最相关的代码区域。首先通过基于关键字的启发式方法剪除明显无关的函数，然后将函数体压缩为“签名+被调用者”的极简表示，提供给一个智能体 LLM，由其基于全局视图进行相关性排序。
- **阶段二：上下文扎根**：为每个被优先分析的函数构建**双重上下文**：
    - **局部代码增强**：通过轻量级静态分析，提供函数内引用的宏、常量、类型定义以及深度为1的Callee函数声明与定义。
    - **仓库级上下文摘要**：由一个 LLM 提前为整个仓库生成一份稳定的、结构化的描述，包括：**系统目的、主体模型、受保护对象、信息输出和信任拓扑**。
- **阶段三：结构化 LLM 推理**：要求 LLM 基于扎根的上下文，对目标函数生成**结构化分析结果**：
    - **识别安全敏感汇聚点**：如影响受保护资源的操作或跨信任边界的信息发射。
    - **推导安全条件**：为每个汇聚点推断安全执行所需满足的谓词。
    - **判断局部满足情况**：检查该条件在当前函数的代码中是否**局部被满足**。
- **阶段四：比较验证**：将 LLM 发现的“疑似漏洞”（即安全条件未被局部满足的情况）与**库中相似的汇聚点**进行比较。
    - **算法逻辑**：若同一个未被满足的安全条件在多个结构相似的汇聚点（使用 UniXcoder 编码计算相似度）上反复出现（使用 all-MiniLM-L6-v2 编码计算条件文本相似度），则判定这是一个**项目规范**而非独特漏洞，并予以剪除。
    - **核心机制**：通过动态阈值（基于均值和标准差）控制比较的严格程度，其本质是**基于代码相似性的异常检测**。
- **阶段五：结构化报告**：将最终保留的、被认为是独特漏洞的发现，以包含函数信息、敏感汇聚点、违反的安全条件及其代码证据的 **JSON 格式**输出，确保结果可审计、可追溯。
