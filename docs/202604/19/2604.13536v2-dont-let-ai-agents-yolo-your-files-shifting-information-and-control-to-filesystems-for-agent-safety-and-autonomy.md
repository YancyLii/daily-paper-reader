## 摘要
AI 编程智能体直接在用户的文件系统上运行，经常导致数据损坏、文件删除和密钥泄露。目前的方法被迫在安全性与自主性之间进行权衡：不受限制的访问存在风险，而频繁的权限提示则会增加用户负担并阻碍智能体运行。为了深入理解这一问题，我们对智能体滥用文件系统的情况进行了首次系统性研究，分析了 13 个框架中的 290 份公开报告。我们的分析表明，目前的智能体对其在文件系统上产生的影响了解有限，且缺乏足够的控制能力。因此，我们主张将这些信息和控制权转移到文件系统本身。基于这一原则，我们设计了 YoloFS，这是一种采用三种技术的智能体原生文件系统。暂存（Staging）在提交前隔离所有变更，为用户提供纠正性控制；快照（Snapshots）将这种控制权扩展到智能体，使其能够检测并纠正自身的错误；渐进式权限（Progressive permission）通过最少交互的访问门控为用户提供预防性控制。为了评估 YoloFS，我们引入了一种捕获用户-智能体-文件系统交互的新方法。在 11 个具有隐藏副作用的任务中，YoloFS 使智能体在 8 个任务中实现了自我纠正，同时保持所有影响处于暂存状态且可供审查。在 112 个常规任务中，YoloFS 在保持与基准相当的成功率的同时，减少了用户交互。

## Abstract
AI coding agents operate directly on users' filesystems, where they regularly corrupt data, delete files, and leak secrets. Current approaches force a tradeoff between safety and autonomy: unrestricted access risks harm, while frequent permission prompts burden users and block agents. To understand this problem, we conduct the first systematic study of agent filesystem misuse, analyzing 290 public reports across 13 frameworks. Our analysis reveals that today's agents have limited information about their filesystem effects and insufficient control over them. We therefore argue for shifting this information and control to the filesystem itself.   Based on this principle, we design YoloFS, an agent-native filesystem with three techniques. Staging isolates all mutations before commit, giving users corrective control. Snapshots extend this control to agents, letting them detect and correct their own mistakes. Progressive permission provides users with preventive control by gating access with minimal interaction. To evaluate YoloFS, we introduce a new methodology that captures user-agent-filesystem interactions. On 11 tasks with hidden side effects, YoloFS enables agent self-correction in 8 while keeping all effects staged and reviewable. On 112 routine tasks, YoloFS requires fewer user interactions while matching the baseline success rate.

### 1. 核心问题与整体含义（研究动机和背景）
*   **研究动机**：AI 编程智能体（如 Claude Code, Cursor, Copilot）已广泛应用，但它们直接操作用户文件系统时存在严重风险，包括误删文件、损坏数据和泄露密钥。
*   **核心矛盾**：现有的防御机制（如频繁的权限提示）在“安全性”与“自主性”之间存在权衡。频繁提示会导致用户产生“审批疲劳”，从而开启“YOLO 模式”（全自动运行），使安全机制形同虚设。
*   **核心发现**：通过对 290 份公开的智能体误操作报告进行系统研究，作者发现根本原因在于**信息鸿沟**（智能体和用户看不清文件系统的实际变化）和**控制鸿沟**（缺乏有效的预防和事后纠正手段）。

### 2. 方法论：核心思想与关键技术
论文主张将信息和控制权从智能体框架下移到**文件系统层**，设计了 **YoloFS**，其核心包含三大技术：
*   **暂存区（Staging）**：
    *   **核心思想**：所有文件修改先重定向到一个隔离层，不直接修改底层文件系统。
    *   **技术细节**：采用“扁平文件存储”与“内存覆盖树（Override Tree）”分离的设计。通过“追加式目录日志（Journal）”记录所有重命名、删除和修改操作，确保变更在用户点击“提交（Commit）”前是可撤销且可审计的。
*   **快照与非破坏性旅行（Snapshots & Travel）**：
    *   **核心思想**：允许智能体在执行过程中拍摄快照，并在发现错误时“穿越”回之前的状态。
    *   **技术细节**：不同于传统的破坏性回滚，YoloFS 的“旅行”机制会保留所有分支历史。通过日志段（Segments）管理，智能体可以探索不同路径，且所有尝试过的错误路径仍可供用户事后审查。
*   **渐进式权限（Progressive Permission）**：
    *   **核心思想**：权限规则随会话动态演进，而非静态配置。
    *   **技术细节**：基于路径的层次化规则树（Rule Tree），支持 Allow, Deny, Ask, Hidden 等状态。当智能体访问未授权区域时触发按需提示，用户决策后可自动生成覆盖该子树的新规则，减少后续重复提示。

### 3. 实验设计
*   **误操作案例研究**：收集并分类了 2024-2026 年间的 290 份真实误操作报告，建立了原因分类法。
*   **隐藏副作用任务（11 个）**：设计了具有不同“不透明度”（L1 到 L∞）的任务，例如看似正常的 `make` 命令背后隐藏了删除源码的脚本。测试智能体是否能利用 YoloFS 的信息反馈实现自纠错。
*   **常规任务（112 个）**：涵盖读、写、删、跨目录操作等，测试 YoloFS 在日常使用中对用户交互频率的影响。
*   **对比方法**：对比了 Claude Code (原生), Codex, Copilot, Gemini 等主流智能体框架。
*   **性能基准**：对比了 Ext4 (原生), OverlayFS, BranchFS，测试 I/O 吞吐量、元数据延迟和快照扩展性。

### 4. 资源与算力
*   **硬件环境**：实验在配备 AMD EPYC 7302P (16核, 3GHz) 处理器、125GB DDR4 内存和 SATA SSD 的服务器上运行。
*   **模型使用**：主要使用 `claude-3.5-sonnet` 作为智能体背后的 LLM。
*   **算力说明**：论文未涉及模型训练，因此**未提及 GPU 训练算力**。实验重点在于文件系统内核模块与智能体交互的系统开销。

### 5. 实验数量与充分性
*   **实验规模**：
    *   分析了 290 份真实报告，样本量足以支撑原因分类。
    *   进行了 123 个功能性实验（11 个复杂任务 + 112 个常规任务）。
    *   进行了详尽的性能消融实验，包括不同快照数量下的延迟测试。
*   **客观性与公平性**：作者开发了一套新的基准测试框架，通过虚拟终端捕获用户-智能体交互，确保了不同框架在相同任务下的对比是客观的。实验覆盖了从简单的脚本到预编译二进制文件的多种复杂场景。

### 6. 主要结论与发现
*   **自纠错能力**：在 11 个具有隐藏破坏性的任务中，YoloFS 使智能体在 8 个任务中实现了**完全自纠错**（检测到异常并自动回滚），其余 3 个任务虽未自动回滚，但变更被成功拦截在暂存区供用户拒绝。
*   **交互效率**：在常规任务中，YoloFS 的渐进式权限模型显著减少了用户干预次数（平均每任务 0.4 次交互，远低于基准方法的 0.9-2.2 次），且保持了 99% 的任务成功率。
*   **系统性能**：YoloFS 的 I/O 开销极低（接近原生 Ext4），且快照机制具有良好的扩展性，支持数百个快照而无明显性能衰减。

### 7. 优点
*   **视角独特**：首次提出“智能体原生文件系统”概念，将安全防线从不可靠的 LLM 提示词转向可靠的内核级控制。
*   **系统性强**：从真实世界的问题调研出发，到系统设计，再到针对性基准测试的开发，形成了一个完整的闭环。
*   **实用性高**：YoloFS 作为 Linux 堆栈式文件系统实现，无需修改底层文件系统或智能体核心算法即可部署。

### 8. 不足与局限
*   **平台限制**：目前仅实现为 Linux 内核模块，对于 Windows 或 macOS 原生环境的支持尚需进一步开发。
*   **集成成本**：虽然不需要修改智能体核心，但仍需要智能体框架提供简单的 Hook（如 Claude Code 的 PreToolUse）来调用 YoloFS 的 CLI 工具。
*   **不可逆操作风险**：虽然文件修改可以暂存，但某些外部副作用（如通过网络发送 API 请求或删除云端数据）无法通过本地文件系统快照撤销。
*   **用户依赖**：对于“目标对齐”的破坏性操作（即智能体认为删除是正确的，且用户也误判时），系统仍依赖于用户的最终审核质量。
