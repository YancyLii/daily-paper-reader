## 摘要
我们提出了 ATLAS-RTC，这是一种用于自回归语言模型的运行时控制系统，可在解码过程中强制执行结构化输出。ATLAS-RTC 在每一步监控生成过程，利用轻量级信号检测与输出契约的偏离，并应用针对性的干预措施，如偏置、掩码和回滚。与事后验证或静态约束解码不同，它以闭环方式运行，能够在错误显现之前进行修正。在结构化生成和工具调用任务中，ATLAS-RTC 将首次尝试成功率提高了 20 到 37.8 个百分点，在故障频发的场景下，延迟降低了高达 88%。结果表明，许多失败源于解码伪影而非对任务的误解，这促使运行时控制成为大语言模型系统中一个独立的层级。

## Abstract
We present ATLAS-RTC, a runtime control system for autoregressive language models that enforces structured output during decoding. ATLAS-RTC monitors generation at each step, detects drift from output contracts using lightweight signals, and applies targeted interventions such as biasing, masking, and rollback. Unlike post-hoc validation or static constrained decoding, it operates in a closed loop, enabling correction before errors materialize. Across structured generation and tool-calling tasks, ATLAS-RTC improves first-attempt success rates by 20 to 37.8 percentage points, with up to 88% latency reduction in failure-dominated settings. Results show that many failures arise from decoding artifacts rather than task misunderstanding, motivating runtime control as a distinct layer in LLM systems.

### 2. 方法论
ATLAS-RTC 将自回归生成建模为一个**受控随机过程**，核心思想是在每个解码步骤观察轨迹、预测漂移并应用阶梯式干预。
*   **闭环控制流程**：观察（Observe）→ 预测（Predict）→ 控制（Control）→ 采样（Sample）→ 更新（Update）。
*   **关键技术细节**：
    *   **运行时状态（Runtime State）**：维护一个包含生成序列、Logit 分布熵、合同进度、启发式漂移分数和学习到的失败概率的状态向量。
    *   **正式输出合同（Output Contract）**：定义为 $C = (S, T, O, V, \pi, \Phi)$。它是一个**阶段感知（Stage-aware）**的状态机，能区分“结构决策点”（需严格控制）和“值生成区”（允许模型自由发挥）。
    *   **阶梯干预策略（Ladder Policy）**：根据风险值应用不同强度的操作：无操作（noop）→ Logit 偏置（bias）→ 温度调节（temperature）→ Token 掩码（mask）→ **中途回滚与重引导（Rollback & Re-steering）**。
    *   **KV 缓存回滚**：当检测到严重偏离时，通过截断 KV 缓存回退 $n$ 个 Token 并重新生成，无需重启整个序列。

### 7. 优点
*   **阶段感知控制**：这是该方法最大的亮点。它不会在模型生成具体内容（Value）时过度干预，从而保留了模型的推理灵活性。
*   **非侵入性**：无需修改模型权重，作为推理层的插件即可运行。
*   **中途修正能力**：引入回滚机制，使系统具备了“自愈”能力，而非简单的过滤。

### 8. 不足与局限
*   **语义与结构的脱节**：ATLAS-RTC 只能保证“格式正确”，无法保证“内容正确”或“语义完整”（例如，JSON 格式对了，但关键字段内容缺失）。
*   **特定模式下的退化**：在处理需要多行字符串或高度灵活值的模式时，性能会有所下降。
*   **计算开销**：在基准模型已经表现很好的简单任务中，逐 Token 的监控会带来额外的延迟负担（如发送邮件任务延迟增加了 129%）。
*   **漂移检测依赖性**：系统的有效性高度依赖于启发式规则和轻量级分类器的准确性。
