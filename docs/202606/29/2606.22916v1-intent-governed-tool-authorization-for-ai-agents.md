## 摘要
AI代理越来越频繁地通过外部工具执行操作：它们读取私有数据、构造结构化负载、提交写请求、导出记录，并在应用边界间协调工作流。现有的授权机制通常询问某个集成凭证、应用或令牌能否调用某个工具。这个问题是必要的，但不够完整。一个工具调用可以通过静态凭证获得授权，但仍然可能不符合用户当前请求的意图。例如，一个能够读取和导出记录的凭证，在用户仅要求有限摘要时不应暴露其导出权限；而一个模型生成的删除调用，不应仅仅因为集成具有删除范围就予以执行。本文提出了意图驱动的访问控制（IGAC），这是一种服务器端授权层，将用户表达的意图视为AI代理工具使用的一个单调的、可审计的策略属性。IGAC引入了意图证书、会话范围的策略收窄、意图感知的清单过滤以及意图-工具-负载一致性检查。其核心不变性在于，用户意图只能缩减静态集成策略所授予的权限；它绝不会扩大作用域、数据策略、租户边界或审查要求。我们将IGAC映射到OpenPort上，这是一个现有的治理基座，已经实现了依赖于授权的发现、作用域和基于属性的访问控制（ABAC）策略检查、草稿优先写入、预检影响绑定、状态见证检查、幂等性、稳定的原因代码和审计功能。

## Abstract
AI agents increasingly act through external tools: they read private data, construct structured payloads, submit write requests, export records, and coordinate workflows across application boundaries. Existing authorization mechanisms usually ask whether an integration credential, app, or token can call a tool. That question is necessary but incomplete. A tool call can be authorized by static credentials and still be unjustified by the user's current request. For example, a credential that can read and export records should not expose export authority when the user only asked for a bounded summary, and a model-generated delete call should not execute merely because the integration has a delete scope. This paper proposes Intent-Governed Access Control (IGAC), a server-side authorization layer that treats the user's expressed intent as a monotone, auditable policy attribute for AI-agent tool use. IGAC introduces intent certificates, session-scoped policy narrowing, intent-aware manifest filtering, and intent-tool-payload consistency checks. The central invariant is that user intent may only reduce the authority granted by static integration policy; it never expands scopes, data policy, tenant boundaries, or review requirements. We map IGAC onto OpenPort, an existing governance substrate that already implements authorization-dependent discovery, scope and ABAC-style policy checks, draft-first writes, preflight impact binding, state-witness checks, idempotency, stable reason codes, and audit.

### 2. 论文提出的方法论：核心思想、关键技术细节

论文提出 **意图治理访问控制（Intent-Governed Access Control, IGAC）**，作为一个服务器端授权层，在不信任LLM规划器或分类器的基础上，将用户意图转化为可审计的、单调的策略属性。

*   **核心思想：** 意图只能**收窄**（narrow）静态集成策略所授予的权限，绝不能扩大作用域、数据策略或租户边界。
*   **关键技术组件：**

### 6. 论文的主要结论与发现

*   **机制有效性：** IGAC能够实现其设计的单调性不变式（意图不扩大权限），并能在合成环境中将**不安全执行率降至0**。即时注入、信息不足等攻击在到达服务器端强制点后，被有效转化为澄清请求或阻止。
*   **剩余风险集中化：** 残余的授权失败（非执行）并非随机分布，而是集中在**“受约束的创建”场景**，即LLM生成的证书或负载保留了操作的宽泛类别，但丢失了精确的金额、资源或效果边界。
*   **组件重要性排序：** 清单过滤降低了不安全工具的可见性，但单靠它不足以消除风险；**确定性一致性检查和效果控制层（OpenPort）的结合**是达成零不安全执行的关键。
*   **分类器挑战：** 当前的LLM在识别宽泛意图类别上表现尚可，但在提取精确的资源、效果边界以生成安全的授权决策上存在明显困难。这证明了**将LLM仅视为顾问而非信任根**这一设计原则的正确性。

### 7. 优点：方法或实验设计上有哪些亮点

*   **安全模型创新：** 创造性地将用户“意图”作为一个时间限定、可审计、单调的策略属性引入访问控制模型，完美契合了LLM代理时代“最小权限”的动态需求。
*   **分层解耦架构：** 明确分离了“意图控制层”和“效应控制层”，逻辑清晰。IGAC与现有OpenPort基座协同工作，而非推倒重来，提升了方案的可行性和兼容性。
*   **不信任模型的原则：** 安全边界建立在服务器端逻辑上，将LLM归类为不可信的顾问，从根本上防御了模型层面的多种攻击（如间接即时注入、工具描述投毒）。
*   **异常精细与诚实的评估体系：**
    *   **多元指标：** 引入UAR与UER区分“接受了但不安全的权限”和“真正执行的不安全操作”；区分严格与安全的BCR，以及单独的澄清率，评估维度比许多同类工作更细致。

    *   **分类器瓶颈：** 系统的安全宽度严重依赖于意图分类器生成的证书质量。实验证实LLM在此任务上精度不足，会导致高误报率（过度防御）或漏报。
    *   **运行时成熟度：** 当前的参考实现为“最小化、内存型”，非生产就绪。缺乏持久化证书存储、分布式限流、防重放等生产级特性。
    *   **复杂效应绑定难：** 对于具有隐式副作用、复合工作流或领域语义特定的工具，精确估算并绑定其影响到证书上仍是一个开放挑战。
*   **实验覆盖与偏差风险：**
    *   **规模与分布偏差：** 基准测试规模较小，且任务为手工创作或少量适配，与现实世界的复杂性和长尾分布仍有差距。良性和攻击任务比例可能带有作者偏见。
    *   **动态对抗性缺失：** 实验场景仍是静态的任务执行，未涉及自适应攻击者。攻击者可能通过多轮交互逐步探索并试图绕过一致性检查。
    *   **外部基准集成不完整：** 仅适配了小批量的外部基准任务，且评分器仍是自定义的，未使用原始基准的模拟器和完整评分标准，因此无法直接与在那些基准上发表的原始论文结果进行公平比较。
