## 摘要
基于 OpenClaw 等开源运行时的自主 AI 智能体默认向每个会话开放所有可用工具，而不考虑具体任务。例如，摘要任务与代码部署任务拥有相同的 Shell 执行、子智能体生成和凭据访问能力，这种 15 倍的过度配置比例被我们称为“能力过度配置问题”。现有的防御措施（包括 NemoClaw 容器沙箱和 Cisco DefenseClaw 技能扫描器）解决了容器化和威胁检测问题，但无法学习每种任务类型的最小可行能力集。我们提出了 Aethelgard，这是一个四层自适应治理框架，通过学习到的策略为 AI 智能体强制执行最小权限原则。第一层是能力治理器（Capability Governor），它动态界定智能体在每个会话中感知的工具范围。第三层是安全路由器（Safety Router），它在执行前利用基于规则和微调分类器的混合方法拦截工具调用。第二层是强化学习（RL）学习策略，它在累积的审计日志上训练 PPO 策略，以学习每种任务类型的最小可行技能集。

## Abstract
Autonomous AI agents built on open-source runtimes such as OpenClaw expose every available tool to every session by default, regardless of the task. A summarization task receives the same shell execution, subagent spawning, and credential access capabilities as a code deployment task, a 15x overprovision ratio that we call the capability overprovisioning problem. Existing defenses, including the NemoClaw container sandbox and the Cisco DefenseClaw skill scanner, address containment and threat detection but do not learn the minimum viable capability set for each task type.   We present Aethelgard, a four layer adaptive governance framework that enforces least privilege for AI agents through a learned policy. Layer 1, the Capability Governor, dynamically scopes which tools the agent is aware of in each session. Layer 3, the Safety Router, intercepts tool calls before execution using a hybrid rule based and fine tuned classifier. Layer 2, the RL Learning Policy, trains a PPO policy on the accumulated audit log to learn the minimum viable skill set for each task type.

### 1. 核心问题与整体含义
*   **研究动机**：当前的自主 AI 智能体运行时（如 OpenClaw）普遍存在“能力过度配置”（Capability Overprovisioning）问题。无论任务是简单的“文档摘要”还是复杂的“代码部署”，系统默认都会向智能体开放所有工具（如 Shell 执行、子智能体生成等）。
*   **安全风险**：这种 15 倍的权限冗余导致了巨大的攻击面。一旦发生提示词注入（Prompt Injection）或供应链攻击，攻击者可以利用智能体拥有的多余权限执行恶意代码或窃取凭据。
*   **核心目标**：通过学习每种任务类型的“最小可行技能集”，实现动态的最小权限原则（Least Privilege），让智能体“无法滥用它不知道存在的工具”。

### 2. 方法论
Aethelgard 是一个四层治理框架，其核心思想是将能力范围界定视为一个**学习优化问题**：
*   **第一层：能力治理器 (Capability Governor)**：在会话开始前，通过注入动态生成的 `AGENTS.md` 和修改配置文件，从语义层面限制智能体对工具的感知。
*   **第二层：强化学习策略 (RL Learning Policy)**：使用 **PPO (近端策略优化)** 算法。
    *   **状态 (State)**：任务类型 + 信任等级。
    *   **动作 (Action)**：17 种工具的二进制掩码（显示/隐藏）。
    *   **奖励函数**：平衡任务成功率、技能经济比（SER）和安全性（拦截违规调用的次数）。
*   **第三层：安全路由器 (Safety Router)**：作为中间人代理（MITM Proxy），在工具执行前进行两阶段拦截：
    *   **阶段 1 (规则驱动)**：毫秒级匹配高危指令（如 `rm -rf`）和敏感路径。
    *   **阶段 2 (LLM 驱动)**：使用微调后的 Qwen2.5-1.5B 模型对工具调用的意图进行分类。
*   **第四层：沙箱层**：集成现有的容器化技术（如 NemoClaw）进行物理隔离。

### 6. 主要结论与发现
*   **权限缩减**：在摘要任务中实现了 73% 的工具缩减，并 100% 消除了危险工具。
*   **效率提升**：经过 PPO 训练后，技能经济比（SER）在真实会话中提升了 **337%**。
*   **安全性**：在 500 组测试中，100% 拦截了 `exec`（执行）和 `sessions spawn`（生成子智能体）调用。
*   **关键发现**：即使提示词注入在 LLM 层面成功（模型生成了恶意指令），Aethelgard 也能在基础设施层面将其拦截。

### 7. 优点
*   **动态自适应**：超越了传统的静态 YAML 防火墙，能够根据任务类型自动进化治理策略。
*   **防御深度**：结合了语义限制（让智能体不知道有该工具）和运行时拦截（不准执行该工具）。
*   **指标创新**：提出了 SER 这一量化指标，为智能体权限治理提供了科学的衡量标准。

### 8. 不足与局限
*   **语义内容盲区**：该框架仅治理“工具调用”，如果智能体通过纯文本回复（如教用户如何手动执行恶意命令）进行攻击，该系统无法拦截。
*   **推理延迟**：LLM 分类器引入了约 0.8 到 2 秒的额外延迟，可能影响高频交互体验。
*   **样本偏差**：微调分类器所用的训练数据（273 条）规模较小，且存在严重的类别不平衡（95% 为拦截类），可能导致在某些良性写操作上出现误报。
*   **状态空间限制**：目前的 RL 策略未考虑用户身份或历史行为，个性化治理能力有限。
