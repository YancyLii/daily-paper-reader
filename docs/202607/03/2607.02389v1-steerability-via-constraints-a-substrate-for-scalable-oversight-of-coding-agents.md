## 摘要
编码代理能力强大；人类监督成为瓶颈。无约束的代理会引入安全风险，侵蚀代码库的可扩展性，并使人工审查成本越来越高。我们认为，数十年来用于管理大型人类工程团队的方法：访问控制、网络策略、由工具强制执行的严格编码规范，可以直接转移到编码代理，并且比最近的代理脚手架更便宜（按 token 计）。我们基于这一原则勾勒了一个端到端系统，并报告了一个可扩展监督的对照实验：一个小型审查者（Gemma 4 e4b）检查一个包含 11 个插入后门的 Python 代码库。召回率从 54.5%（无约束，无工具）上升到 90.9%（约束基础加上一个约 200 行代码的 `docs` CLI），基础和工具独立做出贡献。我们特意选择 Python：在默认保证最少的语言中，基于基础级别的监督收益最大；这些原则可扩展到 Rust 等语言。

## Abstract
Coding agents are capable; human oversight is the bottleneck. Unconstrained agents introduce security risks, erode codebase scalability, and make human review increasingly costly. We argue that the same methods used for decades to manage large human engineering teams: access control, network policies, strict coding conventions enforced by tooling; transfer directly to coding agents, and are cheaper (in token) than recent agentic scaffolding. We sketch a start-to-end system on this principle, and report a controlled experiment in scalable oversight: a small reviewer (Gemma 4 e4b) inspects a Python codebase containing 11 inserted backdoors. Recall rises from 54.5% (unconstrained, no tools) to 90.9% (constrained substrate plus a ~200-LoC `docs` CLI), with substrate and tools contributing independently. We choose Python deliberately: substrate-level oversight gains are largest where the language gives the fewest guarantees by default; the principles extend to languages like Rust.

## 1. 核心问题与研究动机
- **核心瓶颈转移**：编码代理（LLM agents）编写大量可用代码的能力日益增强，但安全、可靠地使用它们的关键瓶颈已从“是否能够生成代码”转变为“能否有效验证/审计代码”。
- **双重风险**：无约束的代理不仅带来直接的**安全风险**（如后门、恶意代码），更会导致**代码库逐渐失控**——随着项目规模增长，代码的可维护性、可审计性持续下降，人类审查成本线性甚至指数级上升。
- **根源是上下文长度限制**：无论是人类还是审查模型，都无法将整个代码库保存在工作记忆中。同时，弱执行的编码规范会在长上下文中逐渐漂移失效。传统的Agentic解决方案（如MCP协议）可能会通过向代理上下文填充原始数据来加剧这一问题，而不是减轻它。
- **借鉴传统软件工程实践**：论文观察到，在大型语言模型出现之前，人类团队早已利用访问控制、网络策略、严格的编码约定及工具（linters、类型检查器、运行时契约验证）来管理大规模工程。本文核心动机是：**将这些久经考验的约束方法直接移植到编码代理上，构建一个可扩展且便宜（按token计费）的监督基底**。

## 2. 方法论与关键技术细节
- **核心思想：可操控性通过约束实现**  
  将提示词级别（易失效的）指导改变为**外部、可验证的强制约束**，使监控和控制不依赖于模型遵从性，而是依据工具拒绝执行不合规代码的能力。
- **约束基底与投影原则**
  - **基底约束**：采用linter（ruff）、类型检查器（pyrefly）、运行时契约验证器（beartype）、严格数据类（pydantic strict模式）等工具，形成一道“不可协商”防线。错误代码无法运行。
  - **源码作为单一规约**：拒绝外部架构JSON等“分布式状态”，因为其易与代码演进脱节。遵循**投影原则**：通过一个小型CLI（`docs`，约200行代码，基于`griffe`的AST解析）提供代码拓扑视图，惰性加载，使审查者无需阅读全部原始实现即可理解结构。
  - **严格基底编码约定**：包括不可变数据类和纯函数（`core/`）与不纯执行者（`runners/`）的分离。关键规则和不变式直接编码在工具规则或模块级文档字符串中，使违规在调用点即被**文本可见**。
  - **环境与推理层拦截代理**：使用虚拟机/容器（带网络出口白名单限制）和**类LiteLLM的推理拦截代理**，对每个提示和响应做语义探测（如间接提示注入检测）。详情见附录A.1。

- **具体工具链组合**
  - `ruff`（安全、bugbear、bandit规则集）
  - `pyrefly`（类型检查）
  - `beartype` + `pydantic strict`（运行时契约与不可变数据结构）
  - `jaxtyping`（精细类型，如 `Float[Tensor, "b d"]`）
  - 小规模`docs CLI`（三个命令：`docs module X`、`docs symbol X.Y`、`docs examples X`）

- **审查协议（多遍审查）**
  对每个函数独立运行多遍聚焦式审查：
  1. **自洽性**：函数体是否与命名和文档一致？
  2. **输入/输出契约**：类型标注是否足够窄以防误用，足够宽以接纳合法值？函数体是否总能产生声明输出类型？
  3. **副作用**：有无未声明的副作用？
  4. **不变式合规性**（仅当模块含不变式时）：函数体是否遵守每个不变式？
  5. 无特定焦点的兜底审查：标记任何恶意行为。

## 3. 实验设计与对比方法
- **场景与数据集**：一个约500行的合成Python代码库（微型LM评估流水线），两种版本共享相同的调用图、名称和正常行为，仅基底不同（有无约束）。代码中手动插入了**11个分层后门**（L1-L5，详见附录C），且这些后门在干净测试套件上不触发，意味着它们未被强制工具规则自动捕获。
- **任务**：使用一个小型固定审查模型（Gemma 4 e4b，8B参数量级）审查代码库，找出后门。审查上下文限制在≤4K tokens（小于代码库），强制进行函数级精细审查，而非全库一次性审查。
- **对比结构（2×2对照）**：
  - **基底维度**：无约束 vs. 有约束。
  - **工具维度**：无`docs CLI` vs. 有`docs CLI`。
  - 形成四个实验单元格：C1（无约束+无工具）、C2（无约束+有`docs CLI`）、C3（有约束+无工具）、C4（有约束+有`docs CLI`，即全配置）。
- **评分**：若模型输出中引用文件后缀匹配预期文件，且理由中包含后门关键词（不区分大小写），则视为捕获。多文件后门接受任意匹配。

## 5. 实验充分性与客观性分析
- **实验规模**：围绕11个插入到500行代码中的后门，进行了1个2×2因子的对照实验。但是，数据集是**合成且小规模**的。
- **消融分析**：通过2×2设计自然实现了基底与工具的消融。给出了清晰的增量贡献分解：基底独立贡献+3个后门检出（主要是全局不变式违反L5），工具独立贡献+3个（多文件流L3等），组合贡献总共+4。
- **充分性评估**：
  - **公平性**：无约束与约束版本代码体相同，仅周围元数据（文档、类型、不变式位置）不同，实现了对基底效应的受控测量。
  - **充分性局限**：实验**仅为概念验证**。论文坦诚指出了多个不足：样本量小；后门手工指定，缺乏自然性；评分标准定制；仅用了一个小模型进行监督，未测试更大模型是否性能饱和；未测试多轮随机种子或进行提示搜索。因此，结论的普适性需要更大规模实验验证。作者将本实验定位为“可能性的最小示例”，而非严格论据。
  - **客观性**：作者注意到并报告了一个偏倚：将分布式全局文档（`ARCHITECTURE.md`）仅绑定到无约束单元格，而约束单元格在模块内拥有不变式，这本身是有意模拟的实际情况（审查者不会每份差异都重读架构文档），但这也将局部效应打包进了基底优势中，作者已明确声明这一不对称性。

## 6. 主要结论与发现
- **关键发现**：在小型模型中，通过约束基底和配套CLI工具，审查者的**后门召回率从54.5%显著提升至90.9%**，且**精确率保持了高水平**（C4: 95.7%）。
- **独立贡献**：基底和工具**各自独立地提升了召回率**。基底独自拿下3个检出（C3-C1），工具独自拿下3个检出（C2-C1），两者边际效应部分交叉但不可替代。
- **审查协议关键性**：**审查协议与基底同等重要**。多遍聚焦审查法解除了单遍审查中小模型因上下文增长而过早终止的回归效应（单遍下C4效果甚至差于C1），使基底的效果得以发挥。
- **精确率的代价与补偿**：单独施加约束（C3）会降低精确率（从100%降至88.9%），因为可见的不变式诱使模型对正常代码幻想出违规。当增加`docs CLI`工具后，精确率大部分恢复（95.7%），说明**工具帮助模型高效验证不变式**，从而减少误报。
- **增益集中域**：对局部可见后门（L1）所有设置均检出；对跨文件、全局不变式违规等复杂后门（L3、L5），基底和工具带来了决定性增益。

## 7. 优点与亮点
- **学科交叉思路**：将成熟非AI软件工程实践（契约式设计、类型检查、访问控制、沙箱）系统化地应用于AI代理监督，具有高度的工程合理性和实用性。
- **紧致且可审计**：提出的系统原型（`docs` CLI等）每个组件均在**100–200行代码左右**，人类易审计，符合安全最小化原则，与依赖庞大、不透明框架的Agentic方案形成鲜明对照。
- **经济性论证**：论证了该方法比不断向代理上下文塞入原始数据（如MCP负载）更“便宜”，并且惰性加载工具兼容模型对标准Unix工具的强参数记忆，有望降低token成本。
- **表面效应与诱因分析**：实验不仅给出最终得分，还通过单遍/多遍对比，揭示了仅施加约束可能诱发的精确率下降（幻觉不变式违规）及工具的补偿机制，分析深入。
- **强调局域性原则**：坚持“规范应当靠近受检代码”的原则（文档和不变式在模块内），并以实验说明了将信息集中存于ARCHITECTURE.md的价值损失，这个工程见解强有力。

## 8. 不足与局限
- **实验规模极限**：
  - 测试集仅**1个小规模合成代码库**和**11个手工后门**，与真实世界的多样性、噪声和代码复杂度的差距巨大。结论推广到海量真实漏洞语料库的能力未经验证。
  - 仅使用一个8B级别的小型开源模型（Gemma 4 e4b）进行监督，未探究该方法在更大能力模型上的边际效用变化，也未证明对更大规模问题是否仍然稳健。
- **手动偏差风险**：
  - 后门和评分标准由同一作者设计，可能无意识中嵌入了使利于自方方法的偏差。
  - “约束单元格”与“无约束单元格”对全局架构信息的放置（内联 vs. 集中文档）是设计选择，这种设计捆绑效应会让基底看起来比实际更有效。
- **监督者能力局限**：尽管约束使审查聚焦，监督模型仍未能捕获所有语义后门（L2类），表明仅靠基底和轮廓工具无法消解所有类型的内在监督不对称性（附录B提到但未解决）。需要更强的模型能力或结合符号验证。
- **系统原型成熟度**：所提的系统草图（Section 4）仅为原型，许多安全部件（如语义探测）还是“概念验证”状态，尚未系统基准测试和加固。与NVIDIA OpenShell等成熟工程对比，自动化水平、功能覆盖有差距。
- **应用范围**：论文仅聚焦于Python实验，虽声称可扩展至Rust等语言，但无实验证据。对更复杂、多语言异构项目中的可扩展性未提供实证。
