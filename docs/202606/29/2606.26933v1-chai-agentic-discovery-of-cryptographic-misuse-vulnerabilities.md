## 摘要
AI辅助的漏洞发现已在内存安全等缺陷类别中被证明有效，这些类别通过插桩确认内存违规并高效过滤误报。然而，许多危险的漏洞类别，例如密码误用，缺乏可比的插桩手段。本文提出Chai，一个基于人工智能的系统，通过自然产生的信号发现并验证密码误用漏洞。为此，Chai重新思考经典的差分测试技术，利用AI：1）提升检测库中真实安全问题的精度，2）将通常被忽视的差异重新用作下游应用中实质性漏洞的线索。由此，Chai颠覆了当前AI漏洞发现的主流范式：不再针对单个代码库审计诸多缺陷，而是在库级别编录缺陷，并将其传播到密码依赖图中，实现复合效率增益。我们在X.509、JWT和SAML库上评估了Chai。Chai在一个支撑数十亿设备的SSL库中发现了一个此前未知的严重漏洞，还在一个主流浏览器背后的库以及主要Linux发行版中的另一个库中发现了安全缺陷。这些技术总共发现了超过100个漏洞。

## Abstract
AI-assisted vulnerability discovery has proven effective for bug classes like memory safety, where instrumentation confirms memory violations and efficiently filters false positives. Many dangerous vulnerability classes, such as cryptographic misuse, however, lack any comparable instrumentation. In this work, we present Chai, an AI-based system that discovers and validates cryptographic misuse vulnerabilities through naturally occurring signals. To achieve this, Chai rethinks the classical technique of differential testing by leveraging AI to 1) improve precision for detecting real security issues in libraries, and 2) repurpose commonly overlooked discrepancies as leads for tangible vulnerabilities in downstream applications. In doing so, Chai inverts the prevailing paradigm of AI vulnerability discovery: instead of auditing one codebase for many flaws, it catalogs flaws at the library level and propagates them across a cryptographic dependency graph, delivering compounding efficiency gains. We evaluate Chai across X.509, JWT, and SAML libraries. Chai discovered a previously unknown critical vulnerability in an SSL library that powers billions of devices, along with security bugs in one library behind a major web browser and another in major Linux distributions. In total, these techniques surfaced over 100 vulnerabilities.

## 1. 论文的核心问题与整体含义
- **核心问题**：密码误用（cryptographic misuse）漏洞，如 X.509、JWT、SAML 等协议实现中的缺陷，缺乏类似内存安全漏洞（如 AddressSanitizer）那样的插桩信号来自动化检测和过滤误报。现有 AI 漏洞发现主要面向单一代码库的开放式审计，效率低，且难以跨越协议实现间的语义差异。
- **背景与动机**：
  - 密码协议通常由多个规范和众多库实现，规范中的模糊角落导致不同实现的安全假设不一致，从而产生下游漏洞（如 CVE-2018-0114 中 JWT 库错误信任嵌入的密钥）。
  - 传统差分测试能发现跨实现的不一致，但噪声大、需手工构建协议专用语法，且仅停留在库级别，忽略了这些差异在下游应用中的安全影响。
  - 现有 AI 辅助漏洞发现范式（如图 2 所示）是针对单个项目进行多缺陷搜索，但密码缺陷往往由库-应用间的契约错配引起，仅审计应用代码无法发现。
  - 该工作旨在通过 AI 与差分测试结合，按库→依赖图的路径成规模地发现密码误用漏洞，并实现复合效率。

## 2. 方法论
Chai 系统由两个独立阶段组成，以“差异”作为衔接的单一产物。

### 2.1 第一阶段：放大测试（Amplified Testing）
- **核心思想**：用 AI 生成测试输入，通过差分测试在所有已接入的库实现上同时运行，以判断接受/拒绝等行为的差异；AI 负责提供协议知识和自适应搜索，差分测试提供可验证的信号。
- **关键细节**：
  - **变异（Mutation）**：从一个已知正常种子开始，AI 代理提出变异程序（有序操作列表），操作目录覆盖协议的信任边界（如篡改签名保护摘要、重排断言、注入文档类型声明等）。
  - **构造（Construction）**：构建器（builder）以确定的方式将 (种子, 变异程序) 对组装成标准格式的二进制消息，确保可复现。代理只需推理“测试什么行为”，而非“如何编码”。
  - **自适应迭代（Adaptive Iteration）**：每次迭代前，检索代理（RAG）从已知 CVE 数据库中查询与当前目标相关的历史披露模式，引导下一次变异。接受/拒绝分裂等浅层差异会触发后续输入以试图将其升级为更高影响的安全差异。
  - **资源分配（Resource Allocation）**：将搜索空间划分为可租用的“变异组”（(cluster, operation) 对），采用 UCB1 多臂赌博机策略分配并行代理的租约，奖励独特的新行为，抑制重复搜索。
  - **输出分析**：对产生的原始差异进行重放、最小化（逐个移除变异操作，确保差异不变），再通过分类器贴标签（涉及的校验行为），并划分为漏洞、安全缺陷、模糊点或无关差异。

### 2.2 第二阶段：差异追踪（Discrepancy Tracing）
- **核心思想**：第一阶段确认的模糊点（如库未强制实施某个安全属性，但其他库实施）并不一定是直接漏洞，但可能在依赖该库的下游应用中成为可触发的漏洞。Chai 沿依赖图反向搜索，仅对那些可能继承该行为的应用进行针对性审计。
- **步骤**：
  1. **枚举**：基于 OpenSSF Criticality Score 的 50,000 个关键仓库，构建跨 17 个生态系统的依赖图（通过包清单解析），将库映射到所有直接或间接依赖它的项目。
  2. **初始扫描**：编码代理对候选应用进行轻量只读分析，判断该模糊点是否可能暴露（如应用是否信任令牌内嵌的密钥）。
  3. **概念证明（PoC）流水线**：对可能受影响的候选应用，在沙箱虚拟机中生成多个独立的 PoC，并迭代优化到端到端攻击脚本，最终在干净 VM 上重新验证，输出 setup、run 脚本和 git diff 等证据。
  4. **报告生成**：组装披露报告，搜索公开 issue 和 CVE 去重，经人工审核后再披露。

### 2.3 复合效率
- 放大测试：一个输入同时测试所有库实现，而非逐个审计应用。
- 反向搜索：一个库的模糊点仅沿依赖图追踪到受影响的应用，而非扫描所有项目。
- 针对性审计：对每个应用仅检查单一具体行为，而非开放式搜索漏洞。

## 6. 主要结论与发现
- Chai 发现了超过 100 个密码误用漏洞/安全缺陷，包括一个影响数十亿设备的 wolfSSL 关键证书链绕过漏洞（CVE-2026-11310、CVE-2026-11999），以及其他主流浏览器和 Linux 发行版库中的问题。
- Chai 的放大测试效率远超现有差分测试/模糊测试工具：发现 2 倍以上的唯一差异，且成本更低，输入数量少几个数量级；差异分布更广，尤其能捕获少数库接受多数库拒绝的“左尾”强信号差异。
- 差异追踪能成功将库级别的模糊点转化为下游应用中的具体漏洞（如撤销绕过、重放攻击），且这种缺陷对传统开放式代理审计来说很难发现（盲 wolfSSL 实验显示 Agents 几乎不能稳定找到）。
- 该方法不依赖单一前沿模型，开源模型（Kimi K2.6）亦可产出大量差异，说明设计是核心驱动力。

## 7. 优点
- **范式反转**：从单一代码库审计转为库缺陷编录+依赖传播，挖掘了“契约模糊”这一盲区。
- **复合效率**：一次测试放大至所有库，一个模糊点追踪至所有受影响应用，一次审计只问一个具体问题。
- **AI与差分测试结合**：用AI替代手工语法和开放式分析，用差分信号提供可验证的“正确性”依据，互相弥补弱点。
- **设计鲁棒性**：实验显示多种模型均可有效运行，开源模型适用；资源分配（UCB1）自动聚焦高产搜索区域。
- **实践影响**：在广泛使用的库中发现严重、已确认的0-day，并快速得到维护者认可和修复，证明方法具有真实安全价值。
- **系统完整性**：包含从输入生成、差分执行、输出分析到下游追踪和报告生成的完整流水线。

## 8. 不足与局限
- **威胁模型限制**：仅检测攻击者能提交协议消息的场景，且要求至少两个实现产生分歧；无法发现单一实现独有的缺陷或规范本身的设计漏洞。
- **覆盖不完全**：搜索是经验性的概率性过程，并不保证发现所有差异或漏洞，某些类别可能遗漏。
- **依赖库的一致性**：当某个库的实现过于宽松或不严格时，会引入噪声，抑制差异检测，且这些库可能最需要审查。
- **下游追踪依赖人工**：虽然自动化了大部分步骤，但最终报告仍需人工审核来避免虚假报告，人力负担依然存在。
- **领域范围**：目前仅在 X.509、JWT、SAML 这三个标准化程度高、接口明确的密码协议上验证，推广到其他领域可能需要额外工程。
- **差分信号的局限性**：差异仅表明分歧，不一定直接对应漏洞，仍需分析员分类，且某些类型的安全缺陷（如侧信道）不会产生可见的输出差异。
