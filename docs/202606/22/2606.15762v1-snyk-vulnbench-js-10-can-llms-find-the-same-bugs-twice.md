## 摘要
我们运行了300次重复的漏洞发现扫描，以衡量在相同的JavaScript代码、提示和基准测试框架下，智能体大语言模型（LLM）安全审查的可重复性。主要结果是，LLM的安全发现可重复性不均匀：与参考结果匹配的发现是稳定的，但额外的模型报告在不同运行之间差异很大。在250次模型运行中，161个唯一未匹配发现中有80个仅在五次相同重复中的一次出现，而只有22个在所有五次中都出现。相比之下，当Claude匹配到Snyk Code参考发现时，行为稳定得多：158个唯一参考匹配发现中有134个在所有五次重复中出现。基准测试还显示了互补性。模型一致地发现熟悉的、高信号的利用模式，并在一个案例中揭示了可能的Snyk Code产品空白。Snyk Code静态应用安全测试（SAST）是确定性的，并且更擅长系统地枚举重复的数据流汇。结果支持将智能体LLM审查与确定性SAST相结合，而不是将任何一种技术视为另一种的替代品。

## Abstract
We ran 300 repeated vulnerability-finding scans to measure how repeatable agentic large language model (LLM) security review is on the same JavaScript code, prompt, and benchmark harness. The headline result is that LLM security findings were unevenly repeatable: reference-matched findings were stable, but extra model reports varied heavily from run to run. Across 250 model runs, 80 of 161 unique unmatched findings appeared in only one of five identical repetitions, while only 22 appeared in all five. By contrast, when Claude matched a Snyk Code reference finding, the behavior was much more stable: 134 of 158 unique reference-matched findings appeared in all five repetitions. The benchmark also shows complementarity. Models consistently found familiar, high-signal exploit shapes, and in one case surfaced a likely Snyk Code product gap. Snyk Code static application security testing (SAST) was deterministic and better at systematically enumerating repeated data-flow sinks. The results support combining agentic LLM review with deterministic SAST rather than treating either technique as a replacement for the other.

## 1. 论文的核心问题与整体含义
- **核心问题**：当同一个大语言模型（LLM）在完全相同的代码、提示词和执行环境下进行安全审查时，其输出的漏洞报告是否可重复？传统静态应用安全测试（SAST）工具是确定性的，规则不变则结果不变；而 LLM 具有概率性，其审查结果可能因运行次不同而产生波动。
- **整体含义**：该研究旨在量化 LLM 在安全审查中的可重复性，并评估其与确定性 SAST（Snyk Code）的差异和互补性。它并非单纯比较谁“更好”，而是说明两者在漏洞发现上具有不同的盲区和强项，支持将智能体（agentic）LLM 审查与确定性 SAST 结合使用。

## 2. 论文提出的方法论
- **基准设计**：Snyk VulnBench JS 1.0 包含 10 个基于 Express 的 JavaScript 小应用，共计 44 个 Snyk Code 参考发现。项目规模从单文件片段到含路由、数据库、上传功能的多文件 todo 应用。
- **模型配置与执行**：使用相同的直接审计提示（direct audit prompt），通过 Claude Code harness（Anthropic TypeScript Agent SDK）运行 5 种 Claude 模型配置：Claude Opus 4.6 Medium、High，Opus 4.7 Max，Sonnet 4.6 Medium、High。每个配置对同一任务重复执行 5 次，总共 300 次运行。
- **评分规则**：采用宽松匹配策略——只要模型报告与参考发现属于同一漏洞类型即算命中，不要求文件、行号、严重程度或数据流路径完全一致。主要协议指标为 **Snyk‑reference F1**（以 Snyk Code 发现为参考计算精确率与召回率的调和平均），但作者强调这并非漏洞检测准确率的绝对度量。
- **稳定性分析**：针对“未匹配发现”（模型输出但不在 Snyk Code 参考中的发现）和“参考匹配发现”分别统计其在 5 次重复中出现的频率，以此量化模型输出的波动性。

## 3. 实验设计
- **数据集/场景**：10 个 JavaScript/Express fixture 项目，覆盖硬编码凭据、跨站脚本、命令注入、路径遍历、SQL 注入、ReDoS、SSRF 等多种漏洞类型。其中包含简单片段和一个较复杂的多文件应用（js‑project‑nightowl）。
- **基准（baseline）**：Snyk Code SAST 自身作为确定性参考。它在每次运行中都能 100% 复现自己的参考发现，F1 为 100%，标准差为 0。

## 6. 论文的主要结论与发现
- **可重复性不均匀**：
  - 当 LLM 发现与 Snyk Code 参考匹配的漏洞时，其行为较稳定：158 个唯一参考匹配发现中有 134 个（84.8%）在全部 5 次重复中被检出。
  - 但额外的（未匹配）发现极不稳定：161 个唯一未匹配发现中，80 个（49.7%）仅出现于 5 次重复中的 1 次，只有 22 个（13.7%）在所有 5 次中出现。这意味着模型产生的非参考报告队列波动大，会带来分诊负担。
- **模型与 SAST 的互补性**：
  - LLM 擅于发现命令注入、代码注入、硬编码凭据、SQL 注入、SSRF 等典型高信号漏洞模式。
  - 在资源限制、数据清洗、类型校验、不安全传输、信息泄露以及需要系统枚举重复数据流汇（如路径遍历）的类别上，LLM 表现较弱，这正是 SAST 的强项。
  - 模型输出的“未匹配发现”并不全是误报，其中可能包含 SAST 未覆盖的真实漏洞（如 js‑project‑nightowl 中的 SQL 注入），提示了 SAST 规则的改进空间。
- **成本与效果不成正比**：Claude Opus 4.7 Max 成本是 Opus 4.6 Medium 的 5.7 倍，token 用量是 1.9 倍，但其 Snyk‑reference F1（68.8%）反而低于 Opus 4.6 Medium（75.4%）。
- **整体结论**：不应将 LLM 看作 SAST 的替代品，而应将两者的审查结果结合使用。

## 7. 优点
- **明确的可重复性量化**：通过严格的重复扫描实验，首次用数据展示 LLM 安全审查输出的稳定性分化（参考匹配仍稳定，额外发现高度随机）。
- **透明的基准与分析方法**：公开基准设计、宽松匹配规则以及不确定性分析，避免将 F1 误读为绝对准确率。
- **互补性视角**：不局限于对比优劣，而是揭示 LLM 与 SAST 各自的盲区及协同价值，并给出一个实际的 SAST 缺失案例。
- **成本‑收益分析**：将推理成本纳入评估，警示“更贵不等于更安全”。

## 8. 不足与局限
- **参考集依赖**：参考发现完全由 Snyk Code 提供，存在循环验证风险；虽作者已声明不是绝对真值，但仍可能低估其他有效发现或高估一致性问题。
- **宽松评分**：仅按漏洞类型匹配会掩盖模型在定位准确性、重复路径处理上的缺陷，实际效果可能比报告的 F1 更差。
- **基准规模与范围**：仅为 10 个小型 JavaScript/Express 项目，不含 TypeScript、大型 mono‑repo、复杂业务逻辑、BOLA 等类型，外部推广泛化能力存疑。
- **未评估组合工作流**：实验仅对比单独模型与单独 SAST，未测试将两者实际融合（如模型获得 SAST 上下文后）的效果。
- **分析深度有限**：对大量未匹配发现未做进一步人工裁决，仅通过个别案例说明其价值，误报与真实补充的比例仍不清楚。
