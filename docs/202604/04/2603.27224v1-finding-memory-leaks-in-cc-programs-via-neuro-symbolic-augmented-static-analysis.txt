## Abstract 

Memory leaks remain prevalent in real-world C/C++ software. Static analyzers such as CodeQL provide scalable program anal-ysis but frequently miss such bugs because they cannot recog-nize project-specific custom memory-management functions and lack path-sensitive control-flow modeling. We present MemHint , a neuro-symbolic pipeline that addresses both limitations by combin-ing LLMs’ semantic understanding of code with Z3-based symbolic reasoning. MemHint parses the target codebase and applies an LLM to classify each function as a memory allocator, deallocator, or neither, producing function summaries that record which ar-gument or return value carries memory ownership, extending the analyzer’s built-in knowledge beyond standard primitives such as 

malloc and free . A Z3-based validation step checks each summary against the function’s control-flow graph, discarding those whose claimed memory operation is unreachable on any feasible path. The validated summaries are injected into CodeQL and Infer via their respective extension mechanisms. Z3 path feasibility filtering then eliminates warnings on infeasible paths, and a final LLM-based vali-dation step confirms whether each remaining warning is a genuine bug. On seven real-world C/C++ projects totaling over 3.4M lines of code, MemHint detects 52 unique memory leaks (47 confirmed/-fixed, 4 CVEs submitted) at approximately $1.7 per detected bug, compared to 19 by vanilla CodeQL and 3 by vanilla Infer.

## 1 Introduction 

C/C++, as programming languages with explicit memory man-agement, require developers to manually deallocate memory re-gions that are no longer needed. Failure to do so results in memory leaks : unreleased memory cannot be reclaimed by the operating system, gradually depleting system resources and potentially re-sulting in system failures [ 12 , 70 ] or exploitable security vulnerabil-ities [ 14 , 44 ]. Recent high-profile incidents in late 2025 underscore this risk: a memory leak in Windows 11’s update service led to unbounded RAM consumption of up to 20GB, rendering affected systems unresponsive [ 46 ], while a memory management flaw in MongoDB allowed unauthenticated attackers to extract cleartext credentials [ 54 ]. With over 1,700 CVEs (Common Vulnerabilities and Exposures) attributed to memory leaks as of March 2026 [ 50 ], effective automated detection of such defects remains a pressing need for software reliability and security. Memory leaks are notoriously difficult to detect. Dynamic anal-ysis approaches such as fuzzing [ 1, 21 , 32 , 67 ] can identify leaks along executed paths but impose substantial runtime overhead that limits scalability for large-scale or long-running software [ 58 , 69 ], and may fail to exercise the specific paths needed to trigger a leak within a practical time budget (as discussed in Section 2). Static analysis is a more attractive alternative, as it can reason over all program paths without requiring execution. Static analyzers such as CodeQL [ 17 ] and Infer [ 13 ] perform interprocedural dataflow analysis to trace how memory is allocated and deallocated, de-tecting mismatched or missing deallocations, and have seen wide industrial adoption [ 10 , 18 ]. However, these analyzers lack semantic understanding of custom memory management and path feasibility ,resulting in both missed detections and spurious warnings. In real-world C/C++ projects, developers often implement cus-tom variants of Memory Management (MM) functions rather than directly calling standard primitives such as malloc and free . This practice is especially prevalent in large open-source codebases that span many modules from diverse contributors [ 12 , 59 ], where a single project may define hundreds of custom memory allocation and deallocation functions (e.g., tree_init and policy_tree_free

in OpenSSL [ 51 ]). Effective leak detection thus requires identifying these custom functions and their allocation/deallocation pairings. Unfortunately, existing static analyzers such as CodeQL by de-fault operate with a fixed list of standard primitives, remaining blind to custom MM functions and silently missing leaks medi-ated by them. Prior work [ 4 , 12 , 38 , 42 , 65 ] resorts to heuristics or natural language processing (NLP) techniques that infer mem-ory management roles from function names or documentation, yet these heuristics do not check whether the inferred allocation or deallocation is reachable on any feasible intraprocedural path. A function may contain a syntactic call to free inside a branch that is never satisfiable; labeling such a function as a deallocator will cause the analyzer to wrongly assume the memory can be freed through it, suppressing genuine leak reports. Even when custom MM functions are correctly identified, the analyzers themselves remain path-insensitive: they often over-approximate program be-havior and report warnings along paths that are infeasible under actual runtime conditions, flooding developers with false alarms that demand costly manual triage [25, 31, 48]. Our key insight is that addressing the limitations above requires both semantic understanding and symbolic reasoning: identifying custom MM functions requires comprehending developer intent from code context, while validating those identifications and filter-ing false positives demands rigorous symbolic analysis of program control flow. We realize this insight in MemHint , which is a neuro-symbolic pipeline. For the neuro component, we leverage Large Language Models (LLMs), which have shown strong capabilities across programming tasks [ 24 , 28 , 60 ], to infer the memory man-agement roles of functions. However, LLMs are susceptible to hallu-cinations [ 27 , 72 ] and may produce incorrect annotations. We thus complement them with a symbolic component that validates each LLM-generated annotation and filters analyzer output against path feasibility constraints using the Z3 Satisfiability Modulo Theories (SMT) solver [ 8]. By composing the two, existing static analyzers can be augmented for more precise memory leak detection. Concretely, MemHint operates in three stages. It first parses the target codebase to extract functions along with their signatures, bodies, and callees, as well as macros. An LLM (Gemini 3 in our study) then analyzes each function and its callees to label it as a memory allocator, deallocator, or neither, producing a function sum-mary that records the function’s MM role and which argument or return value carries memory ownership. Since LLMs may produce incorrect annotations, MemHint further constructs a control-flow graph for each annotated function and encodes its path conditions into Z3 to check whether the claimed allocation or deallocation is reachable on any feasible path, discarding unsound annotations. In the second stage, the validated MM function summaries are injected into static analyzers (CodeQL and Infer in our study) via their respective extension mechanisms. In the final stage, MemHint 

uses Z3 to check whether each reported leak path is feasible and discards warnings on infeasible paths; a final LLM-based validation step confirms whether each remaining warning is a genuine bug. We evaluate MemHint on seven widely-used open-source C/C++ projects totaling over 3.4 million lines of code. MemHint detects 52 unique memory leak bugs, of which 47 have been confirmed or fixed by project maintainers or through our submitted patches, and 4 have been submitted as CVE reports currently under review. In comparison, current open-source and industrial-strength static an-alyzers detect far fewer: the best-performing baseline, vanilla Cod-eQL, finds only 19, while vanilla Infer detects only 3, Semgrep [ 56 ]only 2, and the recent state-of-the-art academic tool LeakGuard [ 38 ]finds 20. We construct a human-labeled dataset of 1,532 functions; evaluation on it shows that MemHint achieves 100% precision and recall on reachable memory management function identification, substantially outperforming LeakGuard (36.8% precision, 58.7% re-call). An ablation study confirms that each pipeline component independently contributes to the overall effectiveness. Addition-ally, the total cost per detected bug is approximately $1.7, and the static analysis portion of the pipeline runs 2.3–2.6 × faster than LeakGuard. These results demonstrate that MemHint effectively bridges the gap between semantic understanding and symbolic reasoning, enabling precise and scalable memory leak detection in real-world C/C++ software. In summary, our paper makes the following contributions: 

• We are the first to frame memory leak detection as a neuro-symbolic task: LLMs infer custom memory-management func-tions from code context, while Z3 validates the reachability of the inferred functions and filters infeasible leak warnings. 

• We demonstrate that LLMs can produce MM function summaries that encode allocator/deallocator semantics, and that these sum-maries can be injected into static analyzers to boost their detec-tion performance. 

• We implement MemHint , a neuro-symbolic pipeline that gener-ates and validates function summaries via LLM and Z3, injects them into CodeQL and Infer, and applies Z3-based path feasibility filtering and LLM-based validation to suppress false alarms. 

• We evaluate MemHint on seven real-world C/C++ projects and detect 52 unique memory leak bugs (47 confirmed/fixed, 4 CVEs submitted), outperforming vanilla CodeQL (19), vanilla Infer (3), Semgrep (2), and LeakGuard (20).

These techniques combine coverage-guided input generation with run-time sanitizers (e.g., LeakSanitizer [ 6]) that monitor memory alloca-tion/deallocation events along executed paths. However, dynamic approaches can only detect leaks along paths that the fuzzer ac-tually exercises, and may silently miss leaks that require specific branch combinations that are difficult to reach through crafted in-puts. However, current static analyzers still suffer from several limitations that MemHint is designed to address. The first and foremost limitation is that these tools cannot recog-nize custom memory management functions. Real-world software often implements project-specific MM functions that wrap stan-dard primitives such as malloc and free behind multiple layers of abstraction [ 12 , 59 ]. Although the underlying primitives are transi-tively reachable, static analyzers such as CodeQL [ 17 ], Clang Static Analyzer [ 41 ], and Infer [ 13 ] do not fully inline every function but instead rely on predefined function summaries, which are abstract descriptions of how a function affects memory (e.g., “ malloc takes a size argument and returns a heap-allocated pointer”), for a fixed set of standard primitives. For custom wrappers outside this set, the analyzer has no summary and thus cannot determine whether a call allocates or frees memory. Even if the analyzer attempted to look inside these wrappers for standard primitives, multi-layer del-egation and conditional logic would break dataflow tracking across call boundaries. As a result, the analyzer cannot establish that a wrapper’s return value carries heap ownership or that calling a wrapper constitutes a deallocation, leaving allocation/deallocation pairings invisible and memory leaks undetected. The final limitation lies in the static analyzers themselves. Even when a static analyzer is given an accurate memory model with comprehensive MM functions and their summaries, it may still report spurious warnings [ 20 , 25 , 31 , 48 ]. These tools are largely path-insensitive: they merge control-flow branches into a single abstract flow and cannot determine whether a reported warning is reachable under actual program semantics. A memory leak that appears along one branch may in fact be guarded by conditions that make the path infeasible at runtime, yet the tool reports it regard-less. For example, two branches may contain mutually exclusive conditions that the analyzer fails to recognize, causing it to report a leak on a path that can never execute. Without a mechanism to filter out such warnings with infeasible paths, users must manually triage a large number of false alarms, significantly reducing the practical utility of the tool [20, 35, 47]. 

MemHint addresses these limitations by combining LLMs’ se-mantic understanding of code with symbolic reasoning through constraint solving. MemHint first uses an LLM to infer the mem-ory management roles of custom functions from their source code and callees, producing function summaries that extend the ana-lyzer’s built-in knowledge beyond the standard primitives. Each summary is then validated by Z3 against the function’s intraproce-dural control-flow graph, ensuring that the identified allocation or deallocation is reachable on a feasible path. Additionally, MemHint 

encodes each analyzer warning as a path constraint and uses Z3 to discard those on infeasible paths, reducing false alarms before they reach the developer. Together, these steps form a neuro-symbolic pipeline in which semantic understanding and formal reasoning reinforce each other at every stage. 

## 3 MemHint 

As shown in Figure 3, MemHint operates in three stages. In Stage 1 (Summary Generation) , MemHint parses the target codebase to extract each function’s metadata (e.g., signature, body, callees), and filters functions via pointer-type heuristics. An LLM then classifies potential MM functions and produces structured function sum-maries that specify whether a function acts as a memory allocator or deallocator and which argument or return value carries own-ership. Each summary is validated with the Z3 SMT solver [ 8] to ensure the claimed allocation or deallocation is reachable on a fea-sible control-flow path. In Stage 2 (Summary-Augmented Analysis) ,the validated summaries are injected into CodeQL and Infer via their respective extension mechanisms, while in Stage 3 (Warning Validation) , MemHint applies Z3-based path feasibility checking and uses an LLM to filter false alarms and confirm genuine bugs. 

## 3.1 Stage 1: Summary Generation 

Phase 1: Code Extraction. MemHint begins by parsing the target codebase to extract metadata for each function. Specifically, we use Tree-sitter [ 2], a widely-used parsing library, to extract each function’s name, return type, parameter list with types, full source body, and the set of direct callee names appearing in call expressions. We also extract C preprocessor macros that expand to function-like constructs, converting them into the same record format so they can be analyzed for memory-management semantics. In addition, we collect pointer-typedef aliases, which are used in the subsequent pre-filtering step to recognize pointer types that do not contain an explicit * in the function signature. 

Phase 2: LLM Summary Generation. In this phase, MemHint 

classifies each candidate function as an allocator, a deallocator, or neither, and produces a structured function summary for each iden-tified MM function. A function summary is an analyzer-agnostic annotation that records the function’s MM role (allocator or deallo-cator) and which argument or return value carries memory own-ership; it can be directly translated into the extension formats of different static analyzers. To reduce LLM cost, we apply a light-weight pre-filter and retain only functions whose signatures in-volve pointer types (e.g., return type or parameters containing * or pointer-typedef aliases). Macros are always retained, as they may expand to pointer operations not visible in their signature. Entry-point and test functions (e.g., main , wmain , and functions whose names contain test ) are excluded. Each remaining function is analyzed by an LLM using its sig-nature, full source body, and the source code of up to five direct callees as context. Providing callee context is important because many project-specific memory-management functions delegate to helper functions, making their semantics non-local. Functions classified as neither are discarded. For each identified MM func-tion, the LLM produces a structured function summary contain-ing three fields: (1) name , the function’s name identifier; (2) role ,the function’s memory management role, either Allocator or 

Deallocator ; and (3) target , which specifies return for allocators (indicating the return value carries heap ownership) or arg 𝑖 for deal-locators (indicating the 𝑖 -th argument is freed). For example, given  

> freerdp_certificate_clone

Phase 3: Z3-Based Summary Validation. LLM-generated sum-maries require validation before use for two reasons. First, the LLM may misclassify functions; e.g., allocator-like names may not imply actual allocation, or control flow may obscure deallocation reachability. Second, operations such as malloc or free may occur only on infeasible paths, rendering the summary invalid. To ad-dress this, MemHint validates each summary using the Z3 SMT solver [ 8 ]. For each summary, MemHint constructs an intraproce-dural control-flow graph (CFG), encodes path conditions as con-straints, and checks whether a feasible path exists on which the claimed allocation or deallocation occurs in the MM function iden-tified by the summary. Summaries without such a feasible path are discarded, and the remaining ones are retained as validated sum-maries . ## 3.2 Stage 2: Summary-Augmented Analysis 

This stage injects the validated summaries into CodeQL [ 17 ] and Infer [ 13 ] and runs both analyzers to detect memory leaks. The validated summaries are analyzer-agnostic : each summary specifies whether a function allocates or frees memory and which argument or return value carries ownership, without relying on any tool-specific representation. We select CodeQL and Infer due to their complementary analysis foundations (declarative interprocedural dataflow and bi-abduction). In both analyzers, the summaries ex-tend the memory model to support project-specific allocators and deallocators without modifying the underlying codebase. For CodeQL, a declarative dataflow analysis engine that traces value propagation across function boundaries, MemHint injects the validated summaries through its extension mechanism, pop-ulating allocationFunctionModel and deallocationFunctionModel ,which are CodeQL’s built-in extension points for registering cus-tom MM functions. With these extensions, CodeQL recognizes each function in an allocator summary as returning newly allocated memory and each function in a deallocator summary as freeing the specified argument. However, CodeQL’s default memory leak queries (e.g., MemoryNeverFreed , MemoryMayNotBeFreed ) are largely path-insensitive: they merge control-flow branches into a single abstract flow and check only whether a deallocation exists some-where after an allocation, without distinguishing which branch performs it. As a result, a leak that occurs on a specific branch, such as an error-handling path that returns early without freeing, may be missed if another branch does free the memory. To address this, 

MemHint augments CodeQL with extended queries that (i) track deallocation on a per-branch basis, reporting a leak when any in-dividual branch fails to free the allocated memory; (ii) specifically model error-handling paths (e.g., goto cleanup labels and early re-turns after failed checks) as potential leak sites; and (iii) detect cases where deallocation is guarded by a condition that may not always hold, leaving the memory unfreed on the alternative path. For Infer, MemHint translates each validated summary into the configuration format of Infer’s Pulse engine [ 13 ], a memory and value analysis engine developed by Meta that is based on separation logic and tracks heap ownership along execution paths. Allocator summaries are registered via –pulse-model-alloc-pattern , so that Infer treats their return values as newly allocated heap objects with caller ownership; deallocator summaries are similarly registered so that calls to these functions are recognized as deallocation op-erations. With these summaries in place, Infer can track memory ownership through project-specific wrappers in the same way it handles standard primitives such as malloc and free . While Infer’s Pulse complements CodeQL’s declarative dataflow analysis, both analyzers may still report warnings on infeasible paths, which are addressed in Stage 3.

Phase 5: Z3-Based Path Feasibility Filtering. Static analyzers may report memory leaks along infeasible paths, i.e., paths that exist syntactically but are unreachable at runtime due to implicit invariants or mutually exclusive branch conditions. To address this,

Warnings without such a feasible path are discarded. Concretely, for each warning, MemHint parses the flagged func-tion and constructs an intraprocedural control-flow graph (CFG), following the same node classification as in Phase 3. • alloc (𝑛 ): memory for 𝑝 has been allocated at or before node 𝑛 ;

Phase 6: LLM-Based Warning Validation. After Z3-based fea-sibility filtering, the remaining warnings correspond to feasible execution paths but may still include residual false positives that intraprocedural symbolic analysis alone cannot eliminate. In par-ticular, Z3-based reasoning lacks sufficient semantic knowledge to model complex program behaviors such as callback-based dealloca-tion and project-specific conventions (e.g., borrowed pointers or non-owning references). For instance, in systems like Redis, certain functions may return pointers to internally managed data struc-tures, where ownership is retained by the callee and the caller is not responsible for deallocation; static analyzers may therefore incorrectly report a missing free as a memory leak. Such cases rely on implicit API usage conventions and program context that are difficult to capture precisely in constraint-based analysis. For each candidate warning, MemHint constructs a prompt that includes (i) the full source code of the function, (ii) the warning type (memory leak) together with the warning message and location, and (iii) the warning trace produced by the SAST tool, including code snippets at each step. The LLM is asked to determine whether the reported issue is a genuine bug or a false alarm, together with a brief justification. Warnings classified as false alarm are discarded, and the remaining ones are reported as the final output. The full prompt template is included in our replication package. 

## 4 Evaluation 

We evaluate MemHint on seven real-world C/C++ projects to iden-tify previously unreported memory-leak bugs. All warnings re-ported by the tool are manually inspected to determine whether they correspond to real bugs, and confirmed cases are reported to the respective repositories. Our research questions are: 

RQ1 (Effectiveness): How many memory-leak bugs does MemHint 

detect, and how does it compare with existing tools? 

RQ2 (Summary Quality): How accurate is the summary genera-tion in Stage 1? 

RQ3 (Component Contribution): How does each component of the pipeline contribute to overall bug detection performance? 

RQ4 (Efficiency): What is the runtime and computational cost of 

MemHint ?

## 4.1 Experiment Setup 

In Phase 2 (LLM summary generation), we use Gemini 3 Flash, while in Phase 6 (LLM warning validation), we use Gemini 3.1 Pro. We choose a lightweight model for summary generation because it needs to process a large number of functions, where efficiency and cost are critical, whereas warning validation requires higher accuracy and thus benefits from a stronger model. When invoking the LLM, we follow the batching strategy of Li et al.’s work [ 37 ] to reduce latency. Specifically, we batch multiple queries into a single prompt, using a batch size of 20 functions per call for summary generation (Phase 2). For LLM-based validation (Phase 6), we vali-date one function per call. We discuss the potential internal threats introduced by batching in Section 5. We use CodeQL CLI version 2.23.9, Infer version 1.2.0, and Z3 version 4.15.4. We evaluate MemHint on seven widely used open-source C/C++ projects: Vim, tmux, OpenSSL, Redis, FreeRDP, curl, and FFmpeg. For each project, we run the full MemHint pipeline (Stages 1–3) on the entire codebase. 

## 4.2 RQ1: Effectiveness 

Table 1 summarizes the bug detection results of MemHint across all seven projects. In total, MemHint detects 52 unique memory-safety bugs, of which 47 have been confirmed or fixed by project maintainers or through our submitted patches, and 4 have been sub-mitted as CVE reports currently under review. As CodeQL and Infer both receive the same validated summaries, they share 21 overlap-ping findings that both analyzers can detect once project-specific memory models are provided. However, CodeQL contributes 42 findings and Infer contributes 31, leaving 21 bugs found only by CodeQL and 10 found only by Infer. These non-overlapping cases arise from differences in analysis strategies: CodeQL’s declarative dataflow queries capture patterns that Infer’s bi-abduction misses, while Infer’s interprocedural path-sensitive reasoning detects cases beyond CodeQL’s scope. 

Comparison with Baselines. We compare MemHint against four baselines: vanilla CodeQL [ 17 ], vanilla Infer [ 13 ], LeakGuard [ 38 ], and Semgrep [ 56 ]. Vanilla CodeQL and vanilla Infer serve as the most direct baselines, representing what off-the-shelf compilation-level SAST tools can detect without custom memory-model an-notations. LeakGuard is a state-of-the-art memory-leak detection approach that infers paired memory-management functions and integrates them into static analysis. We use the authors’ released implementation, and discuss replication correctness in Section 5. Semgrep is a widely used source-level SAST tool that does not require compilation; we include it to provide a comparison with a pattern-based approach operating at a different abstraction level. ## 4.3 RQ2: Summary Quality 

To evaluate summary generation, we construct a human-labeled dataset of functions drawn from the subject projects, partitioned into three ground-truth categories: (1) MM + Reachable : custom MM functions whose allocation or deallocation is on a feasible control-flow path; (2) MM + Not Reachable : custom MM functions that contain allocation or deallocation calls, but the call site lies on an infeasible path (e.g., guarded by an unsatisfiable condition); and (3) Not MM : functions that are not memory-management functions. This three-way split allows us to evaluate the LLM and Z3 compo-nents separately: the LLM should classify both MM categories as MM functions and reject Not MM; Z3 should retain MM + Reachable summaries and discard MM + Not Reachable summaries.

## 4.5 RQ4: Efficiency 

Table 6 presents the cost and runtime breakdown of MemHint .Summary generation dominates both cost and runtime ($80.36 in total, approximately $11.5 per program, and about 2.1 hours per program), as it must process all candidate functions across the codebase. Warning validation for each analyzer’s output incurs minimal overhead ($2.71 for CodeQL and $3.18 for Infer), due to the aggressive filtering performed by Z3 in earlier stages. A key advantage of this design is that summary generation is a one-time cost; once generated and validated, summaries can be reused across multiple analyzers, project versions, and repeated analyses without regeneration. In practice, summaries only need to be regenerated when significant code changes occur (e.g., new custom allocators or major API refactoring); incremental updates between releases require no re-generation. This design amortizes the upfront LLM cost over all downstream uses. The static analysis phases (Phases 1, 3, 4, and 5) require only 37 minutes per program for CodeQL and 41 minutes for Infer on average, confirming that the pipeline’s recurring cost is modest. Compared to LeakGuard, which averages 96 minutes per program, MemHint achieves a 2.3-2.6 × speedup in the static analysis phase. Overall, MemHint demonstrates high efficiency: the most expensive step, summary generation, is shared and reusable, while all downstream stages are fast and inexpensive. 

## 5 Threats to Validity 

Threats to Internal Validity. Incorrect usage or misconfiguration of baseline tools (i.e., LeakGuard [ 38 ]) may lead to unfair compar-isons. We adopt the authors’ released implementation and evalu-ate it on six overlapping repositories (Vim, tmux, OpenSSL, Redis, FreeRDP, and FFmpeg) using the same versions as in the origi-nal study. Overall, we reproduce 34 out of 45 (76%) reported bugs. In particular, we achieve 100% reproduction for Redis and tmux, demonstrating faithful replication of LeakGuard’s results. In addi-tion, our batching strategy (multiple functions per prompt) may introduce context interference and affect LLM outputs. We miti-gate this by applying subsequent Z3-based validation to filter out infeasible cases, reducing the impact. 

Threats to Construct Validity. Our evaluation relies on the num-ber of detected and confirmed bugs. All detected bugs are manually verified and reported to the corresponding repositories via issues or pull requests. However, confirmed bugs depend on maintainers’ responses, which may be delayed or incomplete, potentially under-estimating effectiveness. To mitigate this, we report both detected and confirmed bugs. In addition, the evaluation of summary quality depends on the construction of the human-labeled dataset, which may not fully represent all allocation and deallocation patterns. To reduce this risk, we select a substantial and balanced set of func-tions across projects, using a similar number of samples from each. Furthermore, human labeling may introduce subjective bias. We mitigate this by involving two annotators who discuss and resolve disagreements to improve consistency. The two authors achieved an “almost perfect” level of agreement [ 64 ], with a Cohen’s 𝜅 of 0.94, demonstrating that the impact of subjective bias is negligible. 

Threats to External Validity. Our evaluation is conducted on seven C/C++ projects, which may limit the generalizability of the re-sults to other projects or programming languages. To mitigate this, we select a diverse set of widely-used projects spanning different sizes and application domains, thereby enhancing the representa-tiveness of our evaluation. 

## 6 Related Work 

Memory Leak Bug Detection The detection of memory leaks has evolved through various static and dynamic methodologies. Static approaches encompass abstract interpretation and model checking in tools like Sparrow [ 30 ] and Infer [ 13 ], Boolean satisfia-bility modeling in SATURN [ 68 ], and symbolic execution engines like KLEE [ 3] that track dynamic object lifecycles along program paths. A significant research direction simplifies leak detection to graph reachability on sparse value-flow graphs (SVFG), as demon-strated by Saber [ 62 ], FastCheck [ 5 ], Pinpoint [ 61 ], LeakFix [ 16 ], and SMOKE [ 14 ]. Specialized static frameworks further address spe-cific defect patterns, including partial call-path analysis in PCA [ 36 ], intra-procedural inconsistencies in error-handling code via Hec-tor [ 55 ], and practical memory ownership models in Clouseau [ 23 ]. Complementing these are dynamic monitors such as Purify [ 22 ], AddressSanitizer [ 58 ], and Valgrind’s memcheck [ 49 ], which in-strument code to track memory states during execution, as well as hardware-assisted instruction sampling in Sniper [ 29 ]. There are also studies focused on identifying project-specific custom memory management (MM) functions using heuristics, NLP, or usage-aware rules, as seen in Goshawk [ 42 ], K-MELD [ 12 ], our baseline Leak-Guard [ 38 ], and others [ 4, 39 , 40 , 65 ]. However, these approaches rely on surface-level signals and do not leverage LLMs’ semantic understanding of code. In contrast, MemHint combines LLM-based annotation with Z3-based symbolic validation to verify the reacha-bility of inferred annotations and filter infeasible leak paths. 

LLM-assisted Static Analysis Large Language Models (LLMs) have been integrated into static analysis to bridge the semantic gap between code structures and developer intent. A primary applica-tion is the automated inference of specifications and annotations, where frameworks like IRIS [ 37 ] and LAMeD [ 59 ] extract source and sink metadata to guide external analyzers and reduce path explosion. Beyond specification generation, LLMs are utilized for contextual warning triage to mitigate the high false discovery rates of classical tools, exemplified by post-refinement systems like BU-GLENS [ 34 ] and ZeroFalse [ 26 ]. The research frontier has further shifted toward autonomous agentic auditing, where agents such as RepoAudit [ 19 ] and QRS [ 63 ] mimic human code auditors by navigating repositories on-demand to track data-flow facts across feasible program paths. SpecGen [ 43 ] and AutoSpec [ 66 ] lever-age LLMs to generate general specifications. Unlike these tools, 

MemHint frames memory leak detection as a multi-stage neuro-symbolic pipeline where LLM-generated function summaries are formally validated by the Z3 SMT solver for reachability before being injected into industrial analyzers like CodeQL and Infer.

This paper presents MemHint , a neuro-symbolic pipeline for detect-ing memory leaks in C/C++ programs. MemHint uses an LLM to in-fer the memory-management roles of custom functions, producing function summaries that extend the analyzer’s built-in knowledge beyond standard primitives.
