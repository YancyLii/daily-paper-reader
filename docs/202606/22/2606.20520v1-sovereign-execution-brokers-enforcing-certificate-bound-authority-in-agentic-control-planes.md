## 摘要
自主智能体正越来越多地连接到云、部署和数据控制工作流，但生产环境的变更权限不应驻留在非确定性的推理过程中。现有的访问控制机制授权身份，而保障层则认证所提议的操作；两者均无法在变更瞬间为已认证权限提供强制执行点。本文提出主权执行代理（SEB），这是一个面向证书绑定的智能体基础设施的运行时执行边界。SEB 接收由主权保障边界（SAB）颁发的证书，验证所请求的变更是否与已认证的执行契约匹配，检查有效性窗口、策略版本、撤销版本和实时状态漂移，铸造限定范围的执行身份，调用基础设施 API，并记录签名的决策和结果记录。通过将提议、准入和执行分离，SEB 将已认证权限转化为一种短暂、可撤销、可审计的运行时能力，前提是生产变更 API 拒绝非代理身份。我们介绍了 SEB 的执行模型、证书及重放验证谓词、限定范围的身份语义、防绕过部署模式、故障行为，以及一个具体的原型实现。我们在 AWS 和 Kubernetes 集群上评估了该原型，测量延迟开销、撤销传播、漂移检测和故障注入下的安全性。

## Abstract
Autonomous agents are increasingly connected to cloud, deployment, and data-control workflows, but production mutation authority should not reside inside non-deterministic reasoning processes. Existing access-control mechanisms authorize identities, while assurance layers certify proposed actions; neither alone provides a mandatory enforcement point for certified authority at the moment of mutation. This paper introduces the Sovereign Execution Broker (SEB), a runtime enforcement boundary for certificate-bound agentic infrastructure. SEB consumes certificates issued by the Sovereign Assurance Boundary (SAB), verifies that the requested mutation matches the certified execution contract, checks validity windows, policy epochs, revocation epochs, and live-state drift, mints scoped execution identity, invokes infrastructure APIs, and records signed decision and outcome records. By separating proposal, admission, and execution, SEB turns certified authority into a short-lived, revocable, auditable runtime capability, provided that production mutation APIs reject non-broker identities. We present the SEB execution model, certificate and replay-verification predicates, scoped identity semantics, bypass-prevention deployment patterns, failure behavior, and a concrete prototype implementation. We evaluate the prototype on AWS and Kubernetes clusters, measuring latency overheads, revocation propagation, drift detection, and security under fault injection.

### 2. 方法论：核心思想与关键技术细节

- **核心思想：提议、准入、执行三权分离**
  - **智能体**：仅负责提议，不得持有任何生产环境变更凭据。
  - **SAB**：负责准入，验证提案语义安全后签发证书（Ω）。
  - **SEB**：作为唯一的变更凭据持有者与强制执行点，在运行时拦截所有变更请求，验证证书有效性后，动态铸造短暂的、限定范围的执行身份，调用目标API并记录全过程。
- **关键技术细节：基于验证管线的运行时执行模型**
  - **SEB的核心接口**：`Execute (Ω, req, St, Platform) → D or O`，输入为SAB证书（Ω）、具体执行请求（req）、目标实时状态（St）和平台描述符，输出为签名的决策记录（D，表示拒绝）或结果记录（O，表示执行尝试）。
  - **形式化验证谓词链（V）**：SEB在铸造执行身份前，必须顺序通过8项检查：
    1. `Φsig`（签名验证）：验证SAB对Ω的数字签名有效性。
    2. `Φmatch`（契约匹配）：确认req中的操作、目标、参数与Ω中嵌入的合约（C）完全一致，并绑定证书ID（cid）和一次性随机数（nonce）。
    3. `Φtime`（有效性窗口）：确认当前执行时间在证书有效期内。
    4. `Φpolicy`（策略纪元检查）：确认证书签发时的策略版本（Pver）与当前活跃策略版本（Pactive）一致。
    5. `Φrev`（撤销纪元检查）：确认证书签发时的撤销纪元（ρrev）等于当前全局撤销纪元（ρactive），防止使用已被撤销的证书。
    6. `Φdrift`（状态漂移检查）：查询目标基础设施实时状态（St），与证书中记录的准入时证据状态（Eadmit）对比，计算差异距离是否超出合约C定义的容忍阈值（ϵC）。
    7. `Φreplay`（重放防护）：在数据库中以原子事务方式插入（cid, nonce）键值对，利用主键约束确保一个nonce只能被成功使用一次。
    8. `Scopeable`（作用域可执行性检查）：确认合约C中所有的约束（操作、资源、参数、时间）是否能被目标平台的原生IAM/RBAC或SEB自身强制执行的代理/准入控制器所覆盖。若不可行，则拒绝执行。
  - **限定范围的执行身份（IDexec）模型**：验证通过后，SEB通过平台适配器（如AWS STS、Kubernetes TokenRequest API）动态铸造一个短暂、最小权限的执行凭据，其权限被严格限定于：`操作`、`资源`、`参数`（通过代理/准入控制器）、`有效时间`、`证书与Nonce`（通过会话标签）、`策略与撤销纪元`（SEB侧验证）。
  - **防绕过部署模式**：基于“零常驻特权”原则，通过服务控制策略（SCP）和RBAC配置，确保目标平台API拒绝除SEB角色及其临时会话之外的所有主体的生产环境变更请求，强制所有智能体变异流量经过SEB。
  - **故障安全默认（Fail-Closed）**：当SEB无法联系撤销服务、漂移评估失败或任何检查未通过时，默认拒绝执行并签署决策记录。在发生崩溃后，采用保守策略，除非有幂等性证据证明原操作未执行，否则已消耗的nonce不得重试，需重新向SAB准入。

### 3. 实验设计

- **实验场景与基础设施**：在AWS `us-west-2`区域的Amazon EKS v1.28集群上进行，SEB以高可用模式部署在3个`m6i.xlarge` EC2实例上，使用Amazon RDS PostgreSQL v15作为账本后端。
- **评测基准（Benchmark）与对比方法**：实验针对两类典型工作负载——**K8s补丁（K8s Patch）和AWS安全组更新（AWS Security Group Update）**——将“完全版SEB”与以下基线配置进行了端到端延迟比较：
  - **直接IAM/RBAC**：无任何运行时检查。
  - **静态策略代理（OPA/Gatekeeper）**：仅执行策略代码检查。
  - **仅审计日志**：仅记录操作，不进行阻断。
  - **无强制执行SAB**：有证书但不通过SEB强制执行。

  2. **撤销传播延迟（RQ2）**：测量从全局撤销纪元递增到SEB100%拒绝请求的时间差。
  3. **凭据铸造成本（RQ3）**：测量AWS STS和K8s TokenRequest的API调用延迟。
  4. **漂移检测效果（RQ4）**：在证书签发后、执行前，通过后台修改目标资源，评估SEB的漂移检测率。

### 6. 主要结论与发现

- **安全性定理**：在目标平台拒绝非代理权限的部署前提下，任何非受信的智能体运行时都无法绕过SEB，其引发的任何生产环境变更都必须与一份有效、未过期、未撤销的SAB证书相匹配，且必须在实时状态漂移可容忍和作用域可执行的条件下。
- **性能开销可接受**：SEB为Kubernetes变更增加约28.2ms（服务端）和40.7ms（客户端端到端）的p50延迟；为AWS安全组变更增加约136.9ms（服务端）和221.9ms（客户端端到端）的p50延迟。这种量级的开销对于高风险、低频次的云基础设施变更操作是完全可以接受的。
- **性能瓶颈在外部I/O**：延迟主要来自AWS STS凭据铸造（占AWS开销34.1%）和实时状态漂移查询（占AWS开销62.3%），而非SEB内部的密码学或逻辑验证（<1%）。
- **高安全性，低性能适用**：SEB不适用于微秒级的高频控制循环（如实时CPU调度），但在云控制平面变更等场景下，提供了一种以可控性能代价换取强制安全性的工程化方案。

### 7. 优点

- **创新性架构**：明确识别并解决了“证书绑定权限”在运行时缺乏强制执行点的关键安全缺口，创造性地将类似Simplex安全外壳的思想引入到云基础设施变更路径。
- **理论完善**：对执行过程进行了严格的形式化定义和谓词分解，安全属性和定理有清晰的证明框架，增强了系统的可信度。
- **全面的工程化实践**：设计了一套完整的工程实现方案，包括防绕过的部署架构、限定范围的身份铸造、可插拔的平台适配器以及故障安全处理机制，为生产化部署提供了蓝图。
- **实验评估深入**：不仅验证了功能的正确性（100%攻击阻断率），还对性能进行了精细的成本分解，清晰揭示了系统开销的构成和瓶颈所在，为性能调优指明了方向。

### 8. 不足与局限

- **高度依赖平台信任**：安全定理建立在“云厂商IAM/RBAC是完美且无漏洞”的强假设之上。如果云平台自身存在绕过漏洞，SEB的安全基石将被动摇。
- **引入单点故障（SPOF）与运维复杂性**：SEB作为强制流量必经点，其自身的可用性成为所有自主化变更的依赖。维护其高可用、跨多AZ部署需要显著的基础设施开销。此外，在遗留环境中推行零常驻权限需要大量复杂的策略重组工作。
- **性能不适用于高频场景**：受限于对外部API调用（状态查询和凭据申请）的高延迟依赖，SEB不适用于高频、低延迟的自动化控制回路。
- **无法保证语义层面的绝对安全**：SEB是执行时契约强制者，而非事实审查者。如果SAB的准入验证器出错或被收买，签发了内容本身不安全的证书，SEB会忠实地强制执行该恶意操作。这是保障层与执行层职责分离的固有结果。
- **状态观察的最终一致性问题**：漂移检查依赖查询云API状态，但云控制平面通常具有最终一致性，可能因状态传播延迟导致SEB基于过时状态做出错误判断（错误放行）。
