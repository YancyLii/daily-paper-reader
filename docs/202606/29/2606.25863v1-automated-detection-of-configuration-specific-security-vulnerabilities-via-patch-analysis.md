## 摘要
我们研究了高度可配置的 C/C++ 系统中的安全补丁如何映射到编译时变体空间。我们形式化了漏洞影响条件（VIC）——一个关于配置选项的布尔谓词，它表示所有包含原始缺陷的变体——并引入了 PatchLens，一种纯静态技术，通过将抽象语法树级别的补丁片段与源代码级别的存在条件对齐，并通过轻量级构建系统分析解析文件包含，来恢复 VIC。在 1,192 个 Linux 内核、289 个 FFmpeg 和 100 个 PHP 补丁上评估 PatchLens 后，我们计算出了精确、人类可读的 VIC，而无需编译任何系统变体。得到的谓词紧凑（Linux 平均 1.84 个变量，FFmpeg 平均 3.23 个，PHP 平均 1.04 个），并表明只有一小部分漏洞是系统范围的，这些漏洞具有更高的 CVSS 评分；同时，CVE 文本几乎从不编码所需的选项（平均召回率约为 1%），这促使了使用 VIC 自动丰富 CVE 描述。PatchLens 及其附带的数据集可以在 CI（变体感知的分类和测试选择）、有针对性的采样和模糊测试以及特性风险评分中立即应用，为高度可配置软件中的漏洞评估提供了一条可扩展、可解释的路径。

## Abstract
We study how security patches in highly configurable C/C++ systems map onto the space of compile-time variants. We formalize the Vulnerability Impact Condition (VIC) - a Boolean predicate over configuration options that denotes all variants that contained the original flaw - and introduce PatchLens, a purely static technique that recovers VICs by aligning AST-level patch hunks with source-level presence conditions and resolving file inclusion via lightweight build system analysis. Evaluating PatchLens on 1,192 Linux kernel, 289 FFmpeg, and 100 PHP patches, we compute precise, human-readable VICs without the need to compile any system variant. The resulting predicates are compact (avg. 1.84 variables for Linux, 3.23 for FFmpeg, 1.04 for PHP) and show that only a small fraction of vulnerabilities are system-wide, which carry higher CVSS scores; meanwhile, CVE texts almost never encode the required options ($\approx$ 1% average recall), motivating automated enrichment of CVE descriptions with VICs. PatchLens and the accompanying dataset enable immediate applications in CI (variant-aware triage and test selection), targeted sampling and fuzzing, and feature risk scoring, offering a scalable, explainable path to vulnerability assessment in highly configurable software.

## 1. 论文核心问题与研究动机

- **核心问题**：在高度可配置的 C/C++ 系统（如 Linux 内核、FFmpeg、PHP）中，安全漏洞的触发往往依赖于特定的编译时配置选项，即同一漏洞仅在部分变体中实际存在。
- **现实痛点**：CVE 描述通常仅给出版本范围，几乎从不提及导致漏洞暴露的配置选项（文中发现平均召回率仅约 1%），导致维护人员难以准确判断哪些部署变体真正面临风险，造成不必要的恐慌或错误的安全优先级分配。
- **研究目标**：提出一种完全自动化的静态分析方法，从安全补丁中直接恢复**漏洞影响条件（VIC）**，即一个关于配置选项的布尔谓词，精确刻画所有受影响的内核变体，从而支撑 CI 集成、定向模糊测试、特性风险评分等下游任务。

## 2. 方法论

- **总体思路**：PatchLens 基于两个关键观察：
  1. 安全补丁必须触及存在缺陷的代码区域，而该区域在变体中的“存在”由预处理器条件和构建系统共同决定。
  2. 不必编译系统，只需结合 AST 级补丁片段与源级存在条件，并解析构建脚本中的文件包含逻辑，即可推导出 VIC。

- **技术流程**（图 2、图 3 举例）：
  1. **补丁解析**：使用 `git format-patch` 作为输入，提取每个文件的改动 hunks。
  2. **AST 对齐**：利用 Tree-sitter 对**未预处理**的 C 源文件前后版本解析出语法树，通过比较两棵树将 hunks 映射到不同的 AST 子树。
  3. **源级存在条件提取**：对于每个影响子树，向上遍历至翻译单元根，收集所有包围该代码的 `#ifdef`/`#if`/`#ifndef` 等预处理器指令条件，将其合取（conjunction）构成该子树的存在条件。同时会过滤掉头文件保护宏（header guards）。
  4. **构建系统条件注入**：在源级条件中插入特殊谓词 `building("path/to/file")`，由项目特定的构建解析器（Linux 的 Kbuild、PHP 的 Autotools、FFmpeg 的 Make）计算文件级存在条件，并与源级条件合取，得到该 hunk 的总条件。
  5. **Hunk 条件合并**：一个补丁可能修改多个文件/hunks，将这些 hunk 级别的条件取析取（disjunction），得到最终的 VIC：*只要任一 hunk 条件成立，该变体即受影响*。
  6. **简化与输出**：使用 RESTRICT、z3 等工具对布尔表达式进行最小等价化简，生成人类可读的谓词。

- **构建系统解析适配**（核心挑战）：由于缺乏统一构建框架，PatchLens 针对三个项目开发了轻量级的“由内向外”解析器，避免全局构建分析：
  - **Linux**：从目标文件所在目录开始，向上逐层解析 Kbuild/Makefile，合取目录级守卫，支持 tristate（`y`/`m` 均为真）。
  - **PHP**：利用 Autotools 的常见模式（`PHP_NEW_EXTENSION`、`PHP_ARG_ENABLE` 等），从 `config.m4`/`configure.ac` 中提取 `--enable`/`--with` 标志，保守地过近似无法解析的项。
  - **FFmpeg**：搜索所有 Makefile 中的 `OBJS-$(CONFIG_X)` 等模式，建立目标文件到配置选项的映射，并传播程序级守卫，最终形成组件级守卫。
- **保守性设计**：当解析遇到非标准构造而无法安全确定条件时，工具会**过近似**（假定文件无条件包含），宁可产生包含额外变体的 VIC（可能假阳性），也不遗漏真正受影响的变体（避免假阴性），以符合安全分诊需求。

## 6. 主要结论与发现

- **自动化与准确性**：PatchLens 能以约 88%～97% 的成功率快速计算 VIC，无需编译任何变体。人工验证未发现不正确的 VIC（仅有少量安全过近似）。
- **漏洞影响范围**：绝大多数漏洞非系统范围（Linux 仅 6.24% 的补丁 VIC 为永真式，FFmpeg 为 0%，PHP 为 39.7%），反映各项目变异性设计差异。
- **严重性关联**：影响所有变体的漏洞（系统范围）拥有显著更高的平均 CVSS 影响评分（+1.18），且差异统计显著。
- **CVE 信息贫乏**：仅约 1.50% 的 CVE 描述提及至少一项所需配置选项，平均选项召回率仅 0.90%，表明可大幅通过 VIC 增强 CVE 元数据。
- **VIC 特性**：VIC 极为紧凑（Linux 平均 1.84 个变量，FFmpeg 3.23，PHP 1.04），主要使用合取和析取，少量使用否定。高频选项与各系统的核心子系统高度吻合（如 Linux 的 `CONFIG_NET`、FFmpeg 的 `CONFIG_AVCODEC`、PHP 的 `PHP_LIBXML`）。

## 7. 优点

- **无需编译**：纯静态方法，不依赖编译器或构建环境，极大降低了使用门槛。
- **可解释性强**：输出的布尔 VIC 人类可读，可直接用于配置条件判断和文档增强。
- **安全偏向的保守设计**：在不确定性条件下宁可过近似（假阳性），不产生假阴性，适合安全分诊场景。
- **构建系统适配灵活**：为三个完全不同的构建机制（Kbuild、Autotools、GNU Make）提供了轻量、由内向外的解析策略，且展示了扩展可能性（每种新构建系统只需约 200–400 行代码）。
- **丰富的应用前景**：不仅能丰富 CVE 描述，还可指导 CI 测试选择、定向模糊测试、特性风险评分和软件物料清单（SBOM）增强。
- **可复现**：提供完整代码、数据及实验说明。

## 8. 不足与局限

- **语言与变体机制限制**：仅支持 C/C++ 的编译时变体（预处理器+构建系统），不适用于运行时变体、Java/Python 等动态语言项目。
- **头文件影响缺失**：当前未分析补丁修改头文件时对多个翻译单元的影响，可能导致 VIC 低估受影响范围。
- **构建系统覆盖不全**：对非标准构建脚本或复杂宏结构会解析失败，导致保守过近似或无法生成条件（如 PHP 仍有 32% 未成功）。
- **补丁来源偏差**：训练数据集主要依赖 GitHub 提交与 PatchDB，可能遗漏通过邮件列表、专有补丁等其他渠道修复的漏洞，影响系统类型分布的代表性。
- **验证规模**：虽然手工审查了 500 个样本，但相对于成千上万的补丁总量，仍有大部分 VIC 未经严格手工核实，尤其是那些自动通过但可能包含微妙错误的案例。
- **上下文深度**：解析器基于 Tree-sitter 轻量处理，对极其复杂的宏展开或条件编译陷阱可能无法完美处理，精度依赖错误容忍机制。
