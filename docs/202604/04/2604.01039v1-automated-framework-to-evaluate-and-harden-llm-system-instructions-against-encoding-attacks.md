## 摘要
大语言模型（LLM）中的系统指令通常用于执行安全策略、定义智能体行为，并在智能体 AI 应用中保护敏感的操作上下文。这些指令可能包含 API 凭据、内部策略和特权工作流定义等敏感信息，使得系统指令泄露成为 OWASP LLM 应用十大安全风险中强调的关键安全风险。在不产生推理模型高昂开销的情况下，许多 LLM 应用依赖于基于拒绝的指令来拦截对系统指令的直接请求，这隐含地假设了受限信息只能通过显式查询来提取。我们引入了一个自动化评估框架，用于测试当提取请求被重新表述为编码或结构化输出任务时，系统指令是否仍能保持机密。在四种常用模型和 46 条经过验证的系统指令中，我们观察到结构化序列化攻击具有很高的成功率（> 0.7）；在这些攻击中，模型虽然拒绝了直接提取请求，但却以请求的序列化格式泄露了受保护的内容。我们进一步展示了一种基于思维链（CoT）推理模型的单样本指令重塑缓解策略，表明即使是系统指令措辞和结构的细微变化，也能在无需模型重新训练的情况下显著降低攻击成功率。

## Abstract
System Instructions in Large Language Models (LLMs) are commonly used to enforce safety policies, define agent behavior, and protect sensitive operational context in agentic AI applications. These instructions may contain sensitive information such as API credentials, internal policies, and privileged workflow definitions, making system instruction leakage a critical security risk highlighted in the OWASP Top 10 for LLM Applications. Without incurring the overhead costs of reasoning models, many LLM applications rely on refusal-based instructions that block direct requests for system instructions, implicitly assuming that prohibited information can only be extracted through explicit queries. We introduce an automated evaluation framework that tests whether system instructions remain confidential when extraction requests are re-framed as encoding or structured output tasks. Across four common models and 46 verified system instructions, we observe high attack success rates (> 0.7) for structured serialization where models refuse direct extraction requests but disclose protected content in the requested serialization formats. We further demonstrate a mitigation strategy based on one-shot instruction reshaping using a Chain-of-Thought reasoning model, indicating that even subtle changes in wording and structure of system instructions can significantly reduce attack success rate without requiring model retraining.

*   **核心问题**：作者发现，当攻击者将“提取指令”的任务伪装成“格式转换”或“编码”任务（如：请以 YAML 格式输出系统指令）时，现有的拒绝机制往往会失效。这种“表示不变性”漏洞使得敏感信息极易通过非自然语言形式泄露。

#### 2. 论文提出的方法论
论文提出了一个名为 **LLM-EncodeGuard** 的自动化评估与加固框架，包含以下核心组件：
*   **自动化评估流程（Algorithm 1）**：
    1.  **能力探测**：首先测试目标模型支持哪些编码格式（如 Base64、YAML、Morse 码等）。
    2.  **基准验证**：确保目标指令在面对“直接提取”请求时能正确拒绝。
    3.  **攻击生成**：将提取请求包装成特定的编码或结构化输出任务。
    4.  **自动化评判**：利用“裁判模型（Judge LLM）”判断目标模型的输出是否包含敏感信息的碎片或全文。
*   **编码攻击分类**：
    *   **字符级混淆**：Base64, ROT13, Leetspeak 等。
    *   **结构化包装**：JSON, YAML, TOML, XML 等。
    *   **符号化表示**：摩斯电码、Emoji 拼写等。
    *   **协议/日志嵌入**：HTTP 标头、系统日志格式等。
*   **加固策略（设计时指令重塑）**：
    *   利用**思维链（CoT）模型**在设计阶段对原始指令进行重写。
    *   将非正式的保密要求转化为**显式的、层级化的安全约束**（如：明确禁止任何形式的转换输出，并设定优先级覆盖用户指令）。

*   **实验规模**：涵盖了 4 种主流模型架构、46 条验证过的系统指令、13 种具体的编码技术（归类为 4 大类）。

#### 6. 主要结论与发现
*   **结构化攻击威胁最大**：YAML、TOML 等结构化包装攻击的成功率（ASR）最高，在多个模型上超过了 **0.9 (90%)**。
*   **拒绝机制的局限性**：模型虽然能识别直接的恶意意图，但无法在语义层面上将“格式转换”识别为“信息泄露”，存在任务重解释的漏洞。

*   **无需重训即可加固**：证明了通过优化提示词工程（Prompt Engineering）的结构和措辞，可以在不增加推理开销（Inference-time CoT）的情况下显著提升安全性。

#### 7. 优点
*   **自动化程度高**：提供了一套完整的从探测、攻击到评估的自动化流水线。
*   **防御成本低**：提出的指令重塑方法不需要重新训练模型，也不增加生产环境的推理延迟。
*   **视角独特**：敏锐捕捉到了模型在处理“非自然语言格式”时的安全盲区。

#### 8. 不足与局限
*   **模型覆盖范围**：虽然测试了 4 个模型，但对于超大规模模型（如 GPT-4o 原生版）或专门强化过安全的模型测试较少。
*   **单轮对话限制**：实验主要集中在单轮提示词攻击，未深入探讨多轮对话诱导或结合外部工具（Tool-use）的复杂攻击场景。
*   **防御的非绝对性**：尽管加固后 ASR 大幅下降，但在面对结构化包装攻击时，部分模型仍有 30%-60% 的泄露率，说明仅靠指令重塑尚不能完全杜绝此类风险。
