Abstract вЂ”LLM-based vulnerability detectors have shown promising results in identifying memory-safety bugs and well-established vulnerability classes, where violations can often be expressed in terms of established security properties such as unsafe data propagation, bounds violations, or invalid memory accesses. Logic vulnerabilities, however, pose a fundamentally different challenge, as their identification requires inferring application-specific security invariants and often relies on implicit assumptions about intended behavior. Even frontier agentic models struggle in this setting, despite their ability to inspect and traverse large repositories, because the security invariants relevant to these vulnerabilities are often implicit and buried among large amounts of unrelated code. Moti-vated by this gap, we present A NTAEUS , a framework for detecting logic vulnerabilities that grounds LLM reasoning in repository-level code context. A NTAEUS follows a repository-scale pipeline that combines function prioritization, context-grounded reasoning, comparative validation, and structured reporting. First, it ranks functions using lightweight repository-wide security signals, directing costly LLM analysis toward the most relevant code regions and reducing model calls, cost, and triage effort. For each prioritized function, A NTAEUS grounds the model in explicit repository evidence, combining local code context with a repository-level view of the applicationвЂ™s functionality, security-relevant resources, and trust boundaries. This grounded input enables the model to reason about how the function is executed within the broader application rather than as an isolated snippet. A NTAEUS then identifies security-sensitive sinks, derives the safety conditions required for safe execution, and checks whether those conditions are locally satisfied. Candidate findings are subjected to comparative vali-dation, which prunes concerns that reflect project-wide norms rather than distinctive violations. Finally, A NTAEUS reports the sinks, the violated safety conditions, and the supporting evidence, making findings specific, actionable, and traceable. We evaluate ANTAEUS on 28 real-world repositories with confirmed logic vulnerabilities and compare it against both function-level LLM analysis and frontier agentic models, in-cluding Opus 4.8 Agentic and Codex 5.4. A NTAEUS detects and explains 15 vulnerabilities, substantially outperforming state-of-the-art baselines while maintaining a comparable token usage and cost budget. 

1. Introduction 

Logic vulnerabilities arise from failures to enforce the application-specific security invariants that govern intended program behavior [1]. Unlike memory-safety bugs, which often manifest through invalid memory accesses, or dataflow vulnerabilities, which can often be expressed as propaga-tion from sources to sinks, logic vulnerabilities stem from missing, misplaced, or incorrectly composed enforcement of application-specific security invariants [2], [3]. These invari-ants define who may perform an operation, which resources may be accessed, what information may be exposed, and under which workflow conditions an action is safe. Their detection depends on reconstructing the intended security invariants implicitly encoded by the surrounding codebase, rather than matching local code against known dangerous patterns. Logic vulnerabilities have long been recognized as an important but comparatively underexplored class of software defects [4]. Most automated vulnerability-detection techniques have focused on bug classes with clearer syn-tactic or semantic anchors, such as memory-safety errors, injection flaws, or tainted source-to-sink flows. Traditional pattern-based and taint-based tools, such as CodeQL [5] and Semgrep [6], are effective when a vulnerability can be tied to a recurring code shape, a known dangerous operation, or a traceable source-to-sink flow. For logic vulnerabilities, however, there is often no universal source, canonical sink, or distinguished value to track. Although project-specific analyses can sometimes encode these invariants manually, building and maintaining them requires substantial expertise and does not scale across repositories. Existing work on logic vulnerabilities has made impor-tant progress, but it has largely studied web applications [7], [8]. White-box approaches [1] infer application invariants from normal executions and check whether alternate pro-gram paths violate them, while black-box approaches [9], [10] infer behavioral models from HTTP traces and gener-ate tests that break observed workflow or data-propagation patterns. These works show that logic vulnerabilities vary by application and that detection depends on recovering intended security invariants rather than matching fixed code patterns. However, their assumptions remain tied to the web domain. They ground the analysis in requests, sessions, forms, URLs, navigation graphs, database-backed state, and

user workflows. The problem remains less explored for repository-level C and C++ software. In this setting, security invariants are harder to recover because they are often distributed across large codebases and encoded through APIs, callbacks, flags, resource handles, configuration interfaces, and project-specific conventions. The evidence needed to judge a func-tion is rarely contained in the local code alone. A function may look benign in isolation, while its safety depends on repository conventions, trust boundaries, or checks enforced by comparable code. Detecting logic vulnerabilities in this setting, therefore, requires recovering security invariants from code structure and repository context, rather than from web interaction semantics. Recent work applies Large Language Models (LLMs) to vulnerability detection [11], [12], including agentic systems that can inspect repositories, navigate code, and produce natural-language explanations. These systems are promising because frontier models can reason about program intent, integrate heterogeneous evidence, and discover real vul-nerabilities when appropriately scaffolded [13], [14]. The limitation is not the intrinsic modelвЂ™s incapability, but the way evidence is selected, structured, and budgeted during reasoning [15]. Function-level analysis provides too little context, forcing the model to infer project-specific secu-rity invariants from an isolated code fragment. Exhaus-tive repository exploration, on the other hand, can provide the missing evidence, but only by spending a substantial analysis budget on large amounts of irrelevant code. Even when effective, such search-heavy strategies are difficult to scale and can still cause the model to lose focus on the specific semantic facts needed to assess a candidate operation. The key issue is therefore context quality and cost, not simply context quantity. LLM-based vulnerability reasoning remains breakable without structured program-semantic support [16], [17], especially when interprocedural dependencies and vulnerability-triggering constraints must be inferred from raw code alone. At the same time, recent systems show that LLMs can be effective when embedded in pipelines that select relevant context, structure the reasoning task, and validate candidate findings [18]. Yet effectiveness cannot be measured only by whether a model eventually finds a bug. Recent work [19] argues that the relevant cost includes the full path from candidate generation to valida-tion, impact assessment, remediation packaging, and main-tainer triage. Cheap candidate generation can therefore shift the bottleneck toward review and remediation rather than eliminating the cost of vulnerability discovery. For logic vulnerabilities, this means providing sufficient repository evidence to recover application-specific security invariants, without so much unfiltered code that the model loses focus or consumes an impractical analysis budget. We argue that repository-level logic vulnerability detec-tion requires context grounding, where model reasoning is anchored in explicit evidence about how the project defines principals, resources, trust boundaries, and security-sensitive operations. Without this grounding, model-based detectors risk relying on local syntax or generic security priors, which are insufficient for vulnerabilities whose defining property is a project-specific security invariant violation. In our setting, context grounding denotes the process of recovering evi-dence necessary to reconstruct repository-specific security invariants. This notion differs from simply increasing the amount of repository context available to the model. Addi-tional context is useful only insofar as it helps explain why a particular security check exists, under what assumptions it operates, and which repository components collectively en-force the underlying security invariant. We therefore frame repository-level logic vulnerability detection as an invariant-recovery problem rather than a pure retrieval or reasoning problem. The central question is not how much repository context can be supplied to a model, but whether that context can be used to reconstruct the security assumptions under which the repository is intended to operate, and then to judge a candidate operation against them. Based on this observation, we present ANTAEUS , a repository-level framework for detecting C and C++ logic vulnerabilities through context-grounded model reasoning. It incrementally recovers repository-specific security invariants from local and project-wide evidence and evaluates each candidate violation against the invariants it has reconstructed rather than against generic priors or local syntax alone. Rather than analyzing every function in isolation, A NTAEUS 

organizes detection as a staged pipeline. It first performs lightweight repository-wide prioritization to identify func-tions likely to involve security-relevant behavior, reducing the cost and triage burden of model analysis. It then con-structs a grounded context for each prioritized function, combining local code augmentation with a stable repository-level security summary. Next, it asks the model to produce structured findings in terms of security-sensitive sinks, re-quired safety conditions, and whether those conditions are locally satisfied. Finally, it applies comparative validation across similar sinks in the same repository, pruning findings that reflect recurring project-wide norms rather than distinc-tive violations. The final output is a triage-oriented report that exposes the sink, the violated safety condition, and the supporting repository evidence, so that each finding can be inspected, justified, and addressed by a human reviewer. We evaluate A NTAEUS on 28 real-world C and C++ repositories containing confirmed CWE-200 and CWE-284 logic vulnerabilities and compare it against both function-level LLM analysis and frontier agentic baselines, including Opus 4.8 Agentic [20] and Codex 5.4 [21]. A NTAEUS iden-tifies 15 vulnerabilities and provides structured explanations for the missing security conditions, substantially outper-forming the comparison baselines. These results show that targeted context grounding can recover repository-specific security invariants more effectively than broader agentic exploration, while staying within a comparable token-usage and cost budget. 2. Motivation 

2.1. Why These Classes Resist Detection 

To illustrate the underlying challenges of logic vul-nerability detection, we use CVE-2020-10701, a CWE-284 access-control flaw in libvirt, as a motivating ex-ample because it compactly captures the difficulties this class poses. As shown in Listing 1, the pre-patch body of 

virDomainAgentSetResponseTimeout is the public entry point for configuring how long libvirt waits for QEMU guest-agent replies. It validates the domain handle, retrieves the connection, dispatches to the per-driver callback, and handles errors through the usual path. Every line is lo-cally correct. The vulnerability is the absence of a sin-gle guard, virCheckReadOnlyGoto(conn->flags, error) , which would reject read-only callers. Without it, a read-only client can set the timeout to zero and induce a denial-of-service. While a human auditor connects this missing check to libvirtвЂ™s read-only connection model and the conventions of comparable APIs, automating the same judgment requires recovering invariants that the function body never states. The following subsections detail three challenges that motivate context grounding in A NTAEUS .

2.1.1. Non-Local Security Invariants. Many logic vul-nerabilities are not confined to the function under anal-ysis. The defining evidence lives in the repository around it. The safety of a function depends not on its local syntax, but on invariants the codebase as awhole is meant to enforce. As Listing 1 illustrates, 

virDomainAgentSetResponseTimeout reads as a standard libvirt API wrapper, and no line within it signals that a read-only check is missing. The violated invariant, that write-privileged public APIs must reject read-only callers, is encoded in libvirtвЂ™s connection model and in how sibling APIs guard the same boundary, not in any artifact reachable by reading the function alone. Connecting the operation to the policy it violates requires reconstructing repository-level conventions, evidence that local context cannot supply.

2.1.3. Distinguishing Violations from Project Norms. 

Detecting a logic vulnerability requires determining not only that a safety condition is unmet, but whether the unmet condition is a genuine anomaly rather than a repository-wide convention. An LLM model evaluating a function locally is usually assertive [22], [23]. Shown a sensitive-looking operation, it tends to raise the safety conditions appropriate to that kind of operation, regardless of whether the surrounding codebase already addresses them. Across a repository, this yields a characteristic false-positive pattern, in which the same unsatisfied condition is reported for many structurally similar sinks, often in similar phrasing. A genuine logic vulnerability, by contrast, is an anomaly [24], [25], a deviation from how comparable code in the same project handles the same operation. When many similar operations are handled uniformly, that uniformity is the norm rather than a defect, and a model that raises the same concern across all of them is flagging the norm rather than a distinctive violation. As Listing 1 shows, the missing read-only check is meaningful only relative to the sibling APIs that do enforce the boundary. A suspected violation, therefore, remains only a hypothesis until it can be set against the projectвЂ™s enforced pattern. 2.2. Insights for Context-Grounded Logic Vulner-ability Detection 

The challenges above motivate context grounding, which constrains model reasoning to explicit evidence about how a repository defines its security invariants rather than letting the model fall back on local syntax and generic priors. Grounding combines two complementary layers. Local pro-gram evidence ties claims about a function to concrete repository artifacts, and repository-level context supplies the application-specific invariants against which a local opera-tion must be judged. This perspective leads to three design insights. Insight 1 : Prioritize candidates before grounding. Because logic vulnerabilities admit no syntactic anchor, every function is a nominal candidate, but grounding each one is impractical. Detection should begin with repository-wide prioritization that surfaces functions likely to involve security-relevant behavior, spending the analysis budget only where a policy violation could plausibly occur. Insight 2 :Pair local evidence with repository context. The information needed to judge a function lies outside it, so the model should reason over both layers simultaneously. Local code, callees, macros, constants, and types anchor claims to con-crete artifacts, while a repository-level security summary of the applicationвЂ™s purpose, protected objects, and trust topology supplies the invariants the operation must satisfy. Insight 3 : A missing safety condition is meaningful only relative to how comparable code in the same repository han-dles the same operation. Each flagged condition should be validated against analogous sinks, discarding those that recur uniformly as project norms and retaining those that set the target apart. These insights shape the design of A NTAEUS ,a staged framework that grounds LLM-based logic vulnera-bility detection in repository evidence. A repository-wide prioritizer narrows the candidate set to security-relevant functions; a grounding stage pairs local code augmentation with a stable repository-level security summary; a structured reasoning stage produces findings in terms of security-sensitive sinks and their required safety conditions; and a comparative validator confirms genuine violations by setting each finding against analogous code. The following section details each stage. 

2.3. Threat model 

We consider an adversary who interacts with the target software through its intended interfaces, such as API calls, configuration options, files, command-line arguments, or network requests. The adversary cannot modify the program code, but can choose inputs, parameters, call sequences, and execution contexts to violate security-relevant assumptions made by the application. We focus on logic vulnerabilities in which the executed code is syntactically valid, but an operation is performed under semantically unsafe condi-tions. A NTAEUS assumes access to the source repository under analysis, but does not assume exploit inputs, runtime traces, or an execution environment. Its goal is to stati-cally detect and explain candidate logic vulnerabilities by linking security-sensitive code to the repository context that determines whether its use is safe. Memory-safety bugs and purely syntactic defects are out of scope. 

3. Design 

The design follows directly from the insights identified in Section 2.2. A NTAEUS is a five-stage repository-level pipeline, shown in Figure 1: function prioritization, context-grounding, structured reasoning, comparative validation, and reporting. These stages also reflect recent state-of-the-art practice in LLM-based vulnerability detection [14], [18]. First, function prioritization ranks functions using lightweight repository-wide security signals, focusing LLM analysis on the most relevant code regions. Second, context-grounding supplies the model with local code augmentation and a repository-level security summary, tying its reasoning to concrete repository evidence. Third, structured reasoning prompts the model to identify security-sensitive sinks to-gether with the safety conditions required for their safe use, and to record whether each condition is locally satisfied. Fourth, comparative validation checks whether a flagged condition is distinctive with respect to similar sinks in the same repository, pruning conditions that recur as project norms. Finally, reporting presents the retained findings as actionable, evidence-grounded explanations. The remainder of this section examines each stage in detail. 

3.1. Prioritization 

At repository scale, the first challenge is deciding where to apply expensive LLM analysis. As discussed in Sec-tion 2.2, exhaustive repository exploration spends most of the budget on irrelevant code, while letting an agentic model search unaided does not guarantee that every security-relevant function is examined and may lead it to rely on superficial cues rather than repository evidence. ANTAEUS therefore begins with a lightweight, high-recall prioritization stage. The goal of this stage is not to classify functions as vulnerable, but to reduce the repository to a ranked set of functions that are most likely to contain logic-relevant behavior. This lets the later LLM stages oper-ate at the repository level while keeping the number of de-tailed analyses tractable. The prioritization stage first prunes the codebase, retaining only C/C++ source and header files. It then applies broad keyword-based heuristics to remove functions that are unlikely to involve authorization, access control, trust-boundary enforcement, configuration changes, or other security-sensitive operations. These heuristics are intentionally loose: they are designed to remove clearly irrelevant functions while preserving recall for functions that expose even weak security signals. After this initial pruning, A NTAEUS compresses the remaining repository into a lightweight function-level representation. For each retained function, it removes the internal implementation logic and keeps only the function signature together with the callees invoked inside the function body, as illustrated in Listing 2. 3.2. Contextual Grounding 

The contextual grounding stage constructs the evidence needed to interpret a function in its repository context. The unit of analysis is a single function, but a function does not stand alone. Its meaning depends on the types of values it receives, the bodies of the functions it calls, the expansion of the macros it uses, the constants it references, and the headers and imports that determine the names in scope. Asking an LLM to reason about a function in isolation forces it to fill in these blanks from general knowledge or from patterns seen during training. In real C/C++ projects, this is unreliable. A project-specific macro may not expand to what the model assumes, a wrapper API may not behave like the closest public-library analog, and a constant may be defined to a value that changes the meaning of a check. For this reason, each function is enriched with a local augmentation bundle extracted from the repository. We build this bundle with a lightweight static analysis pass based on Tree-sitter. The parser identifies function definitions, call sites, macro and constant references, and includes relation-ships, from which we construct a repository-level graph that links each analyzed function to the program artifacts needed to interpret it. This graph is not intended to be a complete semantic model of the program; rather, it provides an efficient index for retrieving nearby evidence across the repository without requiring full compilation or heavyweight whole-program analysis. The local augmentation bundle contains the signatures of direct callees at depth one, the bodies of those callees when they are defined within the project, the textual definitions of macros and constants referenced in the function, and the relevant typedefs and imports in scope. This material is supplied as explicit program evidence rather than left to the model. Claims about what an identifier denotes, what a callee does, or what a constant means can therefore be cross-checked against supplied code. The choice of depth one is deliberate. Expanding the call graph further increases prompt size and dilutes the modelвЂ™s attention, yet still fails to guarantee complete interprocedural coverage. In our setting, one level is sufficient to disambiguate the immediate seman-tics of the function and its directly invoked project APIs, while keeping the input within budget. Deeper semantic dependencies are instead captured, when relevant, through the repository context bundle. Local augmentation pins down what each line of the function means, but it does not explain what the program is for. Logic CWE-284 and CWE-200 vulnerabilities depend on repository-level semantics. Deciding whether a sink is safe requires understanding the kind of artifact the project implements, who interacts with it, the resources it mediates, the information it returns, and where its trust boundaries lie. None of this is reliably recoverable from a single function, even with depth-one expansion. We therefore compute, once per repository, a structured context bundle that summarizes the project at the level needed to interpret individual func-tions. The bundle is produced by a repository-level LLM agent before function-level analysis. The agent is given the complete repository and is asked to extract a stable descrip-tion of the projectвЂ™s purpose, actors, resources, outputs, and exposed interfaces. Its task is not to decide whether any function is vulnerable, but to provide the shared repository context that subsequent function-level prompts can rely on. Each item in the bundle is associated with repository-relative paths and short code excerpts, so that later reasoning can be tied back to concrete evidence rather than to the modelвЂ™s general prior knowledge. The bundle contains five sections. The system purpose 

section describes what kind of artifact the repository imple-ments, its deployment context, and its primary consumers. The principal model section identifies the actors or callers that the code appears to distinguish, together with how their identity is established and through which interfaces they reach the system. The protected objects section records repository objects that appear security-relevant, such as configuration state, service handles, credentials, filesystem paths, or shared state tables, without deciding which prin-cipals are allowed to access them. The information outputs 

section summarizes externally observable outputs, including functional responses, errors, status information, configura-tion data, and internal state exposed through return paths or response builders. Finally, the trust topology section records the interfaces exposed by the project and the structural boundaries they create, such as library APIs, IPC channels, local sockets, filesystem interfaces, or network-facing entry points. This shared bundle gives each function-level query the same repository-level frame while avoiding repeated re-discovery of the projectвЂ™s purpose, actors, resources, outputs, and trust boundaries. 

3.3. Structured LLM Reasoning 

The structured LLM reasoning stage uses the grounded input to produce a function-level finding. The LLM is not asked to return an unconstrained vulnerability label. Instead, it is queried for a fixed-format finding whose elements must be tied to observable code or to explicit repository evidence. This design also aligns with prior observations that explana-tions should be grounded in verifiable evidence rather than unconstrained post-hoc rationalizations [26]. This output format determines what the contextual grounding stage must provide: enough local and repository-level evidence for the model to identify security-sensitive operations and state whether the required safety properties are satisfied by the analyzed function. For each function analyzed, the model first identifies a set of sinks . A sink is an operation syntactically present in the function whose execution is potentially sensitive to the CWE under consideration. For CWE-284, a sink is any operation that affects a protected resource, such as a privileged system call, a write to a resource exposed across a trust boundary, or a state update performed on behalf of a principal. For CWE-200, a sink is any operation that emits data across a trust boundary into a region observable by a less privileged principal, such as returning data through an API, writing to a network channel, logging sensitive information, or exposing internal state through an externally visible interface. For each sink, the model then derives a set of safety conditions . A safety condition is a predicate over the sur-rounding code, local checks, or relevant calling context that must hold for the sink to be safe in this repository. Each safety condition carries a locally_satisfied flag and a short justification. Both must be tied to specific code in the function or to explicit material supplied by the contextual grounding stage. A function is reported as a candidate vulnerability when at least one safety condition of at least one sink is not locally satisfied. This scheme has three purposes. First, it anchors the modelвЂ™s reasoning in concrete evidence. By committing to specific sinks and checkable predicates, the model cannot hide a vague intuition behind generic prose, which mitigates the unfaithfulness of free-form LLM rationales. Second, the resulting output is directly useful to an auditor. The auditor is told which operation is security-sensitive and which predicate must be verified, rather than being handed a binary verdict to validate from scratch. Third, the format enables sharper evaluation. In fact, a finding is only accepted when both the vulnerability judgment and the supporting reasoning are accurate. Findings whose final label is correct but whose justification is inconsistent with the grounded evidence are discarded as incorrectly reasoned. 

3.4. Comparative Validation 

The comparative validation stage operationalizes the ob-servation in Section 2.1.3, that a genuine logic vulnerability is an anomaly relative to its peers while a recurring concern reflects a project norm. It reframes false-positive elimination as anomaly detection over the structured findings produced by the LLM. For each flagged condition, the stage compares the target against structurally similar sinks in the same repository, retaining the condition when it sets the target apart from its peers and discarding it when the same concern recurs uniformly across them. To decide whether an unsatisfied safety condition is distinctive or merely a recurring pattern, the validation stage compares the target sink against similar sinks from the same repository. Similarity is computed in two steps. First, each sink is represented by its sink identifier , i.e., the code fragment that corresponds to the security-sensitive operation reported by the model. We embed these sink identifiers with UniXcoder [27] and use cosine similarity to build a neighborhood of sinks that are syntactically or semanti-cally close at the code level. This step compares opera-tions, not model explanations. Second, validation compares the unsatisfied safety conditions attached to those sinks. Safety conditions are natural-language predicates produced by the model, so we embed them with all-MiniLM-L6-v2 sentence-transformer model and compare them using cosine similarity. A target condition is considered repeated when enough neighboring sinks have an unsatisfied condition that is semantically similar to it. In that case, the condition is treated as a repository-wide recurring pattern rather than as evidence of a distinctive logic violation. Both comparisons require a notion of similar enough .We avoid fixed global thresholds because similarity distribu-tions vary across repositories: some codebases contain many near-duplicate API wrappers, while others are more hetero-geneous. 3.5. Reporting 

The final stage emits the findings retained after com-parative validation as structured, evidence-grounded reports rather than free-form natural language. The report contains one entry per flagged function, where a function is flagged when it contains at least one security-sensitive sink with a required condition that is not locally satisfied. Functions whose sinks all meet their required conditions are not re-ported. This structure makes every claim auditable against concrete repository artifacts and exposes the results in a machine-readable form suitable for downstream triage and aggregation. As shown in Listing 3, an entry identifies the analyzed function by its name, file, line range, and a function 

id . For each function, it enumerates the security-sensitive sinks identified within it. A sink names the operation and gives a short description of why it crosses a trust or exposure boundary. Each sink carries a list of required conditions, where every condition states the invariant the operation depends on, whether that invariant is locally satisfied, and a justification grounded in the code that supports the judg-ment. An unsatisfied condition, together with its justifica-tion, constitutes the reported vulnerability evidence, and the presence of at least one such condition is what causes the function to appear in the report. This design is consistent with recommendations from the explainable security literature, which stresses that ex-planations for security applications should support analyst decision-making while remaining auditable, understandable, and grounded in the evidence used to produce them [26]. 

4. Evaluation 

The central question is whether structuring repository-level context the way A NTAEUS does actually helps an LLM uncover logic vulnerabilities, and whether the gains are worth their cost. We break this down into four research questions:  

> вЂў

RQ1 (Effectiveness). Can a multi-step LLM frame-work detect logic vulnerabilities at the repository level?  

> вЂў

RQ2 (Comparison). How does A NTAEUS compare against state-of-the-art agentic LLMs and function-level baselines?  

> вЂў

RQ3 (Ablation). Which components drive the gains, and by how much?  

> вЂў

RQ4 (Cost). What is the cost of running A NTAEUS ,and how does it compare to alternatives? 

4.1. Experimental Setup 

Datasets: We draw our evaluation targets from Re-posVul [28], a repository-level vulnerability dataset built from NVD CVE records and their patch commits, span-ning thousands of open-source projects across multiple lan-guages. ReposVul annotates each entry at the repository, file, function, and line levels, and ships every CVE with its full repository, allowing us to analyze each vulnerability in its real project context. From this dataset, we select 28 CVEs, comprising 12 CWE-200 and 16 CWE-284 logic vulnerabilities, each paired with its repository and used as ground truth. 

Models: We run ANTAEUS with two frontier models and refer to the resulting configurations by the reason-ing model they use: A NTAEUS O4.7 (Claude Opus 4.7) and ANTAEUS G5.4 (GPT-5.4). We additionally report recall for ANTAEUS O4.8 (Claude Opus 4.8). All analyses run through model APIs and require no local compute.

The two passes are otherwise identical and operate on the same repository. Unless stated otherwise, the per-repository figures we report aggregate both passes. 

Metrics: We report the number of true positives (TPs) and false positives (FPs). Scoring recall for logic vulnerabilities is nontrivial, since a tool may flag the right function for the wrong reason, and a binary vulnerable versus non-vulnerable match would overstate how well the underlying flaw was understood. We therefore exploit A NTAEUS вЂ™s structured out-put. Because each finding names a security-sensitive sink together with the conditions under which it is safe, we count a detection as a true positive only when the reported sink and its conditions actually explain the CVE, rather than when the flagged function merely overlaps with the vulnerable one. Concretely, a vulnerability is counted as detected when the pass corresponding to its CWE class reports a finding that explains it, while the opposing pass contributes only to the false-positive count for that repository. The number of false positives is then the count of reported findings that do not correspond to a real flaw. 

Comparisons: We compare A NTAEUS against two families of baselines. The first is function-level detection , where each function is analyzed in isolation with no repository-level context, using the same models as A NTAEUS (Opus 4.7 or GPT5.4) to separate the contribution of context from that of the underlying model. The second is agentic detection ,where a model autonomously explores the repository, for which we use Claude Opus 4.8, Claude Opus 4.7, and Codex 5.4.

4.3. RQ1: Detection Effectiveness 

Having reduced the analysis space by roughly 31 Г— on average, we now evaluate whether the remaining functions are sufficient to recover real logic vulnerabilities. Table 2 reports the end-to-end effectiveness of A NTAEUS on the 28 repositories in our benchmark, using Opus 4.7 and GPT-5.4 as the reasoning model respectively. We discuss each in turn. With Opus 4.7, A NTAEUS detects and explains 15 out of 28 confirmed logic vulnerabilities, producing for each a structured finding that names the security-sensitive sink and the unsatisfied safety condition behind the detection. Detection spans repositories of widely varying size, from a few dozen functions to several hundred, indicating that the approach does not depend on a particular scale of project. Comparative validation accounts for a meaningful share of the reported precision. It removes 337 false positives from the initial output, reducing the total from 1,682 to 1,345 while leaving every true positive intact, which confirms that validation prunes a substantial fraction of recurring, non-distinctive findings rather than acting as a cosmetic post-processing step. With GPT-5.4, A NTAEUS detects 12 out of 28 vulnerabilities, with validation reducing false positives from 3,927 to 2,918. This suggests that the two models surface partially complementary findings under the same framework, and that the gains of A NTAEUS are not tied to a single reasoning model.

5. Discussion 

Our aim is not a complete or formally sound vulnerability-analysis framework, but a practical method for recovering repository-specific security invariants under realistic analysis budgets. A NTAEUS deliberately trades ex-haustive coverage for tractability at repository scale, and we therefore make no claim of exhaustive detection, com-pleteness over all logic vulnerabilities, or formal correctness of the invariants it infers. The results should be read as evidence that repository-grounded invariant recovery can materially improve discovery effectiveness in practice. In the rest of this section we examine what this trade buys, where its boundaries currently lie, and how the design posi-tions A NTAEUS to extend past them, with those boundaries understood in that light rather than as failures to meet a guarantee we never set out to provide. 

Depth of local grounding. The local augmentation bundle expands the call graph to depth one, which keeps prompts within budget. The consequence is that a safety condition whose justification lives two or more levels away may be judged on incomplete evidence. The repository-level con-text bundle partially offsets this by supplying project-wide semantics, though as a summary it does not restore arbitrary interprocedural reach. How deep to expand the call graph is a tunable point in the design rather than a fixed limit, and deeper grounding can be traded against prompt budget where a target repository warrants it.

Reliance on model judgment. Sink identification and safety-condition derivation come from an LLM, and al-though the structured output ties each claim to supplied code, the model can still omit a relevant sink, propose a spurious condition, or offer reasoning that is plausible but unfaithful [23]. Comparative validation mitigates this by pruning conditions that recur across peers, but it operates on what the reasoning stage reports and cannot recover a sink the model never surfaced. This is the central reason ANTAEUS favors structured, evidence-linked output over free-form verdicts: making each claim inspectable is what allows downstream validation, and what lets a human re-viewer audit a finding rather than take it on trust. 

Role of prioritization. Detection examines only the func-tions that survive prioritization, so this stage bounds what the reasoning stage can ever see. It rests on both keyword heuristics for initial pruning and the agentвЂ™s ranking of the survivors, either of which may overlook code whose relevance is not evident from its name, callees, or structure. We keep the heuristics deliberately high-recall, but surfacing security-relevant code at repository scale without exhaustive analysis is an open problem in its own right. We therefore treat prioritization as a pluggable component, and A NTAEUS 

stands to benefit directly from advances along this line. 

Sensitivity to prompt design. Each LLM stage is driven by prompts, and the space of formulations is effectively unbounded. We iterated on the prompts for prioritization, context extraction, and structured reasoning to obtain reli-able, well-structured outputs, but make no claim of opti-mality: a different phrasing, ordering, or level of instruction could move the results either way. The numbers we report should be read as the outcome of one carefully chosen, not exhaustively searched, configuration, which also means there is likely headroom a more systematic prompt search could recover. 

Calibration of validation. Comparative validation cali-brates its thresholds per repository from the empirical sim-ilarity distribution. Where comparable sinks are few, the neighborhood may be too small for the pruning rule to fire, and the stage defaults to retaining the finding, a conserva-tive choice that favors recall over precision. The strictness parameters expose this trade-off directly, giving an operator a knob to tune toward precision or recall according to how a given deployment weighs missed findings against review effort. 

Generality of the evaluation. Our study covers 28 real-world C and C++ repositories with confirmed CWE-200 and CWE-284 vulnerabilities. Labeled logic vulnerabilities of this kind are scarce, so assembling even this many is non-trivial, and the set is substantial relative to what is available for the class. The sample is nonetheless small in absolute terms, which limits statistical power and may not capture the full diversity of C and C++ conventions; results may likewise not transfer directly to other languages, to pro-prietary codebases with different norms, or to vulnerability classes beyond the two we target. We see broadening across languages and CWE classes as the natural next step, and the language-agnostic structure of the pipeline, prioritize, augment, reason, validate, is intended to make that extension straightforward. 

6. Related Work 

6.1. Traditional Approaches to Logic Vulnerabili-ties 

The earliest systematic treatments of logic flaws come from the web domain. Felmetsger et al. derive likely in-variants from observed executions and report paths that contradict them [1], an approach later extended by black-box techniques that learn behavioral models from request traces and synthesize inputs to break them [9], [10]. Domain-specific systems specialize this idea further to application workflows and e-commerce checkout logic [7], [8]. These works establish that the violated property is defined by the application rather than by a fixed code pattern, but the signals they rely on, such as sessions, forms, and navigation structure, have no analog in systems software. A separate tradition infers correctness rules from the code itself and treats violations as defects. Engler et al. extract implicit programming rules from systems code and surface statis-tical outliers as candidate bugs [24], and Chucky identi-fies missing security checks by contrasting a function with structurally comparable code and reporting the omissions that stand out [25]. For the invariants we target, none of these anchors is available, and project-specific rules can encode them only at a manual cost that does not scale across repositories. A NTAEUS retains the anomaly-based intuition of the missing-check line of work, but applies it over structured, model-produced findings. 

6.2. LLM- and Agentic-Based Approaches. 

A growing body of work applies language models to vul-nerability detection. Many systems operate at the function level, presenting a single function and asking for a label or report [12], [22]. A line of work augments the model with external knowledge, retrieving distilled vulnerability descriptions or CWE references at inference time to steer the judgment [29], [30]. A complementary direction couples models with program analysis so that reasoning is anchored to extracted facts rather than raw text. IRIS infers taint spec-ifications with an LLM and drives a static analyzer over the whole repository [31], LLift and the interleaving approach of Chapman et al. embed model queries inside a static-analysis loop and feed the answers back into subsequent analysis [32], [33], and LLMxCPG compresses a code prop-erty graph into compact slices that let a model reason over segments spanning multiple functions [34]. These systems anchor on a tainted source, a specification, or a graph path, the kind of scaffold that logic flaws do not provide. A further line moves detection from the function to the repository. Benchmarking efforts show that inter-procedural detection is a distinct and largely open problem, and supply call-graph dependencies as the bridge from intra- to inter-procedural reasoning [35]. Autonomous auditing agents instead traverse a repository to locate and explain defects [11]. Both rely on a dependency or anchor to follow, which the CWE-284 and CWE-200 issues lacks, leaving open-ended search prone to both wasted budget and drift. A NTAEUS instead fixes a bounded sequence of steps. It narrows the candidate set, attaches targeted evidence, and prunes by comparison. Recent results show that frontier models can uncover real vulnerabilities when given sufficient scaffolding and com-pute [14]. The open question is therefore not whether the capability exists, but how to harness it at repository scale without making the analysis prohibitively expensive [19]. To the best of our knowledge, A NTAEUS is the first system to address this challenge for logic vulnerabilities, targeting repository-level reasoning while keeping costs under control. 

7. Conclusion 

We presented A NTAEUS , a repository-level framework for detecting logic vulnerabilities by grounding LLM rea-soning in codebase-wide context. A NTAEUS follows a five-stage pipeline that combines function prioritization, context-grounding, structured reasoning, comparative validation, and reporting to make LLM analysis both evidence-grounded and cost-aware. It prioritizes security-relevant functions, grounds each in local and repository-level evidence, derives sinks and safety conditions, validates candidate findings against similar sinks, and reports retained violations with supporting evidence. Our evaluation on 28 real-world C and C++ repositories with confirmed CWE-284 and CWE-200 vulnerabilities shows that A NTAEUS detects and explains 15 vulnerabilities, substantially outperforming function-level LLM analysis and frontier-agentic baselines while maintain-ing comparable token usage and cost budgets. These results suggest that repository-specific grounding and staged valida-tion are key to efficiently applying frontier LLM capabilities to logic vulnerability detection.
