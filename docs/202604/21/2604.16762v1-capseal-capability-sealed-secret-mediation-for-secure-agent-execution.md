## 摘要
现代 AI 智能体通常依赖于 API 密钥和 SSH 凭据等秘密信息，但主流的部署模式仍通过环境变量、本地文件或转发套接字将这些秘密直接暴露给智能体进程。这种设计在面对提示词注入、工具滥用和模型控制的泄露时会失效，因为智能体既可以使用也可以泄露相同的持有者凭据。我们提出了 CapSeal，这是一种能力密封的秘密中继架构，它通过本地受信代理的受限调用取代了对秘密的直接访问。CapSeal 结合了能力发行、模式受限的 HTTP 执行、代理执行的 SSH 操作、防重放会话绑定、策略评估以及防篡改审计追踪。我们描述了一个集成了面向 MCP 适配器的 Rust 原型，制定了关于非公开性、受限使用、防重放和可审计性的条件安全目标，并定义了一个涵盖提示词注入、工具滥用和 SSH 滥用的评估方案。该系统将智能体系统的秘密处理方式从向模型提供密钥重新定义为向模型授予范围狭窄、不可导出的操作能力。

## Abstract
Modern AI agents routinely depend on secrets such as API keys and SSH credentials, yet the dominant deployment model still exposes those secrets directly to the agent process through environment variables, local files, or forwarding sockets. This design fails against prompt injection, tool misuse, and model-controlled exfiltration because the agent can both use and reveal the same bearer credential. We present CapSeal, a capability-sealed secret mediation architecture that replaces direct secret access with constrained invocations through a local trusted broker. CapSeal combines capability issuance, schema-constrained HTTP execution, broker-executed SSH actions, anti-replay session binding, policy evaluation, and tamper-evident audit trails. We describe a Rust prototype integrated with an MCP-facing adapter, formulate conditional security goals for non-disclosure, constrained use, replay resistance, and auditability, and define an evaluation plan spanning prompt injection, tool misuse, and SSH abuse. The resulting system reframes secret handling for agentic systems from handing the model a key to granting the model a narrowly scoped, non-exportable action capability.

#### 1. 核心问题与整体含义（研究动机和背景）
随着自主 AI 智能体（Agents）的普及，它们需要频繁访问敏感凭据（如 API 密钥、SSH 私钥）来执行任务。然而，当前的部署模式通常将这些秘密直接暴露给智能体进程（通过环境变量或本地文件）。
*   **核心矛盾**：智能体既是凭据的“使用者”，又是潜在的“泄露者”。在提示词注入（Prompt Injection）、工具滥用或模型受控的情况下，智能体可以轻易地将持有的原始凭据外传。
*   **研究动机**：传统的静态加密无法解决“使用时（use-time）”的暴露问题。论文旨在构建一种架构，使智能体能够**使用**秘密执行操作，但永远无法**接触**到秘密本身。

#### 2. 方法论：核心思想与关键技术
CapSeal 提出了“能力密封（Capability-Sealed）”的中继架构，将秘密管理从“交付密钥”转变为“授予受限的动作能力”。
*   **核心思想**：引入一个本地受信任的**代理（Broker）**作为参考监视器。智能体不直接持有密钥，而是向代理请求一个“能力句柄（Handle）”，代理在隔离环境中注入秘密并代为执行操作。
*   **关键技术细节**：
    *   **会话绑定与防重放**：利用 Unix 域套接字（UDS）提取对等进程身份，通过单调递增序列号、随机数（Nonce）和时间戳防止调用重放。
    *   **HTTP 能力实现**：不仅限制目标主机和路径，还引入 **JSON 模式（Schema）约束**，确保智能体发送的请求体符合预定义结构，防止通过请求参数走私凭据。
    *   **SSH 能力实现**：采用“代理执行（Broker-exec）”模式，代理根据模板执行特定命令并验证主机密钥指纹，智能体无法获得原始私钥或开启任意转发通道。
    *   **策略引擎（PDP/PEP）**：在执行前进行细粒度的权限检查，支持本地或远程策略评估。
    *   **审计追踪**：生成哈希链结构的防篡改审计日志，记录所有签发、调用和撤销事件。

#### 8. 不足与局限
*   **信任边界限制**：CapSeal 假设本地代理（Broker）是受信任的。如果攻击者获得**本地 Root 权限**，仍可能通过内存扫描等手段获取秘密（论文将其列为 v1 版本的攻击范围之外）。

*   **应用限制**：对于需要极高动态性、无法预先定义命令模板或 Schema 的极少数场景，配置成本可能较高。
