## 摘要
庞大的输入空间与复杂的操作间依赖关系使黑盒 REST API 测试面临挑战。AutoRestTest 结合了语义属性依赖图、多智能体强化学习和大语言模型，智能探索大型 API 输入空间。在 SBFT 2026 REST 联赛中，AutoRestTest 在 11 个 API（317 个操作，平均每个 API 约 29 个）上，在故障检测、整体效率和整体效果三项评估类别中均排名第一，在每 API 一小时的测试预算下，平均发现 67.09 个唯一服务器错误并成功处理 17.27 个操作。

## Abstract
Large input spaces and complex inter-operation dependencies make black-box REST API testing challenging. AutoRestTest combines a Semantic Property Dependency Graph, multi-agent reinforcement learning, and large language models to intelligently explore large API input spaces. In the SBFT 2026 REST League, AutoRestTest ranked first in all three evaluation categories -- fault detection, overall efficiency, and overall effectiveness -- on 11 APIs (317 operations, approximately 29 per API), averaging 67.09 unique server errors and 17.27 successfully processed operations per API under a one-hour testing budget.

## 2. 论文提出的方法论

### 2.1 语义属性依赖图（SPDG）
- 解析 OpenAPI 规格文档，为每个 API 操作构建顶点。
- 利用轻量级 GloVe 词嵌入模型，计算不同操作之间参数、请求体、响应的语义相似度。
- 当相似度超过设定阈值时，为对应操作对添加边，边的权重表示参数依赖强度。
- 该图在测试过程中动态更新，以反映新发现的依赖关系。

### 2.2 多智能体强化学习（MARL）
- 引入四个专用智能体，各自维护一个 Q 表，通过 epsilon‑greedy 策略选择动作：
  - **操作智能体**：选择下一个待探索的端点。
  - **参数智能体**：采用分层采样，决定本次请求包含哪些参数（必选参数全包含，可选参数按覆盖平衡采样）。
  - **值智能体**：决定参数值的来源——从 LLM 预生成的候选中选取、复用先前成功响应中的值、或采用默认值。
  - **依赖智能体**：维护 SPDG 并追踪跨操作依赖。
- 共享奖励机制：根据 SUT 响应的 HTTP 状态码发放奖励：
  - 2xx 响应：高奖励，以鼓励构建连续的请求链。
  - 5xx 响应：高奖励，以鼓励发现服务端错误。
  - 其他状态码：惩罚，引导智能体进行有效探索。
- 每次请求后更新 Q 表，训练参数可配置（学习率 α = 0.1，折扣因子 γ = 0.9，探索率 ε 从 1 衰减到 0.2）。

### 2.3 大语言模型驱动的值生成
- **预处理阶段**：对每个操作并行调用 LLM（Gemini 2.5 Flash Lite），生成与领域相关的参数值，并立即对 SUT 进行验证。
- 若验证失败，利用服务器响应信息重新构造提示词，最多重试 3 次。
- 验证成功的值（或最终重试值）存入值智能体的动作空间，供测试过程高效复用，大幅降低测试时 LLM 的 token 消耗。

### 2.4 请求生成与变异
- 四个智能体分别独立选择动作，组合形成 HTTP 请求并发送至 SUT。
- 为进一步探测边界情况和内部服务错误，对请求体施加随机变异，包括参数类型修改、名称替换、媒体类型变更、边界值注入等，变异概率设为 0.2。

## 3. 实验设计

- **数据集/场景**：SBFT 2026 REST League 竞赛基准，包含 11 个真实世界的 REST API，共计 317 个操作，每个 API 平均约 29 个操作。
- **对比方法**：与五种同期黑盒 REST API 测试工具进行对比——CATS、EvoMaster、RESTest、RestTestGen、Schemathesis。
- **实验流程**：竞赛组织者对每个工具在每个 API 上独立执行 10 次，每次给予 1 小时的测试时间预算，最终取平均值以消除非确定性影响。
- **评价指标**（均经 Z 分数归一化）：
  - 故障检测：唯一 5xx 服务端错误的数量。
  - 整体效率：基于时间序列的故障数量、操作覆盖率、分支覆盖率的曲线下面积（AUC）。
  - 整体效果：运行结束时故障数量、操作覆盖率、分支覆盖率的组合得分。
