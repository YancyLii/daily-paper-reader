## 摘要
大型语言模型（LLM）智能体通过将语言模型与外部工具和环境集成，日益自动化复杂任务。然而，其自主性带来了显著的安全风险：智能体可能执行破坏性命令、泄露敏感数据或违反领域约束。现有的安全方法面临一个基本权衡：手工制定的规则具有可解释性但很脆弱，过于保守的规则会阻止安全操作（高误报），而宽松的规则会遗漏不安全行为（高漏报）。神经分类器缺乏安全关键部署所需的可解释性。我们提出 AutoSpec，一个通过反例引导的归纳合成（CEGIS）和归纳逻辑编程（ILP）指导，从用户安全/不安全标注中自动演化已部署的专家设计安全规则的框架。从专家规则和标注轨迹流开始，AutoSpec 迭代地评估规则，挖掘误报和漏报反例，利用 ILP 学习哪些谓词能区分它们，生成候选规则编辑，并验证候选规则以选择最佳修订。关键见解是 ILP 有效地识别在漏报中频繁出现但在误报中很少出现（反之亦然）的谓词，从而大幅削减规则编辑的指数级搜索空间。这个过程持续直到收敛，产生可解释的规则，平衡精确率和召回率。我们在涵盖代码执行和具身智能体领域的 291 条执行轨迹上评估 AutoSpec。AutoSpec 在两个领域上将规则 F1 值提升至 0.98 和 0.93，实现高达 94% 的误报减少，同时保持高召回率，并在 4-5 次迭代内收敛。ILP 引导的方法相比启发式 CEGIS 实现了高达 4.8 倍的 F1 提升。学习到的规则是人类可读、可审计的，并能泛化到未见过的场景。

## Abstract
Large language model (LLM) agents increasingly automate complex tasks by integrating language models with external tools and environments. However, their autonomy poses significant safety risks: agents may execute destructive commands, leak sensitive data, or violate domain constraints. Existing safety approaches face a fundamental tradeoff: hand-crafted rules are interpretable but brittle, with overly conservative rules blocking safe operations (high false positives) while permissive rules miss unsafe behaviors (high false negatives). Neural classifiers lack the interpretability required for safety-critical deployments.   We present AutoSpec, a framework that automatically evolves deployed expert-designed safety rules from user safe/unsafe annotations through counterexample-guided inductive synthesis (CEGIS) guided by inductive logic programming (ILP). Starting from the expert rules and a stream of annotated traces, AutoSpec iteratively evaluates rules, mines false-positive and false-negative counterexamples, uses ILP to learn which predicates discriminate them, generates candidate rule edits, and verifies candidates to select the best revision. The key insight is that ILP efficiently identifies predicates that appear frequently in false negatives but rarely in false positives (or vice versa), dramatically pruning the exponential search space of rule edits. This continues until convergence, producing interpretable rules that balance precision and recall.   We evaluate AutoSpec on 291 execution traces spanning code execution and embodied agent domains. AutoSpec raises rule F1 to 0.98 and 0.93 across the two domains, achieving up to 94% false positive reduction while maintaining high recall, and converges within 4-5 iterations. The ILP-guided approach achieves up to 4.8x higher F1 than heuristic CEGIS. The learned rules are human-readable, auditable, and generalize to unseen scenarios.

### 2. 论文提出的方法论
- **核心思想**：AutoSpec 从已部署的专家设计规则出发，将用户标注的执行轨迹作为反例，通过“评估→反例挖掘→ILP 引导谓词学习→候选规则生成→验证→选择”的迭代循环，逐步精化规则集。
- **关键技术细节**：
  - **安全规则建模**：规则基于领域谓词库（如 `destructive_os_inst`、`near_danger_zone`）构建，单条规则为触发条件 $\cdot$ (谓词合取)，任何事件触发规则即判定整条轨迹为不安全。
  - **四阶段迭代**（Algorithm 1）：
    1. **反例挖掘**：在当前规则集上评估标注轨迹，收集误报（FP，安全轨迹被错误拦截）和漏报（FN，不安全轨迹被放过），前者指示规则需收紧，后者指示需放宽。
    2. **ILP 引导谓词学习**：将 FN 和 FP 作为正、负例输入 ILP 求解器（ILASP），学习能区分二者的最小谓词组合，并给出编辑建议（如增加例外、增加合取支路等）。
    3. **候选生成与选择**：基于 ILP 建议，使用四种基本编辑算子（AddConjunct、AddException、AddDisjunct、Relax）生成候选规则修订；结合单步编辑、混合修复链、失败驱动支路等策略生成候选集，再按 F1、精确率、未解决反例数、规则数量等指标贪婪选择最佳修订。
    4. **验证与收敛**：将最佳候选作为下一轮规则，若得分停止改进或达到目标阈值则终止，并通过下降权重避免谓词重复提议。
  - **关键公式**：规则演化定义为离散优化问题：\[ R^* = \arg\max_{R \in \text{Edit}^*(R_0, P)} S(R) \]，其中 \( S \) 为基于 F1 的词典序评分函数。
- **算法流程**（文字描述）：从一个初始规则集 \(R_0\) 和标注轨迹集开始，重复直至达到最大迭代次数或满足收敛条件：评估当前规则集 \(R\)，挖掘 FP/FN 反例；若无反例则停止；否则调用 ILP 学习反例的区分谓词并生成编辑建议；根据建议生成若干候选规则集；验证各候选集并选出最佳者；若最佳候选得分优于 \(R\) 则将其设为 \(R\)，否则返回 \(R\)。

### 6. 论文的主要结论与发现
- **有效性**：AutoSpec 在代码执行与具身智能体领域分别将规则 F1 提升至 **0.98** 和 **0.93**，相对于专家基线规则，不仅大幅减少了误报（FP 从 31 降至 2，代码领域），同时维持了高召回，平衡了精确率与召回率。
- **ILP 的关键作用**：移除 ILP 引导后，F1 从 0.98 跌至 0.20（代码领域），证明启发式谓词排序无法有效识别区分性特征；ILP 引导使最终 F1 比无 ILP 的 CEGIS 最高提升 **4.8 倍**。
- **高效收敛**：AutoSpec 在 **4~5 次迭代**内收敛，且每次迭代计算耗时极短（≤20 秒），适合异步维护。
- **泛化能力**：从 70% 任务中学到的规则应用到其余 30% 未见任务时，F1 仅下降 1.6 个（代码）和 4.7 个（具身）百分点，表现稳健。
- **可用性**：用户研究中，从业者对 AutoSpec 产出规则在可解释性、可操作性、信任度上的评分均显著高于 LLM 分类器，尤其在可操作性上差距最大（4.83 vs. 2.73），表明符号化规则更易被人类理解和修改。

### 7. 优点
- **思想创新**：将安全规则演化建模为 CEGIS，并创造性地用 ILP 剪枝搜索空间，解决了手工编辑组合爆炸的难题。
- **可解释且可审计**：产出的安全规则由领域谓词显式组合而成，每一步决策均可追溯，满足安全关键系统对透明性的需求。
- **从反馈中学习**：直接利用智能体运行中产生的标注轨迹，无需额外数据采集，使规则能与智能体环境同步进化。
- **方法解耦**：谓词库与编辑算子分离，扩展新领域只需增加谓词，无需更改核心演化算法。
- **低计算开销**：整个框架可在普通笔记本上运行，适合作为异步维护工具，不影响主系统性能。

### 8. 不足与局限
- **领域覆盖有限**：仅在代码执行与具身智能体两个领域验证，未在更广泛的智能体场景（如网络操作、数据库代理）进行测试，方法的跨领域泛化性仍有待证明。
- **依赖谓词库质量**：规则表达能力完全受限于提供的谓词库，若谓词覆盖不足，ILP 也无法识别有效的区分性特征；虽然系统能通过未解决的反例提示谓词缺失，但自动生成新谓词不在当前研究范围内。
- **标注粒度**：轨迹级标注只能定位整次交互是否安全，无法精确到具体的不安全动作，这可能导致在复杂长轨迹中学习到次优的谓词组合。
- **全局最优无保证**：AutoSpec 使用贪婪式局部搜索，虽然迭代单调提升，但不保证找到全局最优规则集，且性能可能受初始规则和谓词库影响。
- **时效性操作受限**：当前谓词基于单事件或局部上下文，显式的时间/状态属性的表达需编码在谓词内部，缺少原生时序逻辑支持（作者已指出为未来工作方向）。
- **实验规模**：总轨迹数仅 291 条，虽然进行了多维度的有效性检查，但结论在更大规模真实部署数据上的稳健性尚需验证；用户研究参与者数量较少（10 人），可能影响统计说服力。
