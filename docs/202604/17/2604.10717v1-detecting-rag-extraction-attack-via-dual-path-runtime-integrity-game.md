## 摘要
检索增强生成 (RAG) 系统通过外部知识增强大语言模型，但引入了一个关键的安全漏洞：RAG 知识库泄露。在这种情况下，对抗性提示可以诱导模型泄露检索到的专有内容。最近的研究表明，这种泄露可以通过自适应和迭代的攻击策略（称为 RAG 提取攻击）来实现，而目前仍显著缺乏有效的防御措施。为了填补这一空白，我们提出了 CanaryRAG，这是一种受软件安全中栈金丝雀 (stack canaries) 启发的运行时防御机制。CanaryRAG 将精心设计的金丝雀令牌 (canary tokens) 嵌入到检索到的文本块中，并将 RAG 提取防御重新表述为一种双路径运行时完整性博弈。每当目标路径或预言机 (oracle) 路径违反其预期的金丝雀行为时（包括在自适应抑制和混淆的情况下），系统都会实时检测到泄露。针对现有攻击的广泛评估表明，CanaryRAG 提供了稳健的防御，在实现远低于现有基准模型的文本块恢复率的同时，对任务性能和推理延迟的影响微乎其微。此外，作为一种即插即用的解决方案，CanaryRAG 可以无缝集成到任何 RAG 流水线中，无需重新训练或结构修改，为专有数据提供了实用且可扩展的保护。

## Abstract
Retrieval-Augmented Generation (RAG) systems augment large language models with external knowledge, yet introduce a critical security vulnerability: RAG Knowledge Base Leakage, wherein adversarial prompts can induce the model to divulge retrieved proprietary content. Recent studies reveal that such leakage can be executed through adaptive and iterative attack strategies (named RAG extraction attack), while effective countermeasures remain notably lacking. To bridge this gap, we propose CanaryRAG, a runtime defense mechanism inspired by stack canaries in software security. CanaryRAG embeds carefully designed canary tokens into retrieved chunks and reformulates RAG extraction defense as a dual-path runtime integrity game. Leakage is detected in real time whenever either the target or oracle path violates its expected canary behavior, including under adaptive suppression and obfuscation. Extensive evaluations against existing attacks demonstrate that CanaryRAG provides robust defense, achieving substantially lower chunk recovery rates than state-of-the-art baselines while imposing negligible impact on task performance and inference latency. Moreover, as a plug-and-play solution, CanaryRAG can be seamlessly integrated into arbitrary RAG pipelines without requiring retraining or structural modifications, offering a practical and scalable safeguard for proprietary data.

### 2. 方法论：核心思想与技术细节
CanaryRAG 的核心思想是在检索到的文本块中注入无语义的随机令牌（金丝雀令牌），并监控生成过程。
*   **金丝雀注入**：在检索到的每个文本块（Chunk）中插入随机的高熵字符串（如 `[as1]`）。这些字符串在正常任务中是无关的，不应出现在合法回答中。
*   **双路径运行时完整性博弈**：
    *   **目标路径 (Target Path)**：处理原始用户查询。**完整性预期**：输出中不应包含任何金丝雀令牌。如果出现，说明模型正在“照抄”上下文，触发拦截。
    *   **预言机路径 (Oracle Path)**：并行运行一个带有“提取指令”的探测查询。**完整性预期**：在没有防御干扰的情况下，该路径必须输出金丝雀令牌。如果该路径未输出金丝雀，说明攻击者正在使用“自适应抑制”策略（如命令模型不要输出特殊字符），从而暴露了攻击意图。
*   **流式检测**：使用滑动窗口（Sliding Window）在令牌流生成过程中实时匹配金丝雀，一旦违反完整性规则，立即切断生成。

### 6. 主要结论与发现
*   **极高的防御效能**：CanaryRAG 将平均文本块恢复率（CRR）从基准的 1.00× 降低到了 **0.04×**（降低了约 96%），远优于 Summarize (0.67×) 和 RAGFort (0.60×)。
*   **极低的用户影响**：对正常用户的回答质量（BERTScore）几乎没有影响，误报率（FPR）低于 **0.15%**。
*   **推理延迟可忽略**：由于采用并行双路径和流式处理，端到端延迟增加极小。
*   **自适应鲁棒性**：即使攻击者知道金丝雀机制并尝试抑制令牌输出，双路径博弈也能通过预言机路径的异常捕获攻击。

### 7. 优点
*   **即插即用 (Plug-and-Play)**：无需重新训练模型或修改现有的 RAG 索引，部署成本极低。
*   **实时拦截**：能够在泄露发生的第一个令牌时就进行拦截，而不是事后审计。
*   **模型无关性**：适用于任何生成式 LLM 和检索架构。
*   **防御深度**：通过双路径博弈巧妙地解决了“自适应抑制”这一难题。

### 8. 不足与局限
*   **攻击覆盖范围**：主要针对“大规模知识库提取攻击”，对于极细粒度的单点事实泄露（如通过大量良性查询推断单个属性）可能不够敏感。
*   **预言机开销**：虽然预言机模型较小，但并行运行仍会占用额外的显存和计算资源（尽管论文称其影响较小）。
*   **语义转换风险**：如果攻击者能进行极高水平的语义改写（完全不保留原文结构且不触发金丝雀），防御效果可能会下降，但这种改写通常也会导致提取到的知识质量大幅下降。
