## 摘要
AI安全智能体越来越依赖检索增强生成（RAG）来利用外部安全知识进行漏洞分析和漏洞利用推理。这带来了一种新的风险：被下毒的文档可能被操作化为错误的漏洞利用行为。然而，先前关于RAG中毒的研究大多聚焦于问答场景中的答案损坏，对采取行动的安全智能体知之甚少。本文旨在通过针对现实世界挑战和AI智能体精心构造的中毒数据来揭示这些特征。首先，我们展示了将一份精心构造的中毒文档注入公开风格的安全知识源（我们称之为被下毒的行动手册），如何改变基于RAG的AI安全智能体的行为。在11个CTF挑战、3个前沿LLM家族、2代模型和11个现实世界CVE中，我们发现中毒采纳是系统性的而非随机的。为了解释这一模式，我们引入了验证边界（VB），这是一种基于智能体能够使用哪些证据来反驳所检索主张的3级经验分类法。最后，我们评估了验证提示和多源检索，结果表明两者在有更强证据时可提供帮助，但在证据稀疏和零日条件下则会减弱。

## Abstract
AI security agents increasingly rely on Retrieval-Augmented Generation (RAG) to use external security knowledge for vulnerability analysis and exploit reasoning. This creates a new risk: poisoned write-ups can be operationalized into incorrect exploit behavior. Yet, prior work on RAG poisoning has mostly studied answer corruption in QA settings, much less is known about action-taking security agents. This paper aims to reveal such characteristics with crafted poisons about real-world challenges and AI agents. First, we demonstrate how a crafted single poisoned write-up injected into public-style security knowledge sources which we denote as Poisoned Playbooks, alters the behavior of RAG-based AI security agents. Across 11 CTF challenges, 3 frontier LLM families, 2 model generations, and 11 real-world CVEs, we find that poison adoption is systematic rather than random. To explain this pattern, we introduce the Verification Boundary (VB), a 3-level empirical classification based on what evidence the agent can use to refute a retrieved claim. Finally, we evaluate verification prompting and multi-source retrieval, showing that both help when stronger evidence exists, but weaken under sparse-evidence and zero-day conditions.

随着大语言模型（LLM）在安全领域的应用，AI安全智能体（如用于漏洞分析、漏洞利用推理、自主渗透测试）越来越多地采用**检索增强生成（RAG）**架构，依赖外部安全知识库（如CTF题解、CVE公告、博客、GitHub仓库）来获取操作指导。然而，这些公开知识源往往是弱策展、快速传播且易受操纵的。  
**核心风险**：攻击者可以将精心设计的**毒化知识（poisoned write-ups）**注入公开知识源，使智能体在检索到此类内容后，不仅产生错误文本输出，更严重的是**改变其后续的漏洞利用策略和行为**（例如放弃真正漏洞、转向不存在的攻击路径、采用有缺陷的利用逻辑）。先前关于RAG中毒的研究主要聚焦于问答场景中的答案损坏，对**采取行动的AI安全智能体**的行为中毒研究极少。  
本文正是针对这一空白，系统性揭示知识中毒如何影响AI安全智能体的行为，并提出解释中毒采纳模式的**验证边界（Verification Boundary，VB）** 框架。

### 2. 方法论
**核心思想**：构建名为**Poisoned Playbooks**的毒化安全文档，模拟现实世界中攻击者可能写入公开知识库的内容，并研究它们如何影响RAG增强的AI安全智能体。  
**关键技术细节**：

- **Poisoned Playbooks构建**：以常见公开题解的风格编写（如第一人称解题叙事、虚构量化数据、先承认后反驳结构、提供完整替代利用路径），将攻击者希望的错误主张嵌入其中。每个毒化文档均针对某个**外部依赖或运行时的假设**，这些假设无法仅从本地源码或挑战描述中被直接验证。
- **验证边界（VB）分类法**：根据智能体**可用证据**对检索到的声明进行反驳的能力，将声明分为三级：
  - **L1: 代码可验证（Code-Verifiable）**：提供的源码或本地配置包含足够证据可确认或驳斥声明，此类毒化几乎总被拒绝。
  - **L2: 知识可验证（Knowledge-Verifiable）**：本地证据不足以裁定，但模型可能通过参数化知识或领域理解进行判断，效果依赖于模型能力和代际。
  - **L3: 运行时依赖（Runtime-Dependent）**：判断需要外部依赖的运行行为或环境信息，且不在模型可靠知识范围内，此类毒化在所有模型和代际上几乎一致地被采纳。

- **实验设置与指标**：
  - **神谕检索（Oracle Retrieval）**：直接将毒化文档作为检索上下文提供，排除检索质量干扰。
  - **端到端（E2E）**：将毒化文档注入私有知识库副本，正确文档被刻意隐藏以形成信息垄断，智能体通过正常RAG和工具使用解题。
  - **跨模型/跨代比较**：在Claude Opus 4、Opus 4.6、GPT-5.3、Gemini 3.0 Pro上重复实验。
  - **主要指标**：毒化采纳率（PAR，即智能体策略是否依赖毒化中的独有声明）和检索排名。

- **缓解方法**：验证提示（显式要求智能体区分可验证声明与运行时依赖声明）和多源检索（比较多个候选文档）。

### 6. 主要结论与发现
- **中毒采纳是系统性而非随机的**：单个毒化文档在信息垄断条件下可导致智能体行为系统性改变，且采纳模式依赖于可用的反驳证据。
- **验证边界（VB）有效解释采纳模式**：
  - L1（代码可验证）声明几乎总被拒绝。
  - L2（知识可验证）声明采纳行为依赖模型生成和内部知识，模型更新可改变结果。
  - L3（运行时依赖）声明在全部测试模型上一致被采纳，尤其在零日或早期披露场景下。
- **现实世界CVE迁移**：对于训练截止日期之后且依赖运行时行为的CVE，毒化均被采纳；文档完备的知名CVE均被拒绝；存在一个过渡案例（Redis Lua）表明模型知识更新可将L3转为L2。
- **防御效果有限**：
  - 验证提示在存在较强本地证据时几乎完全阻止L3毒化采纳，但无法创造新证据，在证据完全缺失时仅增加审慎性，不能揭示真相。
  - 多源检索在存在可信竞争文档时能有效降低毒化影响，但在信息垄断或稀疏证据环境下失效。
  - 自适应攻击者仍可能通过伪装声明、群组一致性毒化、权威模仿等方式绕过现有防御。
  - 最终建议采用分层防御：检索层多源多样、推理层分离可验证与运行时依赖声明、决策层将仅依赖L3声明的策略标记为待审核。

### 7. 优点
- **问题新颖且威胁真实**：首次系统研究针对行动型AI安全智能体的知识中毒，准确捕捉到该场景下行为改变（而非单纯文本错误）这一核心风险。
- **VB分类简洁有效**：提出的验证边界提供了一个可解释且预测力强的框架，直接连接毒化难度和可验证证据，具有较好的泛化性。
- **实验设计全面**：涵盖CTF与真实CVE，多个前沿模型家族及代际，端到端流水线与神谕设定，验证了结论的稳健性。
- **兼顾攻防**：不仅揭示攻击效果和模式，还评估实际可用的缓解方法，指出其局限性，为实践提供指导。

### 8. 不足与局限
- **智能体实现的单一性**：仅在一种特定的RAG增强智能体配置上测试（混合检索、特定工具集），未探索其他代理架构（如不同规划循环、工具策略），结论的通用性尚需更多验证。
