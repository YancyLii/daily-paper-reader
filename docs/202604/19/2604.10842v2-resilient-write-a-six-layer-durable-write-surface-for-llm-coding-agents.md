## 摘要
由大语言模型驱动的编程智能体越来越多地依赖于模型上下文协议（MCP）等工具调用协议，在开发者的工作站上读写文件。当写入因内容过滤、截断或会话中断而失败时，智能体通常接收不到结构化信号，导致草稿丢失，并因盲目重试而浪费 Token。我们提出了 Resilient Write，这是一个 MCP 服务端，它在智能体与文件系统之间插入了一个六层持久化写入层。这些层级——预检风险评分、事务性原子写入、断点续传安全分块、结构化类型错误、带外暂存盘存储以及任务连续性移交信封——是正交的且可独立采用。每一层都对应于 2026 年 4 月一次真实智能体对话中观察到的具体故障模式，当时内容安全过滤器静默拒绝了一份包含脱敏 API 密钥前缀的草稿。在使用该系统撰写本文的过程中，衍生出了三个额外工具：分块预览、格式感知验证和日志分析。一个包含 186 个测试的套件验证了每一层的正确性；与朴素基准和防御性基准的定量对比显示，恢复时间缩短了 5 倍，智能体自纠错率提升了 13 倍。Resilient Write 已根据 MIT 许可证开源。

## Abstract
LLM-powered coding agents increasingly rely on tool-use protocols such as the Model Context Protocol (MCP) to read and write files on a developer's workstation. When a write fails - due to content filters, truncation, or an interrupted session - the agent typically receives no structured signal, loses the draft, and wastes tokens retrying blindly. We present Resilient Write, an MCP server that interposes a six-layer durable write surface between the agent and the filesystem. The layers - pre-flight risk scoring, transactional atomic writes, resume-safe chunking, structured typed errors, out-of-band scratchpad storage, and task-continuity handoff envelopes - are orthogonal and independently adoptable. Each layer maps to a concrete failure mode observed during a real agent session in April 2026, in which content-safety filters silently rejected a draft containing redacted API-key prefixes. Three additional tools - chunk preview, format-aware validation, and journal analytics - emerged from using the system to compose this paper. A 186-test suite validates correctness at each layer, and quantitative comparison against naive and defensive baselines shows a 5x reduction in recovery time and a 13x improvement in agent self-correction rate. Resilient Write is open-source under the MIT license.

*   **核心痛点**：当写入失败时，智能体通常收不到结构化信号（甚至是静默失败），导致草稿丢失、Token 浪费在盲目重试上，且无法自主诊断错误原因。
*   **整体含义**：本文提出了一种六层持久化写入架构，通过在智能体与文件系统之间建立一个“韧性层”，确保写入过程可预测、可恢复且具备结构化反馈。

### 2. 方法论：六层防御架构
Resilient Write 作为一个 MCP 服务器实现，其核心思想是将写入操作分解为六个独立且正交的层级：
*   **L0: 预检风险评分 (Pre-flight Risk Scoring)**：在写入前使用确定性分类器（正则匹配）评估内容触发下游过滤器的风险。通过公式计算风险分 $s$，对 API 密钥、JWT、私钥等模式进行加权求和并进行亚线性衰减处理，返回结构化建议。
*   **L1: 事务性原子写入 (Transactional Atomic Writes)**：遵循“先写临时文件 -> fsync 同步 -> 哈希校验 -> 原子重命名”的流程，确保文件要么完整写入，要么保持原样，并记录追加式日志（.jsonl）。
*   **L2: 断点续传分块 (Resumable Chunking)**：支持将大文件拆分为编号分块，支持幂等重试和连续性校验，仅需重试失败的分块。
*   **L3: 结构化错误信封 (Typed Error Envelopes)**：将模糊的错误字符串替换为包含 `error_type`、`reason_hint` 和 `retry_budget`（重试预算）的 JSON 对象，防止智能体陷入无限重试循环。
*   **L4: 带外暂存盘 (Scratchpad)**：提供内容寻址存储，用于存放不宜进入代码库的敏感数据或二进制大对象。
*   **L5: 任务移交信封 (Handoff Envelopes)**：在会话中断时生成 `HANDOFF.md`，记录任务状态、文件哈希快照和后续步骤，便于新会话接管。

### 6. 主要结论与发现
*   **效率提升**：相比朴素方案，Resilient Write 将智能体的故障恢复时间缩短了 **5 倍**。
*   **可靠性增强**：数据丢失概率降低了 **50 倍**。

### 7. 优点
*   **正交性设计**：六个层级相互独立，开发者可以根据需求只采用其中几层（如只用 L1 和 L5）。
*   **结构化反馈**：将“面向人类”的错误提示转变为“面向智能体”的机器可读指令，这是提升 Agent 鲁棒性的关键。
*   **零依赖日志**：使用追加式 JSONL 日志而非数据库，保证了易读性和版本控制友好性。

### 8. 不足与局限
*   **作用域限制**：目前仅限于单工作区根目录，不支持跨多个工作区的复杂编排。
*   **缺乏跨文件事务**：虽然单个文件写入是原子的，但不支持多个文件同时写入的“全成功或全回滚”事务。
*   **分类器覆盖面**：L0 层的风险评分依赖于预定义的正则模式，对于新型敏感数据格式或非英语 PII 可能存在漏报。
*   **本地化限制**：日志和暂存盘均存储在本地，不支持分布式环境下的状态同步。
