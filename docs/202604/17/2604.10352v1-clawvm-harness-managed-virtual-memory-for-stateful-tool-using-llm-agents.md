## 摘要

## Abstract
Stateful tool-using LLM agents treat the context window as working memory, yet today's agent harnesses manage residency and durability as best-effort, causing recurring failures: lost state after compaction, bypassed flushes on reset, and destructive writeback. We present \textsc{ClawVM}, a virtual memory layer that manages state as typed pages with minimum-fidelity invariants, multi-resolution representations under a token budget, and validated writeback at every lifecycle boundary. Because the harness already assembles prompts, mediates tools, and observes lifecycle events, it is the natural enforcement point; placing the contract there makes residency and durability deterministic and auditable. Across synthetic workloads, 12 real-session traces, and adversarial stress tests, \textsc{ClawVM} eliminates all policy-controllable faults whenever the minimum-fidelity set fits within the token budget, confirmed by an offline oracle, and adds median <50 microseconds of policy-engine overhead per turn.

### 1. 核心问题与整体含义（研究动机和背景）
*   **核心问题**：当前的智能体框架（如 OpenClaw）将 LLM 的上下文窗口视为“工作内存”，但在管理状态的驻留性（哪些信息留在窗口内）和持久性（哪些信息存入数据库）时，采用的是“尽力而为”的启发式策略。
*   **研究背景**：这种不确定的管理方式导致了三大类故障：
    1.  **驻留失败**：上下文压缩（Compaction）时丢失了关键指令或计划步骤，导致智能体“失忆”。
    2.  **持久化失败**：在会话重置时跳过内存刷新，或在回写时发生破坏性覆盖。
    3.  **可观测性失败**：内存检索失败时没有明确的错误代码，难以调试。
*   **整体含义**：借鉴操作系统虚拟内存（VM）的概念，将内存管理权从模型手中收回到“框架（Harness）”层，建立一套确定性的、可审计的状态管理契约。

### 2. 方法论：核心思想与关键技术
ClawVM 的核心是将智能体状态视为**类型化页面（Typed Pages）**，并通过框架强制执行管理策略：
*   **类型化页面与不变性**：将状态分为引导/策略（Bootstrap）、约束（Constraint）、计划（Plan）、偏好、证据（工具输出）和对话片段。每种页面都有“最小保真度不变性”（例如：约束页面绝不能被简化到只剩一个指针）。
*   **多分辨率表示（Multi-resolution）**：每个页面有四种表现形式：**完整（Full）**、**压缩（Compressed）**、**结构化（Structured）**和**指针（Pointer）**。系统根据 Token 预算，在不违反不变性的前提下，动态调整页面的精细度。
*   **确定性选择策略**：采用两阶段算法：
    1.  **第一阶段**：安装所有硬固定（Hard-pinned）页面和满足最小保真度的表示。
    2.  **第二阶段**：根据“边际效用/Token 成本”进行贪心升级，直到耗尽 Token 预算。
*   **验证回写协议（Validated Writeback）**：三阶段提交过程（暂存、验证、提交），确保回写是非破坏性的（追加或带版本的设置），并检查架构正确性和权限。
*   **故障模型**：定义了可观测的故障类型（如 Refetch 故障、重复工具调用故障、刷新丢失故障等），使内存失效变得可诊断、可重现。

### 4. 资源与算力
*   **算力消耗**：论文未提及大规模 GPU 训练，因为 ClawVM 是一个**运行时系统层**。
*   **运行开销**：实验显示，ClawVM 策略引擎的每轮中位开销**小于 50 微秒 ($\mu s$)**，峰值内存占用小于 83 KB。这表明该方法对系统性能的影响几乎可以忽略不计。

*   **实验规模**：涵盖了 24 种配置（4 类负载 $\times$ 6 种 Token 预算）、12 个长达 100-200 轮的真实轨迹、30 个独立任务重放以及对抗性压力测试。

### 6. 主要结论与发现
*   **故障消除**：在 Token 预算允许放下最小保真度集合的情况下，ClawVM **消除了所有策略可控的故障**（从基准方法的平均 67.8 个故障降至 0）。
*   **稳定性提升**：相比从业者配置的基准，抖动指数（Thrash Index）降低了 11.4%。
*   **鲁棒性**：在极端预算压力和对抗性重置场景下，ClawVM 依然能保持状态一致性，而传统方法会发生严重的“失忆”或重复劳动。
*   **确定性**：通过将内存契约移至框架层，使得智能体的行为变得可预测和可重放。

### 7. 优点
*   **系统级创新**：将成熟的 OS 虚拟内存概念引入 LLM 智能体领域，解决了当前智能体开发中“玄学”调优上下文的问题。
*   **极低开销**：在几乎不增加延迟的情况下，显著提升了长程任务的可靠性。
*   **可观测性**：引入了明确的故障代码，解决了 LLM 内存管理中“无声失败”的痛点。
*   **多分辨率设计**：避免了“全有或全无”的上下文丢弃，实现了优雅降级。

### 8. 不足与局限
*   **语义正确性**：ClawVM 只能保证状态“在不在”以及“格式对不对”，但无法验证模型生成的更新内容在**语义上是否真实**（即模型可能产生格式正确但事实错误的更新）。
*   **工具模拟限制**：重放实验依赖于工具签名的模拟，对于具有高度非确定性输出的实时服务，其实际表现可能更复杂。
*   **应用范围**：目前主要针对单智能体框架，在多智能体协作中的复杂内存共享和竞争问题尚待研究。
*   **外部验证**：虽然在真实轨迹上做了验证，但缺乏大规模在线用户环境下的端到端任务成功率对比。
