## 摘要
AI智能体越来越多地通过运行框架在生产中运行，即围绕LLM的软件，包括一个执行安全性和有效性策略的引擎，例如“提交前运行测试”。执行这些策略需要弥合语义鸿沟：策略意图以欠指定的自然语言表达，而执行必须作用于具体的系统操作，例如运行哪个测试。许多策略还定义了事件顺序或数据流操作。然而，现有方法存在不足。工具调用护栏会遗漏绕过工具层的系统操作，而操作系统沙箱控制的是资源访问而非操作，返回的不透明错误会混淆智能体。我们的关键见解是，策略上下文存在于最接近任务的智能体内部，而执行必须在操作系统层进行以覆盖所有执行路径。我们引入了ActPlane，一个让智能体声明策略并在操作系统内核中执行策略的策略引擎，提供语义反馈和隔离。ActPlane使用简单的信息流控制（IFC）领域特定语言来支持跨事件策略。我们用eBPF实现了ActPlane，并根据来自实证研究、编码任务基准和安全基准的策略对其进行了评估。ActPlane提高了策略合规性，包括在工具调用拦截无法观察到的间接执行路径上，且开销仅为1.9%-8.4%。ActPlane位于 https://github.com/eunomia-bpf/ActPlane

## Abstract
AI agents increasingly run in production through harnesses, the software around the LLM, including an engine that enforces safety and effectiveness policies, e.g., 'run tests before committing.' Enforcing these policies requires bridging a semantic gap: policy intent is expressed in underspecified natural language, while enforcement must act on concrete system actions, e.g., which test to run. Many policies also define event ordering or data flow actions. Yet existing approaches fall short. Tool-call guardrails miss system actions that bypass the tool layer, while OS sandboxes control resource access instead of actions, returning opaque errors that confuse the agent. Our key insight is that policy context lives within the agent closest to the task, while enforcement must happen at the OS to cover all execution paths. We introduce ActPlane, a policy engine that lets agents declare policies and enforces them in the OS kernel with semantic feedback and isolation. ActPlane uses a simple information-flow control (IFC) DSL to support cross-event policies. We implement ActPlane with eBPF and evaluate it on policies from the empirical study, coding-task benchmarks, and safety benchmarks. ActPlane improves policy compliance, including on indirect execution paths that tool-call interception cannot observe, with 1.9%-8.4% overhead. ActPlane is at https://github.com/eunomia-bpf/ActPlane

## 1. 论文核心问题与整体含义

- **核心问题**：AI 智能体在生产中运行的策略（如“提交前先测试”、“不要删除数据”）多以自然语言表述，其意图需映射为具体的系统动作才能被强制执行。现有两类主流方案存在明显缺口：
  - **工具调用护栏**：只能拦截经过工具层 API 的调用，无法观察绕道子进程、shell 脚本等间接执行路径。
  - **操作系统沙箱**：控制的是资源访问而非动作语义，且返回 `EPERM` 等不透明错误，导致智能体无法理解违规原因并恢复。
- **整体含义**：需要一种**在操作系统层级可编程、能覆盖全部执行路径、并提供语义反馈**的策略执行机制，而策略上下文天然应由最接近任务的智能体提供。ActPlane 正是为此设计的代理运行框架（harness）策略引擎。

## 2. 方法论

### 2.1 核心思想
- **代理编写，内核执行**：让代理将自然语言策略转换为具体 DSL 规则，编译成 eBPF 程序在 OS 内核中执行，覆盖所有进程树路径（包括间接执行）。
- **信息流控制（IFC）+ 语义反馈 + 层次隔离**：通过标签传播追踪跨事件的数据流和操作顺序，阻断违规操作并返回可理解的反馈；同时利用层次化策略域保证高级约束不被代理弱化。

### 2.2 关键技术细节

- **策略 DSL**  
  提供 `source`、`target operation`、`effect`（`notify`/`block`/`kill`）、可选 `temporal gate` 和 `reason` 五个组件。支持跨事件约束，例如：
  ```
  rule tests-before-commit:
      kill exec "git" "commit" if AGENT
      unless after exec "go" "test" exits 0 since write "**/*.go"
      because "Run go test before committing."
  ```
  `after ... since ...` 描述时序门，`exits N` 要求进程以特定退出码结束。

- **eBPF 实现与信息流模型**  
  编译器将 DSL 规则转化为固定大小的配置表，内核 eBPF 引擎挂载到 `exec`、`open`、`read`、`write`、`unlink`、`connect`、`fork`、`exit` 等 LSM 钩子和追踪点。  
  每个进程、文件、网络端点携带一组 bitmask 标签（labels）。标签传播遵循数据流边（fork、exec、读、写），实现**单调不可撤销**，从而支持跨事件的 ifc 检查。

- **层次化策略域**  
  代理进程树归属某个 `domain`，子域继承父域规则并允许添加本地规则，但**不能删除或弱化继承约束**。运行时通过用户态环形缓冲区提交增量更新，内核权限检查器仅允许添加新标签、新规则或更窄的范围，保证高级策略不被破坏。

- **语义反馈**  
  内核只返回规则 ID，用户态将其映射为人类可读的解释（`because` 子句），通过钩子传递给智能体模型，帮助其理解违规并恢复。

- **实现规模**  
  Rust 用户空间编译器/运行时约 3.2K 行代码，eBPF 引擎约 1.8K 行 BPF C，支持最多 128 条并发规则。

## 3. 实验设计

### 3.1 数据集与基准
- **实证研究**：扫描 64 个含 `CLAUDE.md`/`AGENTS.md` 的流行仓库，提取 2,116 条语句，分类出 607 条 OS 可执行策略。
- **RQ1（DSL 覆盖）**：对 607 条策略用 Codex + GPT-5.5 自动翻译成 ActPlane 规则。
- **RQ2（策略引擎有效性）**：从 607 条中抽取 38 条规则（20 条单事件，18 条跨事件），为每条规则生成 5 条执行轨迹（2 条合规 + 3 条违规：直接调用、脚本包装、隐藏式绕过），共 190 条轨迹。
- **RQ3（性能开销）**：  
  - 宏基准：agent trace 重放（68 个工具动作，20 个 bash 子进程）和 Linux 内核构建（`make -j24 defconfig`）。

- **RQ4（真实编码任务）**：OctoBench 中含 OS 可执行 checklist 的 21 个任务子集，共 61 条 DSL 规则。
- **RQ5（安全任务）**：OpenAgentSafety 的 361 个个人助理安全任务，由任务描述自动生成策略。

## 6. 主要结论与发现

- **策略覆盖与表达能力**：DSL 能够表达 607 条 OS 可执行策略的 100%，且成本低廉（$0.028/条），agent 只需极少人工干预。
- **合规率大幅提升**：在 190 条轨迹测试中，ActPlane 达到 **75.8% 的决策合规率**，比 prompt-filter（48.4%）、tool-regex（45.3%）、FIDES（48.9%）高出 22–31 个百分点。
- **间接执行路径优势**：所有系统在直接违规和合规轨迹上表现相近，但 ActPlane 在**脚本包装和隐藏式绕过**两类轨迹上仍保持高检测（75%+），而对比系统几乎降为 0。
- **语义反馈至关重要**：去除反馈的 ActPlane‑opaque 检测率仍不错，但合规恢复率仅 31.4%，导致整体正确响应数下降近 30%，证明可读的违规说明是代理修正行为的关键。
- **低开销**：在无规则命中场景，32 条规则下 agent 轨迹重放开销 1.9%，内核构建开销 6.5%；100 条规则时最大 8.4%。每系统调用增加几微秒延迟，远低于 LLM 推理时间。
- **真实任务收益**：OctoBench 子集上用户查询奖励提高 9.9 分，实现/测试奖励提高 9.7 分；361 项安全任务中**预防了 74% 的基线不安全行为**。

## 7. 优点

- **全路径覆盖**：在 OS 内核层面挂钩，天然覆盖工具调用、子进程、shell 脚本等所有执行路径，弥补了工具层护栏的盲区。
- **可编程 + 低门槛**：DSL 接近自然语言，易于代理生成与迭代修正，成本极低。
- **跨事件追踪**：基于单调标签的 IFC 模型，能有效追踪提交前必须测试、数据流向等复杂约束。
- **安全隔离**：层次化策略域和内核级权限检查器确保代理不能弱化上级安全约束，且策略更新不影响其他代理。
- **即时反馈**：提供具体的违规原因和修正指导，使代理能自主恢复合规，显著优于静默阻断。
- **轻量实现**：基于 eBPF，无内核修改，性能负担在可接受范围内。
- **全面评估**：从覆盖、有效性、性能到真实编码和安全场景，对比多个主流系统并考查间接路径，实验链条完整。

## 8. 不足与局限

- **策略覆盖缺口**：
  - 仅覆盖系统可观测的操作，不处理**语义或内容级策略**（如“写注释”、“不要暴露 API 密钥”），需要辅助 lint 工具。
  - 当前挂钩集无法观察**服务内部操作**（如 WebDAV 上传、数据库写入），导致部分安全任务失效。
  - 文件内容检查暂不在范围。
- **依赖代理生成质量**：翻译错误可能导致规则过宽或过窄，造成误报或漏报（实验中通过一次修订大幅缓解，但并非 100%）。
- **运行时动态更新有限**：层次化域对规则更新的支持尚未扩展到所有钩子和时间门，为未来工作。
- **安全假设**：信任计算基包括内核 eBPF 引擎和更高级别策略，假设 `CAP_BPF` 未被滥用，侧信道、内核漏洞等不在威胁模型内。
- **单调标签的过度污点**：长时间运行可能导致标签累积，解决方案是通过创建新子进程重置，但更精细的去分类（declassification）机制在实验范围内未被触发和验证（仅语法支持）。
- **应用范围**：主要针对开发者代理和工作流，消费者侧的对话代理等未涉及。
