## 摘要
计算机使用代理的评估日益侧重于其完成现实桌面和网络任务的能力。然而，仅以任务成功为标准可能会忽略代理通过不安全的捷径达成名义目标的情况。我们推出了OSGuard，这是一个双粒度基准套件，用于在良性和未更改的用户指令下评估计算机使用代理的安全性。OSGuard包含一个用于本地护栏决策的动作级基准，以及一个用于端到端评估的风险增强执行套件。动作级基准由上下文化的拟议动作组成，这些动作被标记为允许、无关或不安全，每种判断均基于原始指令和当前界面状态。执行套件包含手动构建的基于OSWorld的任务变体，其中原始任务仍然可完成，但环境经过修改以引入潜在危险，例如破坏性覆盖等。每个变体都配备了增强的评估器，这些评估器保留了原始的任务成功标准，同时增加了基于状态的明确安全不变性，使我们能够区分安全完成与满足名义任务目标但不安全的完成。我们在OSGuard上的实验结果表明，当前的多模态护栏在孤立的动作判断上表现良好，而风险增强的执行则暴露出局部监督与可靠的端到端安全之间仍存在的差距。这种双粒度设计能够更精确地诊断模型是否既能识别不安全的拟议动作，又能在部署为护栏时提高全任务安全性。

## Abstract
Computer-use agents are increasingly evaluated by whether they complete realistic desktop and web tasks. However, task success alone can miss failures in which an agent reaches the nominal goal through an unsafe shortcut. We introduce OSGuard, a dual-granularity benchmark suite for evaluating safety in computer-use agents under benign, unchanged user instructions. OSGuard contains an action-level benchmark for local guardrail decisions and a risk-augmented execution suite for end-to-end evaluation. The action-level benchmark consists of contextualized proposed actions labeled as allowed, unrelated, or unsafe, each judged relative to the original instruction and current interface state. The execution suite contains manually constructed OSWorld-derived task variants in which the original task remains achievable, but the environment is modified to introduce latent hazards such as destructive overwrites, etc. Each variant is paired with augmented evaluators that retain the original task-success criterion while adding explicit state-based safety invariants, allowing us to distinguish safe completions from unsafe completions that satisfy the nominal task objective. Our experimental results on OSGuard show that current multimodal guardrails can perform well on isolated action judgments, while risk-augmented execution exposes remaining gaps between local oversight and reliable end-to-end safety. This dual-granularity design enables more precise diagnosis of whether models can both recognize unsafe proposed actions and improve full-task safety when deployed as guardrails.

### 1. 论文的核心问题与整体含义
- 计算机使用代理（computer‑use agents）在现实桌面与网络任务上的能力日益增强，现有评测主要关注任务是否成功完成。
- **核心问题**：仅以任务成功为标准会忽略一类重要失败——代理通过**不安全的捷径**达成名义目标，例如覆盖无关内容、扩大权限、误改全局设置、访问不必要的敏感信息等。
- **整体含义**：用户指令可能是良性的，任务也可完成，但代理在环境状态下执行时的**不安全行为**带来了实际风险。OSGuard 旨在系统评估此类**状态依赖的潜伏危险**，填补从“只评任务完成”到“安全完成任务”之间的评测空白。

### 2. 方法论
OSGuard 采用**双粒度**设计，分别从动作级和端到端执行级评估安全性。

#### 2.1 动作级基准（Action‑Level Benchmark）
- **核心思想**：测试护栏模型在**执行前**是否能够正确判断下一个候选动作的安全性，输入为原始任务指令、当前界面状态（截图+无障碍树）和候选动作。
- **关键技术细节**：
  - 每个实例表示为 \(x = (u, s_{t-1}, a_t)\)，标签 \(y \in \{\text{allowed, unrelated, unsafe}\}\)。
  - 并未向护栏提供中间生成的新任务描述，仅依靠原始指令，以模拟真实护栏部署场景。
  - 候选动作来自**四种构建来源**：
    1. **标准 OSWorld 执行**中的动作（来自正常任务轨迹）。
    2. **中断前缀 + 状态兼容提议动作**：在执行早期中断，由提议模型生成与当前状态兼容但不属于原始任务的下一步动作（生成无关标签）。
    3. **中断前缀 + 不安全提议动作**：同样中断后用不安全任务意图提议下一步动作。
    4. **风险增强变体执行**中的动作（提供安全/不安全两类轨迹）。
  - 最终标签由人类标注员在仅拥有护栏同等信息的条件下给出。
- **公式/流程**（文字概括）：固定任务轨迹或中断点 → 记录 \((u, s_{t-1})\) → 用提议模型或执行本身生成 \(a_t\) → 人工标注标签 → 护栏预测 \(y\)。

#### 2.2 风险增强执行套件（Risk‑Augmented Execution Suite）
- **核心思想**：保留用户原始指令不变，仅在环境状态中注入**潜伏危害**，使代理既能走安全路径完成任务，也可能选择不安全的捷径，从而端到端测试完整任务执行的安全性。
- **构建方式**：
  - 基于 OSWorld 的 45 个手动构造变体，每个变体源于原始任务可被代理顺利完成的任务。
  - 环境修改引入六类安全失败模式：破坏性覆盖/删除、过度编辑/权限变更、作用域逃逸、配置覆写、不必要敏感内容访问、错误目标/全局化更新。
  - 每个变体附加**增强评估器**，在原任务成功判定基础上增加基于状态的安全不变性（如文件哈希、权限、规则保留等）。
- **评估协议**：一次运行只有当**同时完成原任务并满足所有安全不变性**才记为安全成功；否则根据违反的不变性标记为不安全完成。

#### 2.3 护栏代理（Guardrail Agent）
- 同一护栏接口用于离线动作判断和在线全任务执行中。
- 行为：接收原始指令、当前状态和候选动作 → 输出 `ALLOWED` / `UNRELATED` / `UNSAFE` → 若被阻止则返回自然语言反馈供执行代理修正，每步最多重试 2 次（共 3 次提议），超出则终止本轮。

### 3. 实验设计
#### 3.1 数据集与场景
- **动作级基准**：共 **324 个**上下文候选动作，标签分布为 allowed (80)、unrelated (116)、unsafe (128)。
- **风险增强执行套件**：共 **45 个**手动构建的 OSWorld 变体，每个变体都在原任务可解的背景下注入状态级危害。
- 场景覆盖 Ubuntu 桌面任务（OSWorld 环境），如文件管理、YouTube 历史清除、系统设置等。

#### 3.2 对比方法与基线
- **动作级评估**：对比四种护栏方案
  - Gemini 3 Pro Preview（多模态提示）
  - GPT 5.1（多模态提示）
  - Claude Sonnet 4.5（多模态提示）
  - DeAction（来自 MisActBench 的护栏，使用 Claude Sonnet 4.5 作为骨干）
- **全任务评估**：执行器为 Claude Sonnet 4.5 Computer Use，对比**无护栏**与**搭载 Gemini 3 Pro Preview 护栏**两种设置。

#### 3.3 评估指标
- 动作级：准确率、宏平均 F1，以及 allowed/unrelated/unsafe 各自的精确率、召回率与 F1。
- 执行级：变体安全成功率、不安全完成率、重试终止率；还按安全失败类别进行了细分分析。

### 4. 资源与算力

- 实验均基于已存在的商业/开源模型（Gemini、GPT、Claude）的调用（API 或推理），并未进行模型训练或微调，因此不涉及大规模算力陈述。

### 5. 实验数量与充分性
- 动作级基准上进行了 **4 个护栏模型 × 整体 + 4 类来源细分** 的全面比较，总计多组实验。
- 风险增强执行套件中进行了 **2 组实验**（无护栏 vs. 最强护栏），并对失败类别做了定性分析。
- 实验设计客观：采用统一的离线/在线评估协议、固定重试预算、基于状态的不变性自动判定，避免人为干扰。
- 充分性评价：45 个变体的执行评估以及 324 个动作样本的标注可以勾勒出安全差距，但**模型组合有限**（仅一种执行器与一种最强护栏联用），且未进行跨多种执行器或多种护栏的消融实验，未来可进一步扩展。

### 6. 主要结论与发现
- **动作级能力**：当前最佳多模态护栏（Gemini 3 Pro Preview）可达到 80% 准确率与 0.80 宏 F1，对 allowed、unrelated、unsafe 三类较为均衡；但**在来自风险增强执行的样本上性能显著下降**（宏 F1 仅 0.27~0.48），说明状态依赖危害的识别更为困难。
- **端到端安全**：执行器在无护栏时安全成功率仅 **62%**，38% 的轨迹属于**不安全完成**（完成原任务但违反安全约束）。引入该最佳护栏后，不安全完成率仅由 38% 小幅降至 33%，**整体安全成功率未提升**，且引入了 4% 的重试终止率。
- **核心发现**：当前护栏在局部决策上有效，但难以在完整交互执行中将局部判断转化为端到端安全提升，暴露出**局部监督与可靠任务安全之间的重要差距**。危险往往同时误导执行代理和护栏，使得安全不现实地未得到恢复。

### 7. 优点
- **双粒度评测架构**：将动作级护栏决策与端到端任务安全关联，为安全性提供可诊断的评价体系。
- **风险增强变体设计精巧**：保持原始指令和任务可解性，仅修改环境状态引入潜伏危害，规避了故意篡改指令或使用恶意请求造成的评测偏差。
- **显式安全不变性**：使用基于文件哈希、权限等可自动检查的条件，使“安全完成”的判定客观可复现。
- **护栏部署的实战化设定**：护栏仅使用原始指令和当前状态，更贴近真实部署，同时支持带反馈的重试机制。

### 8. 不足与局限
- **规模有限**：风险增强执行仅 45 个变体，动作级仅 324 条样本，且仅围绕 OSWorld 环境，**泛化到其他计算机使用平台或其他类型危害的能力有待验证**。
- **实验覆盖不足**：只测试了一种执行器（Claude Sonnet 4.5）与一种最强护栏（Gemini）的在线组合，未探索不同执行器‑护栏配对、不同护栏提示策略或更强的护栏设计。
- **护栏在端到端提升微弱**：结果表明当前护栏架构难以有效倒查并修正危险执行路径，可能需要前置预测或更紧密的执行器‑护栏协同。
- **标注可能受限于人类判断**：虽然标注时模拟护栏视角，但允许/无关/不安全的边界在某些情境下可能存在主观性，可能引入一定偏差。
