## 摘要
Jupyter Notebook已成为众多领域日益流行的编码环境，尤其在基于Python的数据科学和科学计算中。最初用于原型设计和交互式探索的notebook，如今越来越多地被用来开发更复杂的程序，这导致GitHub等平台上存在缺陷的notebook数量迅速增加。为应对这一趋势，我们提出了JupOtter，一个专门为Jupyter Notebook设计的缺陷检测系统。JupOtter具有三项创新贡献：(1) 一种保留单元格结构的notebook专用分词策略，(2) 一种单元格级缺陷预测技术，以及(3) 一个名为OtterDataset的新标注数据集，包含超过21,000个经过细粒度单元格级缺陷标注的notebook。在三个评估数据集中的两个上，JupOtter的单元格级缺陷检测F1分数超越了静态分析器和大语言模型。

## Abstract
Jupyter Notebooks are an increasingly popular coding environment used across many domains, especially in Python-based data science and scientific computing. Originally used for prototyping and interactive exploration, notebooks are increasingly used to develop more complex programs, leading to a rapid rise in buggy notebooks on platforms like GitHub. To address this trend, we present JupOtter, a bug detection system designed specifically for Jupyter Notebooks. JupOtter features three novel contributions: (1) a notebook-specific tokenization strategy that preserves cell structure, (2) a cell-level bug prediction technique, and (3) a new labeled dataset, OtterDataset, containing over 21,000 notebooks annotated for fine-grained cell-level bug detection. JupOtter achieves cell-level bug detection F1 scores that surpass static analyzers and large language models in two out of three evaluation datasets.

### 2. 论文提出的方法论

#### 2.1 核心思想
- 针对 Notebook 的非线性执行特性和单元格结构，设计**保留单元格边界**的 tokenization 策略，使模型能够感知每个单元格的起止位置；基于此构建**每个单元格独立的缺陷预测**，最终输出一个与 Notebook 单元格序列对应的二进制标签向量（0 表示无缺陷，1 表示有缺陷）。

#### 2.2 关键技术与算法流程
- **OtterDataset 构建**：
  - 通过 GitHub Search API 和 32 个搜索查询（覆盖多种库和异常类型）收集 Python Notebook。
  - 利用 Notebook 已有的执行输出（output cells）提取错误信息（如错误类型、行号等），结合 AST 解析每个单元格以捕获语法错误，排除环境相关错误（如 `ImportError`, `FileNotFoundError` 等），为每个单元格生成二进制标签。
  - 去重并保证训练/测试集在作者‑仓库级别不重叠，最终得到 21,303 个 Notebook，其中 9,937 个含缺陷，包含总计 43,789 个缺陷单元格（缺陷率约 7%）。
- **Notebook 感知的分词策略**：
  - 在每个单元格前后插入特殊的开始/结束标记（如 `<cell_start>`, `<cell_end>`），确保模型在 token 序列中可精确定位每个单元格边界。
  - 将 Notebook 分割为多个非重叠的 **chunk**（块），每个 chunk 由完整的单元格序列组成，其总 token 数不超过预设的最大长度（2500 token）。只有极少数超长单元格会被截断（占 0.15%）。
  - 最终每个 Notebook 被表示为三维张量：输入 token、注意力掩码、标签。
- **单元格级缺陷预测模型**：
  - 使用预训练的 Transformer 编码器（CodeT5-small 或 CodeT5-base）对每个 chunk 进行编码，获得上下文相关的隐藏状态。
  - 通过特殊标记定位每个单元格的 token 范围，对范围内所有 token 的隐藏状态取平均，得到一个单元格嵌入表示。
  - 该嵌入表示通过一个线性分类层输出单个 logit，经 sigmoid 得到该单元格为缺陷的预测概率。
  - 所有 chunk 的预测结果拼接后得到整个 Notebook 的单元格级预测向量。
- **训练策略**：
  - 使用 `BCEWithLogitsLoss` 对所有单元格的预测与真实标签计算损失。
  - 采用混合精度训练和按样本（notebook）梯度累积，以处理长序列带来的内存压力。
  - 每个 notebook 最多使用前 4 个 chunk 进行训练（覆盖 93% 的 notebook 的全部单元格），batch size 设为 4，训练 10 个 epoch。

### 3. 实验设计

#### 3.1 数据集划分
- **OtterDataset 测试集**：从 21,303 个 Notebook 中随机抽取 20% 作为 held‑out 测试集（4,113 个文件，49% 含有缺陷）。
- **外部数据集**：
  - **Jupyter Errors Dataset**（9,313 个 Notebook）：已知至少含有一个运行时错误，用于评估对纯错误 Notebook 的泛化能力。
  - **CodeParrot GitHub Jupyter 子集**（4,769 个 Notebook）：来自大型原始 Notebook 采集，具有更一般的分布，23% 含有缺陷。
- 所有外部数据集都采用与 OtterDataset 相同的自动标注流程进行单元格级标注，并确保与训练集无作者/仓库重叠。

#### 3.2 对比方法
- **单元格级对比（RQ1）**：
  - **Flake8**：静态分析工具，将 Notebook 转为 Python 脚本后运行，将报错行映射回单元格，仅报告至首个致命错误。
  - **GPT‑4o‑mini** 和 **Gemini 3 Flash**：通过逐步提示（incremental prompting）逐个单元格询问是否包含缺陷，每次提示包含前面所有单元格以保持上下文。由于 API 调用开销巨大，每种数据集随机抽取 100 个 Notebook 进行评估。
- **文件级对比（RQ3）**：
  - Flake8、GPT‑4o‑mini、Gemini 3 Flash 均直接对整个 Notebook 进行“是否含缺陷”的二元判断，设置统一的二值回答提示。
  - 由于传统缺陷预测模型（如 DeepLineDP, LineVul）未针对 Notebook 结构设计，未纳入比较。

#### 3.3 评估指标
- **单元格级**：报告按单元格聚合与按文件聚合的 Precision、Recall、F1 和 Accuracy。主论文重点展示按文件聚合的指标，以减轻大 Notebook 的偏差。
- **文件级**：直接比较文件级别的二分类各项指标。
- **RQ2**：针对五种最常见错误类型（`TypeError`, `NameError`, `SyntaxError`, `ValueError`, `AttributeError`）分别评估 JupOtter 的单元格级性能。
- **RQ4**：消融实验：
  - 不使用 Notebook 分词策略、只做文件级预测的基线。
  - 保留单元格边界但不做 chunk 分割的基线。
  - 仅用 token 嵌入直接分类（无编码器）的基线。

### 6. 论文的主要结论与发现

- **单元格级检测**：JupOtter 在 OtterDataset 测试集和 CodeParrot 上取得高 F1 值（small 模型最高 0.93, base 模型 0.89），在 Jupyter Errors 数据集上 F1 较低（0.34），但精度仍高。整体表明 JupOtter 具有良好的泛化能力，尤其对常见实现错误，但对某些运行时错误的召回有限（部分原因是未被执行的含 bug 单元格在标注中计为无 bug）。
- **错误类型差异**：对 `SyntaxError` 检测效果最好（F1 0.87），`TypeError` 和 `ValueError` 次之（0.64, 0.63），对 `NameError` 和 `AttributeError` 较弱（0.34, 0.38）。
- **文件级对比**：JupOtter‑base 在 OtterDataset 和 CodeParrot 上的文件级 F1（0.76, 0.68）优于 Flake8、GPT‑4o‑mini 和 Gemini 3 Flash。LLM 在 Jupyter Errors 数据集上表现很强（F1 0.87），但可能存在偏向“有 bug”的预测。JupOtter 更保守，在缺乏 bug 的 Notebook 上精度更高。
- **消融与分词策略价值**：Notebook 感知的分词和 chunk 分割不仅小幅度提升文件级 F1（+3%），更重要的是使 **84% 原本因长度被截断的 Notebook 得以完整处理**，并且是单元格级定位的基础。无编码器的直接分类完全失败。
- **实际意义**：JupOtter 提供可操作的单元格级反馈，适合集成到 JupyterLab / VS Code 等环境作为实时代码质量助手。

### 7. 优点

- **问题定义新颖且实用**：首次聚焦 Notebook 的单元格级缺陷检测，弥补从文件级/函数级到单元格级的粒度空缺，直接契合 Notebook 的非线性交互特征。
- **技术方案针对性强**：
  - 保留单元格边界的分词策略与按 chunk 处理长序列，既保持结构又解决长上下文限制，使模型能够完整覆盖大型 Notebook。
  - 单元格级预测不依赖假想的执行顺序，通过编码器捕捉跨单元格关系，更贴近真实的非顺序执行模式。
- **数据集贡献突出**：公开发布的 OtterDataset（21k+ 标注 Notebook）是首个大规模单元格级缺陷标注数据集，为后续研究提供基础。
- **实验设计严谨**：
  - 数据集划分避免了作者/仓库级别的重叠，防范数据泄露。
  - 同时使用内部 held‑out 数据和两个外部数据集，多角度评估泛化性。
  - 人工标注验证给出了量化一致性。
  - 与静态分析器和两种现代 LLM 进行了公平比较（虽受 API 资源限制，但文件级评估覆盖完整测试集）。
- **可复现性高**：提供了代码、模型参数、数据集和基线配置（Zenodo），有利于复现。

### 8. 不足与局限

- **部分错误类型检测薄弱**：对 `NameError` 和 `AttributeError` 等常见运行时错误召回率低，可能与训练数据中此类错误占比小、或未执行单元格含 bug 而被标记为“无 bug”有关。
- **标注依赖已执行输出**：依赖于 Notebook 保存时的执行输出，这意味着某些运行时错误（若用户从未运行到该单元格）不会被标记，导致召回评估为下界；且无法发现逻辑错误或不触发异常的缺陷。
- **运行时信息不可用于实际预防**：JupOtter 的目标是在执行前检测，但训练标注来自历史执行输出，这种“先知”信息与实际部署时的输入存在差距。
- **单元格超长截断**：极少数超长单元格（0.15%）会被直接截断，无法处理。
- **Markdown 上下文未使用**：仅处理代码单元格，未利用标题等 Markdown 结构，可能丢失高层的语义脉络。
- **模型规模较小**：仅使用 CodeT5‑small/base，未探索更大模型（如 CodeT5+、CodeLlama 等），可能限制性能上限。
- **LLM 比较不完全**：由于成本限制，单元格级 LLM 比较仅基于随机抽样的小子集，可能不完全代表全貌。
