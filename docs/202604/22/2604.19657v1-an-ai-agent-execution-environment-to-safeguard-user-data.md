## 摘要
AI 智能体有望成为用户的通用个人助手，这要求它们能够访问用户的私有数据（例如个人和财务信息）。这给安全和隐私带来了严重风险。攻击者可能会攻击 AI 模型（例如通过提示词注入）来窃取用户数据。此外，与 AI 智能体共享私有数据需要用户信任可能不道德或已被攻破的 AI 模型提供商。本文提出了 GAAP（智能体隐私保证核算），这是一种为 AI 智能体设计的执行环境，可保证用户私有数据的机密性。通过动态和定向的用户提示，GAAP 从用户处收集权限规范，描述其私有数据可以如何被共享，并强制要求智能体对私有用户数据的披露（包括向 AI 模型及其提供商的披露）符合这些规范。至关重要的是，GAAP 确定性地提供这种保证，无需信任智能体处理私有用户数据，也不要求 AI 模型或用户提示免受攻击。GAAP 通过跟踪 AI 智能体如何访问和使用私有用户数据来强制执行用户的权限规范。它通过新型持久化数据存储和标注增强了信息流控制（Information Flow Control），使其能够跟踪跨单个任务执行步骤以及跨时间分隔的多个任务的私有信息流。我们的评估证实，GAAP 能够阻止所有数据披露攻击，包括那些会导致其他最先进系统向不受信任方披露私有用户数据的攻击，且不会对智能体的效用产生显著影响。

## Abstract
AI agents promise to serve as general-purpose personal assistants for their users, which requires them to have access to private user data (e.g., personal and financial information). This poses a serious risk to security and privacy. Adversaries may attack the AI model (e.g., via prompt injection) to exfiltrate user data. Furthermore, sharing private data with an AI agent requires users to trust a potentially unscrupulous or compromised AI model provider with their private data.   This paper presents GAAP (Guaranteed Accounting for Agent Privacy), an execution environment for AI agents that guarantees confidentiality for private user data. Through dynamic and directed user prompts, GAAP collects permission specifications from users describing how their private data may be shared, and GAAP enforces that the agent's disclosures of private user data, including disclosures to the AI model and its provider, comply with these specifications. Crucially, GAAP provides this guarantee deterministically, without trusting the agent with private user data, and without requiring any AI model or the user prompt to be free of attacks.   GAAP enforces the user's permission specification by tracking how the AI agent accesses and uses private user data. It augments Information Flow Control with novel persistent data stores and annotations that enable it to track the flow of private information both across execution steps within a single task, and also over multiple tasks separated in time. Our evaluation confirms that GAAP blocks all data disclosure attacks, including those that make other state-of-the-art systems disclose private user data to untrusted parties, without a significant impact on agent utility.

### 1. 核心问题与整体含义（研究动机和背景）
*   **研究动机**：AI 智能体（Agent）在执行预订机票、处理邮件等任务时，必须访问用户的私有数据（如身份证号、凭据、财务信息）。然而，现有的智能体架构面临三大威胁：
    1.  **提示词注入攻击（Prompt Injection）**：攻击者通过恶意指令诱导智能体泄露上下文中的私有数据。
    2.  **不可信的模型提供商**：用户可能不希望将原始私有数据暴露给云端 AI 模型及其背后的公司。
    3.  **智能体幻觉或误操作**：即使没有攻击，智能体也可能因错误逻辑导致数据外泄。
*   **核心问题**：如何在**不信任 AI 模型、不信任用户提示词、不信任外部工具**的前提下，为 AI 智能体提供确定性（Deterministic）的私有数据机密性保障？

### 2. 方法论：核心思想与关键技术
GAAP（Guaranteed Accounting for Agent Privacy）的核心思想是将智能体的执行从“直接调用工具”转变为“生成代码工件并受控执行”，并结合**信息流控制（IFC）**技术。

*   **核心流程**：
    1.  **代码生成**：智能体不直接操作数据，而是生成一段 Python 代码工件。
    2.  **拦截与分析**：GAAP 环境拦截代码中的 API 调用，利用 Pyre 等工具进行静态和动态的信息流分析，追踪私有数据的去向。
    3.  **权限校验**：在数据发送到外部 API 前，GAAP 检查权限数据库。若无授权，则暂停执行并询问用户。
*   **关键组件**：
    *   **私有数据库 (Private Data DB)**：存储敏感值。智能体仅能看到数据的“键”（Key），只有在代码运行时，GAAP 才会根据策略注入“值”。
    *   **权限数据库 (Permission DB)**：存储“数据类型 + 外部实体”的授权对（如：允许将“生日”发送给“航空公司”）。
    *   **披露日志 (Disclosure Log)**：记录所有历史披露。它能识别间接流，例如：昨天发给服务器 A 的数据，今天被读取回来后又发给服务器 B，GAAP 依然能追踪其来源。
    *   **标注框架 (Annotation Framework)**：对外部 MCP（模型上下文协议）服务进行语义标注，精确定义哪些参数会产生泄露，减少过度污染（Overtainting）。
    *   **多轮执行支持 (Multi-shot)**：允许智能体在获得用户许可后，将中间结果传回 LLM 以生成后续代码。

### 3. 实验设计
*   **数据集/场景**：
    *   **自定义基准测试**：包含 20 个复杂任务（改编自 AgentBench, MCP-Bench, AgentDojo），涵盖食品订购、医疗咨询、税务处理、邮件分类等。
    *   **攻击测试**：设计了三种提示词注入攻击：`SSN-leak`（强制在所有调用中加入社保号）、`Phone-leak`（在邮件中加入电话）、`SSN-swap`（将电话替换为社保号以隐蔽外泄）。
*   **对比方法**：
    *   **NP-Agent**：无保护的普通智能体。
    *   **LLM-Judge**：使用另一个 LLM 作为“法官”来审查调用（概率性防御）。
    *   **Conseca**：基于上下文策略的沙箱系统。
    *   **CaMeL**：另一种基于代码生成和 IFC 的系统（但不支持跨任务追踪和多轮执行）。
*   **Benchmark**：使用 **AgentDojo** 评估智能体的效用（任务成功率）。

### 4. 资源与算力
*   **硬件环境**：虚拟机环境，运行 Linux 6.8.0，配备 Intel Xeon E5-2620 CPU 和 64GB 内存。
*   **模型使用**：实验使用了 **OpenAI GPT-5**（注：论文发表时间设定为 2026 年）通过 API 进行测试。
*   **软件实现**：GAAP 核心由 Python 编写（约 3500 行代码），使用 Sqlite 存储数据库。

### 5. 实验数量与充分性
*   **实验规模**：对 20 个任务进行了 5 轮重复实验以评估效用；对 3 种攻击在所有适用任务上进行了全量测试。
*   **充分性评价**：实验设计较为全面。不仅测试了防御成功率（安全性），还深入对比了延迟、Token 消耗成本和用户交互负担。通过引入 AgentDojo 这一第三方基准，增强了实验的客观性和公平性。

### 6. 主要结论与发现
*   **安全性**：GAAP 实现了 **0% 的攻击成功率**，即完全拦截了所有数据泄露尝试，而对比方法（如 LLM-Judge 和 Conseca）均存在漏网之鱼。
*   **效用保持**：GAAP 的任务成功率（76%）与无保护智能体（81%）接近，且在 AgentDojo 上的表现优于其他隐私保护方案。
*   **成本与延迟**：GAAP 虽然增加了代码生成的输出 Token 数量，但由于减少了重复的上下文输入，总成本反而略低于 NP-Agent。延迟方面，GAAP 仅比普通智能体高出约 13%。
*   **用户负担**：随着权限数据库的积累，用户被询问的频率会随时间显著下降（实验中下降了约 35%）。

### 7. 优点
*   **确定性防御**：不依赖 LLM 的道德对齐或识别能力，从系统底层通过 IFC 强制执行隐私。
*   **跨任务追踪**：通过披露日志解决了“数据洗白”问题（即数据经外部服务器中转后丢失标签的问题）。
*   **不信任模型**：私有数据值在本地注入，模型提供商无法接触到未授权的原始敏感信息。
*   **多轮执行**：克服了传统代码生成方案无法处理中间反馈的局限。

### 8. 不足与局限
*   **用户交互压力**：在系统使用初期，用户需要频繁处理权限请求，可能影响体验。
*   **标注依赖**：系统的精确度部分依赖于对外部 MCP 服务的标注，若标注错误或缺失，可能导致过度阻断。
*   **存储开销**：披露日志会随时间无限增长，论文虽提到了压缩思路，但未给出长期的存储管理方案。
*   **应用限制**：目前主要针对基于工具调用的智能体，对于直接在提示词中包含大量私有数据的交互模式支持较弱。
