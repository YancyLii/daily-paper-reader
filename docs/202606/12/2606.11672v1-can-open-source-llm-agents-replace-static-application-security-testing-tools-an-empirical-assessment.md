## 摘要
本文探讨了智能体AI工具在网络安全领域的价值。我们评估了基于通用GenAI大语言模型（GenAI-）的智能体在由Ollama托管的三种不同通用开源模型驱动时的效能。我们使用精确率、召回率、假阳性数量以及基于这些指标相互作用计算得出的综合得分，对照现有的、经过验证的静态应用安全测试（SAST）工具Bandit的基线性能，评估每个智能体的表现。我们的研究结果反驳了这样一种观点，即现代开源GenAI LLM智能体目前适用于在现实条件下执行SAST扫描这一专门任务。

## Abstract
This paper explores the value of agentic AI tools for cybersecurity purposes. We evaluate the efficacy of a general-purpose GenAI Large Language Model- (GenAI-) based agent when powered by three different Ollama-hosted general-purpose open source models. We assess each agent's performance using precision, recall, false positive count, and a calculated composite score based upon the interplay of the captured metrics, against the baseline performance of an existing, vetted Static Application Security Testing (SAST) tool, Bandit. Our findings refute the notion that a modern open-source GenAI LLM-based agent is currently suitable for the specialized task of SAST scanning under realistic conditions.

## 1. 研究动机与核心问题

- **研究背景**：大语言模型（LLM）和智能体（Agentic AI）正被广泛应用于代码生成与软件工程，但在安全领域，尤其是静态应用安全测试（SAST）中，能否由通用 LLM 智能体替代专用工具尚无定论。
- **核心问题**：开源通用 LLM 驱动的智能体在真实场景下的 SAST 扫描任务中，能否达到甚至超越传统规则/模式匹配工具（如 Bandit）的性能？
- **整体含义**：该工作旨在通过实证评估，为“LLM 智能体是否已具备替代专业 SAST 工具的能力”提供一个基于数据的否定性答案，并为后续安全自动化路径选择提供参考。

## 2. 方法论

- **智能体设计**：基于 Python 构建一个名为 **Snitch** 的局部 SAST 智能体。它通过 Ollama 托管多个开源 LLM，递归扫描指定目录下的 Python 源文件，对每个文件调用固定提示词，要求模型输出包含漏洞位置、类型、原因和修复建议的 JSON 对象。
- **提示策略**：采用单一、固定的系统提示，明确定义智能体为资深网络安全工程师，并列出应检测的漏洞类别（SQL 注入、硬编码密钥、加密弱点等），要求按结构化 JSON 返回。
- **评估指标**：
  - **召回率（Recall）**：针对 Bandit 标记的中、高风险问题，计算各模型检测到的比例（完全匹配计 1 分，部分匹配计 0.5 分）。
  - **假阳性率（FP rate）**：人工判断的假阳性数 / 潜在假阳性总数。
  - **综合得分（Composite）**：**复合得分 = 召回率 − 假阳性率**，假阳性率越高，得分越低。
- **对照基线**：使用成熟的 Python SAST 工具 **Bandit**，以其输出作为比较基准和“事实源”。

## 3. 实验设计

- **数据集/扫描目标**：三个开源的 Python 代码仓库，具备不同规模和成熟度：
  - **Beaverhabits**（个人习惯追踪器，71 文件，9,124 行代码）
  - **Fail2ban**（安全守护进程，81 文件，34,569 行代码）
  - **Yum**（企业级包管理器，72 文件，58,123 行代码）
- **对比对象**：Bandit 基线 vs. 三个 **Ollama 托管的开源 LLM**（均为通用模型）：
  - **gemma3**（Google，编码专长）
  - **llama3.1**（Meta，通用型）
  - **qwen2.5**（Alibaba，推理/编码/调试）
- **实验流程**：
  1. 用 Bandit 扫描三个仓库，采集中、高风险漏洞列表。
  2. 对每个仓库，分别用上述三个 LLM 驱动 Snitch 进行扫描（共 9 次）。
  3. 使用另一 LLM（Claude）辅助生成交叉对比脚本，初步匹配输出，再经多人人工审核，判断真实匹配、漏报和假阳性。
  4. 计算每个模型-仓库组合的召回、假阳性率和复合得分。

## 6. 主要结论与发现

- **核心结论**：在真实场景下，当前**开源通用 LLM 智能体无法替代专业 SAST 工具 Bandit**，无论采用哪种模型。
- **量化结果**：
  - 召回率普遍**低于 0.25**，多数接近 0，说明大量已知漏洞未被检出。
  - 假阳性率**高达 40%~90%**，模型频繁将正常代码（如 `.env` 文件、许可证注释）误判为安全漏洞。
  - 综合得分**均为负值**，远低于理想水平。
- **定性观察**：
  - **严重幻觉**：模型经常编造不存在的文件名、行号，或对代码意图产生误解。
  - **上下文窗口限制**：难以正确理解项目架构模式，导致大量误报。
  - **高语言自信、低事实准确**：输出语气往往很自信，但内容错误。
  - **执行速度**：扫描时间比 Bandit 慢数十至数百倍，难以集成到快速开发流水线中。
- **相对表现为**：**qwen2.5** 在三个模型中综合表现最佳，但依然不具备实用价值；gemma3 和 llama3.1 表现更差。

## 7. 优点

- **明确的实证对比**：以真实的、经过验证的 SAST 工具作为基线，直接回答 LLM 智能体能否胜任专项安全任务。
- **多模型、多仓库评估**：选用三个不同来源的通用开源模型和三种不同类型的 Python 项目，具备一定的多样性。
- **指标设计合理**：不仅关注检出数，还结合召回率和假阳性率，用综合得分平衡两类错误，评价更全面。
- **人工校验增强可信度**：对智能体输出进行人工审核和交叉验证，减少了自动脚本可能引入的错误。
- **可复现性**：提供开源代码与数据（Snitch、结果文件），鼓励后续验证和改进。

## 8. 不足与局限

- **模型范围有限**：仅测试了三个开源通用模型，未涉及更强大的闭源模型（如 GPT-4）、专为安全训练的模型或多模态系统。
- **提示词未优化**：固定使用同一个提示词，未进行提示工程迭代。更好的提示或许能提升性能，但作者有意保持“通用”定位以检验现成能力。
- **基线单一**：仅以 Bandit 作为参照，而不同 SAST 工具各有侧重；未来应引入更多工具（如 Semgrep、CodeQL）丰富对比。
- **测试目标有限**：三个 Python 仓库虽然场景不同，但数量和覆盖领域仍不足，结论的泛化性需更多验证。
- **评价粒度较粗**：未区分漏洞严重程度或模型置信度；Bandit 的低严重问题被剔除，但智能体的部分误报可能实质上属于低严重性，这部分被统一视为假阳性可能过于严苛。
- **推理时间统计粗略**：未精确记录每个模型的耗时，给出的总时长仅为估计值。
- **未评估实际修复建议**：论文主要关注检测能力，未对智能体给出的修复方案进行正确性或可用性分析。
