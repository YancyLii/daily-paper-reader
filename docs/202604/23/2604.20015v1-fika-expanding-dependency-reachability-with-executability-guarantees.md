## 摘要
自动化第三方库分析工具通过解决关键的依赖管理挑战（如自动化版本更新、漏洞检测以及破坏性更新检测）来帮助开发人员。依赖可达性分析旨在通过将依赖问题的范围缩小到真正相关的部分，从而提高依赖管理的精确度。大多数依赖可达性分析工具是静态的，并且从根本上受到缺乏执行过程的限制。在本文中，我们提出了 FIKA，这是一个为第三方库调用点提供可执行性保证的流水线。FIKA 生成可执行代码，其执行轨迹为第三方库调用点的实际可达性提供了保证。我们将该方法应用于包含八个 Java 项目的数据集，以实证评估 FIKA 的有效性。平均而言，这些调用点中有 54% 被现有测试套件覆盖，因此具有可执行性的证据。FIKA 进一步将这一覆盖率提高了 20%，并能够证明 2363 个依赖方法的可执行性。在八个项目中的六个里，FIKA 提供了强有力的保证，证明超过 75% 的调用点是可执行的。我们进一步证明了 FIKA 能够改进 Semgrep（一种最先进的静态漏洞可达性分析工具）提供的结果。我们展示了在 Semgrep 产生不确定的可达性结果的情况下，FIKA 可以通过提供更强的可执行性保证，帮助确定漏洞更新的优先级。

## Abstract
Automated third-party library analysis tools help developers by addressing key dependency management challenges, such as automating version updates, detecting vulnerabilities, and detecting breaking updates. Dependency reachability analysis aims at improving the precision of dependency management, by reducing the space of dependency issues to the ones that actually matter. Most tools for dependency reachability analysis are static and fundamentally limited by the absence of execution. In this paper, we propose FIKA, a pipeline for providing guarantees of executability for third-party library call sites. FIKA generates code that is executed, and whose execution trace provides guarantees that a third-party library call site is actually reachable. We apply our approach to a dataset of eight Java projects to empirically evaluate the effectiveness of FIKA. On average, 54% of these call sites are covered by the existing test suites, and therefore, have evidence for their executability. FIKA further improves this coverage by 20% and is able to demonstrate executability for 2363 dependency methods. In six out of eight projects, FIKA provides strong guarantees that more than 75% of call sites are executable. We further demonstrate that FIKA is capable of improving the results provided by Semgrep, a state-of-the-art static vulnerability reachability analysis tool. We show that FIKA can help prioritize the vulnerability updates with stronger guarantees of executability in cases where Semgrep yields inconclusive reachability results.

#### 1. 核心问题与整体含义
在现代软件开发中，第三方库的依赖管理（如漏洞检测、版本更新）面临严重的“通知疲劳”问题。现有的自动化工具主要依赖静态分析来判断代码是否会调用某个库（可达性分析），但静态分析往往存在误报，且无法证明该调用路径在实际运行中是否真的可以被触发。
**FIKA** 的研究动机在于：通过结合静态分析、动态执行和大型语言模型（LLM），为第三方库的调用点提供**可执行性保证（Executability Guarantees）**。它不仅判断“理论上是否可达”，还通过自动生成并运行代码来证明“实际上可以执行”。

#### 2. 方法论
FIKA 采用了一个端到端的自动化流水线，主要包含以下四个阶段：
*   **静态分析（Static Analysis）：** 使用 CHA（类层次分析）算法生成项目的全局调用图，识别从公共入口点（Public Methods）到第三方库调用点（Call Sites）的最短路径，并提取相关的上下文信息（如源码片段、构造函数、Setter方法、成员变量等）。
*   **动态分析（Dynamic Analysis）：** 首先运行项目现有的测试套件，利用 JaCoCo 收集覆盖率数据，确定哪些调用点已被现有测试覆盖，从而建立基准。
*   **LLM 驱动的场景生成：** 针对未被覆盖的调用点，将静态分析提取的路径和上下文信息输入 LLM（本研究使用 DeepSeek V3.2），提示其生成一个“可达性场景”（Reachability Scenario）——一段旨在触发目标调用点的可执行代码。
*   **验证与迭代反馈：** 自动编译并运行生成的场景。如果失败（编译错误或未到达目标），FIKA 会捕获错误日志并将其作为反馈发送给 LLM 进行修复，最多迭代 5 次。

*   **Benchmark（基准）：** 以项目原有的开发者编写的测试套件（Developer-written tests）作为覆盖率基准。

*   **实验规模：** 针对 8 个主项目进行了详细的覆盖率提升实验，并针对 13 个包含已知漏洞的模块进行了漏洞验证实验。

#### 6. 主要结论与发现
*   **现有测试局限：** 即使是维护良好的开源项目，其测试套件平均也只能覆盖 54% 的第三方库调用点。
*   **FIKA 的提升：** FIKA 平均将调用点的可执行性保证提升了 **20%**，在某些项目中（如 Jooby），FIKA 几乎覆盖了所有静态可达的调用点。
*   **深度可达性：** FIKA 能够生成跨越 4 层以上方法调用的复杂场景。
*   **漏洞管理优化：** 在 Semgrep 给出“不确定（Undetermined）”或“松散匹配”的漏洞报告时，FIKA 能够通过生成实际执行代码，为 59 个 CVE 提供了确凿的可执行证据，帮助开发者精准排序修复优先级。

#### 7. 优点
*   **虚实结合：** 巧妙地利用静态分析为 LLM 提供精确的“导航”，克服了 LLM 盲目生成代码的缺陷。
*   **闭环反馈：** 引入了编译和运行时的自动反馈机制，显著提高了生成代码的成功率。
*   **实用性强：** 直接对接现有的漏洞扫描工具，解决了静态分析工具高误报率的痛点。
*   **低成本：** 相比人工编写测试，使用 LLM API 的成本极低。

#### 8. 不足与局限
*   **静态分析局限：** 依赖静态调用图，可能无法处理反射（Reflection）或动态代理等动态特性导致的隐藏路径。
*   **时间开销：** 对于大规模项目，生成数以百计的场景需要数十小时，可能不适合集成在每次提交的 CI 流程中，更适合发布前的深度扫描。
*   **生态限制：** 目前仅实现了对 Java/Maven/JUnit5 生态的支持，对其他语言或构建工具的迁移效果尚待验证。
*   **路径选择：** 目前仅选择最短路径进行尝试，可能错过某些特定条件下才能触发的复杂路径。
