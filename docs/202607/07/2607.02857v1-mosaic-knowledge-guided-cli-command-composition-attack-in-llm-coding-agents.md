## 摘要
LLM编码代理越来越多地通过发出普通CLI命令来完成开发任务。遵循Unix设计，这些命令通过共享操作系统状态进行协作：一个命令可能写入状态，而后续命令读取该状态。尽管这种组合是良性和预期的，但它却创造了一个被忽视的攻击面。现有的攻击和防御主要针对指令层，其中恶意意图表现为敌对文本。相比之下，我们观察到，单个良性命令可以在命令跟踪中形成危险的生产者-消费者状态关系，暴露出我们所谓的CLI命令组合风险（CCR）。鉴于这一新的攻击面，系统性地揭示和表征CCR在现实世界编码代理中的影响至关重要。然而，系统性地理解这种风险颇具挑战性，因为简单的命令枚举和端到端LLM生成大多产生无效的工作流程。我们提出了MOSAIC，一个知识引导的框架，它从CVE、安全公告和研究人员PoC中提炼出经过验证的命令状态行为，形成可复用的摘要，将其组合成利用路径，并实例化为真实的开发者工作流程，用于黑盒代理评估。在五个实际CLI编码代理和五个后端LLM上进行的2525次试验中，MOSAIC在良性开发者任务下实现了96.59%的攻击成功率。

## Abstract
LLM coding agents increasingly complete development tasks by issuing ordinary CLI commands. Following the Unix design, these commands cooperate through shared operating-system state: one command may write state that a later command reads. While this composition is benign and intended, it creates an overlooked exploit surface. Existing attacks and defenses mainly target the instruction layer, where malicious intent appears as hostile text. In contrast, we observe that individually benign commands can form a dangerous producer-consumer state relation across the command trace, exposing what we call CLI command-composition risk (CCR).   Given this new attack surface, it is critical to systematically uncover and characterize the impact of CCR in real-world coding agents. However, systematically understanding this risk is quite challenging, because naive command enumeration and end-to-end LLM generation produce mostly invalid workflows. We present MOSAIC, a knowledge-guided framework that distills validated command-state behaviors from CVEs, advisories, and researcher PoCs into reusable summaries, composes them into exploit paths, and instantiates them as realistic developer workflows for black-box agent evaluation. Across five real-world CLI coding agents and five backend LLMs over 2,525 trials, MOSAIC achieves a 96.59% attack success rate under benign developer tasks.

### 1. 核心问题与研究背景
- **核心问题**：LLM 编程代理越来越多地通过执行常规 CLI 命令来完成开发任务。然而，即便每个命令自身都无害，它们也可能通过共享的操作系统状态（如环境变量、文件、配置文件）形成危险组合，导致超出用户任务范围的安全后果（如远程代码执行、凭证泄露）。本文将这种新型风险定义为 **CLI 命令组合风险（CCR）**。
- **研究缺口**：现有 LLM 代理安全研究几乎全部聚焦于指令层（如提示注入），其攻击信号必须出现在恶意文本或单一动作中，而 **CCR 的恶意效果仅体现在多个良性命令通过状态依赖关系组合的轨迹中**，该攻击面被完全忽视。
- **研究目标**：系统性地揭示并刻画真实编程代理中的 CCR，提供一个知识引导的评估框架，以暴露当前防御的盲区。

### 2. 方法论：MOSAIC 框架
MOSAIC 是一个知识引导的三模块流水线，通过复用已验证的 CLI 安全知识来生成高质量的攻击实例。

- **模块一：CLI 安全知识库构建**
  - **知识获取**：从 NIST NVD、GHSA、Exploit-DB、CISA KEV 及安全研究博客等多源收集约 30,180 条记录，经去重和两阶段过滤（静态 CLI 白名单、LLM 辅助评审结合人工确认）后，保留 **454 条** 包含可复现命令级行为的 CLI 安全证据。
  - **知识组织**：将每条证据提炼为 **命令-状态摘要** $r = (\tau_r, \phi_r, P_r, Q_r, \kappa_r, E_r)$，分别描述命令模板、触发条件、产生的状态集、消费的状态集、暴露的能力和证据支撑。最终将其聚类为 **13 个命令-状态家族**，覆盖环境传播、hook 触发、路径劫持等主要状态传递通道。

- **模块二：命令组合实例生成**
  - **依赖链枚举**：基于状态摘要，识别满足 $P_{r_a} \cap Q_{r_b} \neq \emptyset$ 的生产者-消费者对，并枚举长度 $2 \leq n \leq 4$ 的候选利用路径 $\Pi$。
  - **可行性验证**：通过 LLM 辅助检查，确保链中每个状态转移都具备证据支撑且可在真实系统中实例化。
  - **攻击实例化**：对可行链，使用 LLM 辅助生成完整的攻击实例 $I = (q, R, C, m)$，包括一个良性的用户提示 $q$、一个攻击者控制的资源 $R$、预期的命令序列模式 $C$ 以及预期的外部效果标记 $m$。所有实例需通过模式、证据、状态逻辑和一致性静态校验。

- **模块三：运行时执行与风险验证**
  - 在隔离环境中，将攻击实例以良性开发者任务的形式提交给目标代理。记录实际的命令轨迹 $T$ 和外部效果 $E$。

### 3. 实验设计
- **评估对象**：5 个代表性真实 CLI 编码代理（Claude Code, Codex CLI, Gemini CLI, GitHub Copilot CLI, Trae Agent）与 5 个后端大模型（GPT-5.1, Gemini-2.5 Flash, Claude Haiku-4.5, DeepSeek-V4 Flash, Qwen3.7-Plus）的组合。
- **攻击集**：由 MOSAIC 生成的 **101 个** 覆盖代码执行、容器逃逸、持久化等 6 种终端能力的有效攻击实例，构成 2,525 次攻击试验（101 × 5 代理 × 5 模型）。
- **对比基线**：
  - **指令层攻击**：在完全相同的威胁模型下，使用 AIShellJack 和通用间接提示注入（IPI）两种 SOTA 方法，以 CCR 实例的最终目标为攻击目的进行对比。
  - **防御有效性**：选取五大类防御的代表性工具（指令扫描 PromptGuard 2、能力控制 Progent、信息流控制 CaMeL、静态命令扫描 Semgrep、任务对齐监控 AlignmentCheck）部署于代理之上，评估其对 CCR 的防御效果。
- **消融实验**：对比在无知识库、仅 LLM 自反思、原始安全记录 RAG 等条件下，生成攻击实例的有效率（IVR）和最终攻击成功率（ASR），以量化结构化知识库的贡献。

### 4. 资源与算力
- 文中 **未提及** 任何关于 GPU 型号、数量、训练时长或具体算力开销的信息。因为 MOSAIC 是一个 **无需训练的框架**，其核心流程为基于知识库的LLM API调用（用于内容生成与辅助判断）和黑盒代理交互测试，所需的主要是 LLM API 调用成本，而非传统的训练算力。

### 5. 实验数量与充分性
- **实验规模充分**：核心评估涉及 **2,525 次** 端到端攻击试验，覆盖主流代理和模型组合。此外还有对 101 个实例的防御评估及多组消融实验。
- **对比公平客观**：与指令层攻击的对比完全基于相同的威胁模型和攻击目标，仅攻击媒介不同（命令组合 vs. 恶意文本），公平地衡量了代理对两类攻击的识别差异。防御评估也是透明地将同一批攻击实例置于防御模块监控之下。

### 6. 主要结论与发现
- **CCR 是普遍且高危的风险**：MOSAIC 在真实代理上实现了 **96.59%** 的端到端攻击成功率，且所有代理和模型均高度受影响。同期指令层攻击基线最高成功率仅 **2.18%**，表明当前代理能有效拦截恶意指令，但对命令组合风险几乎无感知。
- **结构化知识库是关键**：消融实验显示，离开知识库后 LLM 几乎无法自行生成有效攻击实例（IVR 低至 8.91%），而完整 MOSAIC 可达 100% 实例有效率和最高的最终成功率，且生成质量不受构建所用 LLM 能力差异的影响。
- **现有防御全面失效**：五大类部署的防御手段中，三种（指令、能力、信息流控制）对攻击成功率无任何影响，最强的防御也仅将其降低至 82.57%，因为它们均无法观察和推理跨命令的 **OS 状态依赖关系**。

### 7. 优点
- **新颖的攻击面**：首次系统化定义了“CLI 命令组合风险”，突破了局限于指令层的安全研究范式，攻击更具隐蔽性且难以通过现有机制防御。
- **知识引导的创新方法**：巧妙利用海量、已验证的 CLI 安全记录（CVE等）作为先验知识，解决了攻击路径盲目生成无效的难题，保证了攻击实例的 **高保真、高可靠及可复现性**。
- **系统而扎实的评估**：实验设计覆盖了主流代理和模型、设计了公平的指令层基线对比，并测试了五种不同类别防御的弱点，结论具有极强的说服力。
- **防御启示**：明确指出未来防御需转向“来源感知”的运行时监控，即追踪并标记命令间产生的状态流转，为社区提供了清晰的研究方向。

### 8. 不足与局限
- **组合路径的覆盖度**：框架基于已有的 CVE 等知识构建了 101 条攻击路径，未能穷尽所有可能的 CLI 状态组合策略，可能存在未知的利用路径未被覆盖。
- **对知识库的依赖**：攻击实例的有效性高度依赖 CLI 安全知识库的广度与时效性。如果未来命令机制更新导致旧证据失效，或新的漏洞模式未收录，生成能力会受限。
- **实验环境的泛化性**：所有实验均在受控黑盒评估环境下进行，虽然模拟了开发者典型工作流，但与极度复杂、高度定制化的真实企业开发环境可能存在差异。
- **防御建议尚未验证**：文中提出了“来源感知防御”的方向，但并未构建和实验该防御原型，其实际有效性和性能开销有待验证。
