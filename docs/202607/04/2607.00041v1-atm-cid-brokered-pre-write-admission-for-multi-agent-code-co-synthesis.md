## 摘要
多智能体LLM系统可将软件工程工作分解为规划、生成、验证和修复，但仍存在一个更具体的系统问题：在实施任何受管理的共享修改之前，系统必须决定哪些并发形成的写入意图可以并行进行，哪些需要确定性的组合或序列化，以及哪些必须采取故障关闭路径。我们通过AI-Atomic框架（ATM）来解决这一问题，该框架是为在单一治理域内操作的软件智能体提供的基于规范的治理基板。ATM将任务意图、仓库范围、写入准入、验证和证据义务绑定到一个治理链中。内容标识符（CID）代理充当共享修改的准入子系统。适配器引导的原子化将写入意图映射到语义原子和有界区域；当持久原子映射覆盖不完整时，虚拟原子提供临时的可审计治理单元，用于保守比较和路由。受管理的共享写入最终由中立的执行员应用，而非由提议的智能体直接应用。评估结合了受控、现场、采用和扩展证据，包括12个场景的确定性设计矩阵、三个存档运行器案例、ATM准入基准、三个存档同文件边界案例、一项为期三周的外部采用者研究，以及一个操作恢复路由基准。结果支持在观察到的单一领域设置内的可行性、可审计性和有界可恢复性，但不声称广泛的比较优势或跨克隆治理。

## Abstract
Multi-agent LLM systems can decompose software-engineering work into planning, generation, validation, and repair, but a narrower systems problem remains: before any governed shared mutation is applied, a system must decide which concurrently formed write intents may proceed in parallel, which require deterministic composition or serialization, and which must take a fail-closed path. We address this problem with the AI-Atomic-Framework (ATM), a specification-grounded governance substrate for software agents operating within a single governance domain.   ATM binds task intent, repository scope, write admission, validation, and evidence obligations into one governance chain. A Content Identifier (CID) broker serves as the shared-mutation admission subsystem. Adapter-guided atomization maps write intents to semantic atoms and bounded regions; when persistent atom-map coverage is incomplete, virtual atoms provide temporary auditable governance units for conservative comparison and routing. Governed shared writes are ultimately applied by a neutral steward rather than directly by proposing agents.   Evaluation combines controlled, field, adoption, and extension evidence, including a 12-scenario deterministic design matrix, three archived runner cases, ATM-AdmissionBench, three archived same-file boundary cases, a three-week external-adopter study, and an operational recovery-routing benchmark. The results support feasibility, auditability, and bounded recoverability within the observed single-domain settings, but do not claim broad comparative superiority or cross-clone governance.

### 2. 论文提出的方法论

*   **核心思想**：
    *   ATM框架是一个“基于规范的治理基板（Specification-Grounded Governance Substrate）”，它在一个单一治理域内，将写入准入问题形式化。其核心思想是**将不透明的、共享代码库的并发修改风险，转化为可定位、可裁决、可重放的治理单元**。

*   **关键技术细节与架构**：
    *   **三层治理平面**：框架由三个平面构成，以解答不同层级的治理问题。
        1.  **任务合同平面**：通过`Task Contract`定义智能体被授权的操作范围，包含任务意图、允许/禁止的文件、验证要求等，并通过“方向锁”和“范围门”强制执行。
        2.  **变更准入平面**：核心是**CID代理（CID Broker）**，它作为共享变更准入的子系统，判断一个写入意图在特定时刻是否可以执行。
        3.  **证据闭合平面**：通过“验证器信封”和“证据阻断器”等机制，确保任务完成时具有可追溯的证据，而非仅靠智能体声明。
    *   **原子化与CID代理**：
        *   **适配器引导的原子化（Adapter-Guided Atomization）**：通过语言/格式特定的适配器（如TypeScript、Python、JSON），将智能体的写入意图保守地映射为**原子（Atom）**——最小的可治理语义单元（如函数、类、JSON记录等）。
        *   **原子映射（Atom Map）**：一个将原子与其所有者、依赖、测试、共享表面连接起来的可查询治理索引。
        *   **虚拟原子（Virtual Atom）**：当原子映射不完整或粒度过粗时，系统会动态创建**虚拟原子**作为临时治理单元。它将未原子化的代码范围包装成一个可比较的逻辑区域，从而支持受限区域的比较和准入裁决。
        *   **内容标识符（CID）**：为原子提供身份签名，分为用于准入比较的**候选CID**和用于验证后证据容器的**封装CID**。
    *   **七层准入门（Seven-Layer Hard Gate）**：
        代理接收到写入意图后，会通过一个确定性的七层管道进行逐层过滤，而非简单的行级比较。若在前几层不能排除冲突，则会进入后续的细粒度检查，直至裁决或进入故障关闭路径。
        1.  **CID身份**：检查两个意图是否指向同一个治理单元。
        2.  **共享表面**：检查是否触碰相同的注册表、生成器、工件等。
        3.  **读写依赖**：检查是否存在已声明的读写冲突。
        4.  **文件范围/虚拟原子**：检查同文件变更是否可被已知原子或虚拟原子分离。
        5.  **冲突键与合并**：检查结构化工件（如JSON）的键是否冲突。
        6.  **CAS基础哈希**：在应用写入前，校验文件基础哈希是否与准入时一致（乐观并发控制）。
        7.  **回退文件锁**：当所有证据都不足以支持并发准入时，采取保守的全文件加锁。
    *   **中立管家（Neutral Steward）**：所有被准予的共享写入，最终由**中立管家**执行，而非由提议的智能体直接写入，以此分离“提议权”与“执行权”。

### 7. 优点

*   **问题定义清晰且新颖**：将预写入准入作为一个独立的一级治理问题，并精准定位了在现有研究中从“文本级”和“AST级”解决方案之间的空白地带。
*   **框架设计精巧且严谨**：
    *   **适配器+虚拟原子**的设计非常实用，通过保守的、可逐步完善的方式解决了不同语言和原子化程度不足的问题，避免了“全有或全无”的AST依赖。
    *   **七层准入门**和**中立管家**的架构，将复杂的治理挑战分解为一组模块化、可组合、确定性的检查项，实现了逻辑关注点的分离，增强了系统的可解释性和安全性。
*   **评估方法高度透明**：采用了包含受控基准、现场证据和外部采用者证据在内的多维评估方法，并公开讨论了证据边界和局限性。冻结基准并进行角色分离的盲审，极大地提升了结论的可信度。

### 8. 不足与局限

*   **实验覆盖度不足**：
    *   **缺乏大规模外部对比**：未与STORM、CoAgent等同类系统进行头对头的统计对比，无法得出“更好”或“更快”的结论。
    *   **场景覆盖单一**：所有核心证据均来自**单一治理域**（同一台机器/工作树），无法外推至大型分布式代码仓库或云原生开发团队。
*   **偏差风险**：
    *   **利基场景**：虽然评估维度多，但多数被视为“利基”或“原型”状态，确定性矩阵的12个场景中也仅有3个完成了完整的运行，证据链的某些部分尚显薄弱。
    *   **作者参与度高**：自托管取证部分，作者既是系统设计者也是用户，这种自我参照证据的强度低于独立验证。
*   **应用限制**：
    *   **跨语言能力弱**：当前显式声明跨语言（如TypeScript和Python耦合的API）的原子身份仍是一个未解决问题。
    *   **可扩展性瓶颈**：单一代理进程的设计在极高并发或超大规模仓库下可能成为性能瓶颈。
    *   **适配器信任问题**：诚实依赖适配器的保守声明，论文明确指出对恶意或错误的适配器声明的防护能力有限。
    *   **不保证最终语义正确**：即使通过了准入，也不保证程序行为绝对正确，仍需验证器和CAS等作为运行时安全网。
