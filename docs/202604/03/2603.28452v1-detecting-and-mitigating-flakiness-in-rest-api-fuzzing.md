## 摘要
测试不稳定性（Test flakiness）是工业界的一个普遍问题，它阻碍了自动化构建和测试工作流的可靠性。现有关于测试不稳定性的大多数研究主要集中在单元测试和小型集成测试上。相比之下，诸如 REST API 等系统级测试中的不稳定性研究相对较少。虽然已有大量文献致力于 REST API 模糊测试，但在该背景下，检测并缓解不稳定性负面影响的研究却受关注较少。为了填补这一空白，本文研究了由文献中广泛应用的 REST API 模糊测试工具 EvoMaster 生成的测试的不稳定性，并利用包含 36 个 REST API 的语料库进行了实证研究。基于实证研究结果，我们通过检查近 3000 个失败的测试用例，对不稳定性的来源进行了分类和分析。在此基础上，我们提出了 FlakyCatch 以检测和缓解 REST API 中的不稳定性，并对其性能进行了实证评估。结果表明，FlakyCatch 能够有效检测和处理由白盒及黑盒模糊测试工具生成的测试不稳定性。

## Abstract
Test flakiness is a common problem in industry, which hinders the reliability of automated build and testing workflows. Most existing research on test flakiness has primarily focused on unit and small-scale integration tests. In contrast, flakiness in system-level testing such as REST APIs are comparatively under-explored. A large body of literature has been dedicated to the topic of fuzzing REST APIs, whereas relatively little attention has been paid to detecting and possibly mitigating negative effects of flakiness in this context. To fill this major gap, in this paper, we study the flakiness of tests generated by one of the popularly applied REST API fuzzer in the literature, namely EvoMaster, conduct empirical studies with a corpus of 36 REST APIs to understand flakiness of REST APIs. Based on the results of the empirical studies, we categorize and analyze flakiness sources by inspecting near 3000 failing tests. Based on the understanding, we propose FlakyCatch to detect and mitigate flakiness in REST APIs and empirically evaluate its performance. Results show that FlakyCatch is effective in detecting and handling flakiness in tests generated by white-box and black-box fuzzers.

#### 1. 核心问题与研究动机
*   **核心问题**：研究 REST API 模糊测试（Fuzzing）中生成的测试用例为何会出现“不稳定性”（Flakiness，即相同输入在相同环境下产生不同结果），并提出自动化的检测与缓解方案。
*   **研究动机**：
    *   测试不稳定性严重损害了开发者对自动化测试的信任，并干扰持续集成（CI）流水线。
    *   现有研究多集中于单元测试，而 REST API 作为有状态、分布式且依赖环境的系统级交互，其不稳定性来源更复杂且研究匮乏。
    *   API 模糊测试工具（如 EvoMaster）能生成大量测试，但若不处理其中的不稳定用例，会导致大量误报。

#### 2. 方法论：FlakyCatch
FlakyCatch 采用“检测-推理-处理”的后处理流水线，核心思想是在不破坏测试结构的前提下，精准识别并屏蔽不稳定的断言。
*   **不稳定性检测（Detection）**：
    *   **重复执行**：将模糊测试生成的测试用例在相同环境下多次运行，对比响应的状态码、响应头和响应体。
    *   **轻量级推理（Lightweight Inference）**：针对重复执行未能触发的潜在不稳定因素（如随机 ID、时间戳），利用模式匹配（正则表达式）识别符合 ISO 8601、UUID、Base64 等标准的字段，将其标记为潜在不稳定路径。
*   **不稳定性缓解（Mitigation）**：
    *   **精准屏蔽**：不直接删除整个测试用例，而是定位到响应体中具体的 JSON 路径。
    *   **断言重构**：将指向不稳定字段的断言（如 `.body("time", ...)`）注释掉，并添加注释说明观察到的不同取值，保留其他稳定的断言以维持测试的覆盖能力。

#### 3. 实验设计
*   **数据集**：使用 Web Fuzzing Dataset (WFD)，包含 **36 个真实世界的开源 REST API**，涵盖了不同的数据库（MySQL, MongoDB, PostgreSQL 等）和业务逻辑。
*   **实验场景**：
    *   **两种模式**：EvoMaster 的黑盒（BB）模式和白盒（WB）模式。
    *   **两种环境**：在两个硬件配置和操作系统版本不同的环境（FuzzEnv 和 ExecEnv）中交叉执行，以研究环境对不稳定性（如 Locale 设置）的影响。
*   **对比基准**：以未经过 FlakyCatch 处理的原始 EvoMaster 生成测试作为 Baseline。
*   **评估指标**：失败率（FR%）、持续失败数（Fc）、不稳定失败数（Fu）、A12 效应量等。

*   **硬件设备**：

#### 5. 实验数量与充分性
*   **实验规模**：
    *   涵盖 36 个 API，总计超过 65 万行代码。
    *   **手动分析了近 3000 个失败的测试用例**，以构建不稳定性分类学。
    *   总计进行了数百小时的模糊测试和数万次的测试回放。
*   **充分性评价**：实验设计非常充分且严谨。通过跨环境对比、多轮重复执行以及统计学显著性检验（p-value 和 A12），确保了结论的客观性和公平性。

#### 6. 主要结论与发现
*   **普遍性**：在 36 个 API 中，有 **31 个（86%）** 表现出了测试不稳定性，说明该问题在 REST API 中极其普遍。
*   **白盒 vs 黑盒**：白盒模糊测试比黑盒更容易暴露不稳定性，因为它能触及更深层的系统状态。
*   **九大来源**：归纳了 9 类来源，其中 **运行时环境（Env）**、**无序集合（Unord）**、**有状态资源（State）** 和 **运行时消息（RunMsg）** 是最主要的诱因。
*   **工具效果**：FlakyCatch 显著降低了测试套件的失败率，在白盒模式下平均处理了超过 1 万个不稳定断言，并在多个 API 中完全消除了不稳定性。

#### 7. 优点
*   **实证深入**：提供了首个针对 REST API 模糊测试不稳定性的系统性分类学，手动标注的数据集对后续研究极具价值。
*   **实用性强**：FlakyCatch 不需要访问源代码（仅依赖响应分析），适用于黑盒和白盒场景，易于集成到工业界现有的模糊测试工具中。
*   **策略精细**：采用“注释断言”而非“删除测试”的策略，最大限度地保留了测试的回归检测能力。

#### 8. 不足与局限
*   **工具单一性**：实验主要基于 EvoMaster 这一种模糊测试工具，虽然它是 SOTA，但结论在其他工具（如 RESTler）上的泛化性有待验证。
*   **难以根治的类型**：对于“有状态资源（State）”和“环境依赖（Env）”引起的不稳定性，仅靠响应分析难以完全解决，可能需要更深层的系统重置或容器化隔离技术。
*   **误报风险**：轻量级推理（模式匹配）可能会误将某些本应稳定的字段标记为不稳定，从而导致过度屏蔽断言。
