## 摘要
使用工具的LLM智能体越来越多地读取不可信内容，同时持有具有副作用的工具，如支付、电子邮件、CRM和基础设施API，然而常见的框架默认设置仍将工具暴露与授权混为一谈。我们审计了LangChain/LangGraph、LlamaIndex和Stripe Agent Toolkit在执行前是否对每个模型发出的调用及其具体参数值进行重新授权。在所选取的公开源代码版本中，这三种框架均默认提供能力门控，但无一默认提供确定性的、失效关闭的逐调用值授权门控。我们提出ScopeGate，一个面向智能体工具调用的五阶段PDP/PEP：范围、授权、金额上限、幂等性和默认拒绝。评估显示，相同的未授权支付调用在LangChain的默认调度下得以执行（并配有LlamaIndex概念验证），但被ScopeGate拒绝；受测控制报告了0/48静态绕过、0/29未授权尝试（40轮自适应运行）、0/10良性误拒，以及Latam-GPT支付智能体在10/10场景下被成功遏制。ASR表示尝试的未授权操作，遏制并非根治方法，部署层面的声明是对所测模型类别的推断，且未断言任何CVE。

## Abstract
Tool-using LLM agents increasingly read untrusted content while holding side-effecting tools such as payments, email, CRM, and infrastructure APIs, yet common framework defaults still conflate tool exposure with authorization. We audit whether LangChain/LangGraph, LlamaIndex, and the Stripe Agent Toolkit re-authorize each model-emitted call, with concrete argument values, before execution. Across pinned public-source commits, all three provide capability gating by default, but none provides a deterministic fail-closed per-call value authorization gate by default. We introduce ScopeGate, a five-stage PDP/PEP for agent tool calls: scope, authorization, money ceiling, idempotency, and default deny. Evaluation shows the identical unauthorized payout call executes under LangChain's default dispatch (with a companion LlamaIndex PoC) but is denied by ScopeGate; the tested control reports 0/48 static bypasses, 0/29 unauthorized attempts (40-iteration adaptive run), 0/10 benign false-denies, and Latam-GPT payment-agent containment at 10/10. ASR denotes attempted unauthorized action, containment is not a cure, deployment-tier claims are inference over measured model classes, and no CVE is asserted.

### 1. 论文的核心问题与研究背景
- **核心问题**：当前主流的LLM Agent框架（如LangChain、LlamaIndex、Stripe Agent Toolkit）在默认设置下，仅提供了**工具暴露的能力门控（capability gating）**，却缺少对**每次工具调用的具体参数值进行重新授权的确定性机制**。当LLM读取不受信任的内容（如邮件、网页、检索文档）并受到间接提示注入攻击时，这种设计缺陷会导致**混淆代理人（confused deputy）失败**：被攻击者控制的模型输出被框架直接执行，从而引发未授权的支付、CRM修改或基础设施操作。
- **背景与动机**：工具使用型LLM Agent已广泛应用于支付、客户管理等高权限场景，传统框架将模型视为受信的“规划者”，但实际上模型只是“不由信解析器”。已有大量工作证实间接提示注入的可行性，但系统层面缺乏对框架默认授权边界的审计。作者的目标不是发现全新的注入攻击，而是证明**能力门控≠授权**这一系统安全原则在主流框架中的缺位，并提供可部署的补救方案。

### 2. 方法论：ScopeGate 五阶段策略决策/执行点
- **核心思想**：在模型输出工具调用后、实际执行副作用前，插入一个确定性的、默认拒绝的策略执行点（PDP/PEP），所有授权决策完全依赖**带外策略**（由运维人员预定义，不受模型上下文影响），实现完整中介。
- **ScopeGate 判定流程（Algorithm 1）**：
  1. **范围检查（Scope）**：工具名称是否在策略允许列表中？未列出的工具直接拒绝。
  2. **授权检查（Authorization）**：参数中受约束的值（如账户ID、支付意图、URL）是否在预定义的**允许列表**中？例如，只允许 `acct_MERCHANT_001`，拒绝 `acct_ATTACKER_999`。
  3. **金额上限检查（Money Ceiling）**：仅对支付/退款工具生效；金额必须是有限实数且在 `[0, ceiling]` 范围内，拒绝 `NaN`、无穷大、复数等形态。
  4. **幂等性检查（Idempotency）**：若工具标记为需要幂等键，上下文中必须存在受信任的幂等键，否则拒绝（防止重放放大攻击）。
  5. **默认拒绝**：通过所有阶段的调用才允许执行；任何运行时错误或策略缺失均导致拒绝（fail-closed）。
- **关键属性**：策略（工具范围、值白名单、金额上限）完全来自外部配置，不能由模型动态生成或来自任何模型输出路径，避免了注入对抗策略本身的风险。

### 6. 主要结论与发现
- 默认框架**普遍混淆能力门控与授权**：LangChain、LlamaIndex、Stripe Agent Toolkit在公开源码层面均未提供按值授权控制，当集成者暴露具有副作用的工具时，框架即成为混淆代理人。

- **ScopeGate能可靠遏制可治理工具上的未授权操作**：在测试集合中实现0误放行、0误拒绝，并成功阻止支付场景下所有攻击尝试，但明确声明其**不治愈提示注入**，只约束受控工具的行为边界。

### 7. 优点
- **安全原则的明确映射**：将经典的混淆代理人、完整中介、最小权限原则清晰落实到LLM Agent语境，概念贡献扎实。
- **可复现性强**：提供完整代码仓库（`scopegate-runtime`），包含一键验证脚本（`run_proof.sh`）和跨框架漏洞验证PoC，审计版本精确到commit hash。
- **贴近真实威胁**：不使用简单的“支付给攻击者”提示，而是采用任务对齐注入手法，更贴近实际攻击场景，测量结果更具说服力。
- **架构设计务实**：ScopeGate被设计为SDK中间件、Sidecar代理或中央网关等多种部署形态，逻辑简单且fail-closed，工程实用性高。

### 8. 不足与局限
- **遏制≠根治**：ScopeGate仅限制工具调用，无法阻止模型生成有害的自然语言输出、无法检测模型权重后门、也无法修复提示注入本身的漏洞。
- **实验覆盖有限**：静态绕过和自适应攻击的样本量及迭代次数有限，不能作为安全无缺陷的证明；良性测试仅10例；Stripe远程MCP服务未审计；多Agent场景下的策略污染等高级威胁未涵盖。
- **测量推断的保守性**：部署层模型分类基于经济性与场景适用性推断，并非供应商实际生产系统的披露；Latam-GPT测试因使用4-bit量化和文本工具协议，结果仅为性能下限。
- **策略配置负担**：ScopeGate的安全性完全依赖运维人员正确配置带外策略（如白名单、金额上限），若配置错误或覆盖不完整，仍可能失效。
