## 摘要
在漏洞数据集中实现可复现性、数量和多样性，长期以来被视为一个内在的三难权衡，改善其中一个维度往往以牺牲其他维度为代价。实践中，可复现性是最常被忽视的维度。这限制了从历史缺陷数据集中自动提取信息的能力，也降低了它们在下游安全研究中的效用。本工作提出了一种方法，通过识别大规模漏洞复现的关键障碍并以通用方案解决，构建了一个能够确保多样化漏洞可大规模复现的新安全数据集。利用该方法，我们将完全可复现性引入最大的开源软件漏洞数据集（OSS-Fuzz），并构建了ARVO数据集（开源软件可复现漏洞图集）。ARVO是一个大规模数据集，包含来自311个项目的6,100多个真实世界漏洞。以可复现性为核心，ARVO与现有数据集的不同之处在于，它提供的每个漏洞都可以在不同版本间一致地重建、触发和分析。可复现性还支持自动识别每个漏洞对应的补丁，并允许在代码变更后直接与漏洞交互，这些能力是现有大规模数据集所不具备的。在我们的评估中，ARVO成功复现了81%的漏洞，并且对定位的补丁达到了89.4%的准确率。我们还讨论了ARVO对上游实践和下游安全研究的影响。

## Abstract
Achieving reproducibility, quantity, and diversity in vulnerability datasets has long been viewed as an inherent three-way trade-off, where improving one dimension often comes at the cost of the others. In practice, reproducibility has been the dimension most often neglected. This has limited what can be automatically extracted from historical bug datasets, and has reduced their utility for downstream security research.   In this work, we propose a method to produce a new security dataset which ensures reproducibility for diverse vulnerabilities at scale by identifying the key obstacles to large-scale bug reproduction and addressing them with general solutions. Using this method, we introduce full reproducibility to the largest open source software vulnerability dataset (OSS-Fuzz) and construct the ARVO dataset (an Atlas of Reproducible Vulnerabilities in Open-source software). ARVO is a large-scale dataset consisting of over 6,100 real-world vulnerabilities across 311 projects. Focusing on reproducibility, ARVO differs from existing datasets by providing each vulnerability in a form that can be consistently rebuilt, triggered, and analyzed across versions. Reproducibility also enables automatic identification of the corresponding patch for each vulnerability and supports direct interaction with vulnerabilities after code changes, capabilities that existing large-scale datasets do not provide. In our evaluation, ARVO successfully reproduces 81% of vulnerabilities and achieves 89.4% accuracy on the located patches. We also discuss ARVO's influence on both upstream practices and downstream security research.

### 论文核心问题与整体含义（研究动机和背景）
在漏洞数据集的构建中，**可复现性、数量与多样性**长期被视为难以兼得的三元对立关系，实践中可复现性最常被牺牲。传统数据集（如 NVD、CVE 相关集）或大规模但无法重建环境、或可复现但覆盖范围极小（如手工制作的 Magma），这严重限制了自动补丁定位、自动化修复评估等下游安全研究的开展。ARVO 的核心目标是**为最大开源漏洞收集平台 OSS-Fuzz 注入完全可复现性，构建一个大规模、真实、多样化且可持续更新的可复现漏洞数据集**，从而打破三难困境，使漏洞不再仅是文本报告，而成为可重建、可触发、可分析的交互式研究资源。

### 论文提出的方法论
ARVO 的核心思想是识别阻碍大规模漏洞复现的关键障碍，并通过通用化、自动化的方案予以解决。主要包含两大部分：**复现器**和**补丁定位器**。

#### 1. 复现器：解决历史版本重建的三大挑战
- **最小侵入式构建插桩（Build Instrumentation）**  
  传统方法（如 OSS-Fuzz 官方方案）强行替换主项目源码，打破了原有构建流程。ARVO 则仅钩子（hook）资源下载类命令，将获取资源的动作与编译动作分离，保持构建过程原生性，避免破坏初始化步骤。

- **全依赖版本控制（Revision Control）**  
  历史漏洞复现失败的主因之一是依赖漂移（dependency drift）。ARVO 不仅回滚主项目到漏洞版本，还根据提交时间戳精确匹配所有依赖库（子模块）的历史版本，确保 API 兼容，解决“不兼容依赖”问题。

- **缺失资源修复（Broken Resource Fixing）**  
  针对源码中引用已失效的外部 URL（如已迁移的代码库）的“比特腐烂”问题，ARVO 维护一个检测–修复循环。它区分核心资源（编译必需）与非核心资源（如语料库），对非核心资源安全跳过，对核心资源通过少量人工维护（每年约数小时）的**可复用修复规则**进行 URL 重定向，使大多数旧漏洞能顺利编译。

复现后，ARVO 通过提供的概念验证（PoC）输入进行验证：漏洞版本必须如期崩溃，修复版本必须不崩溃。只有通过验证的漏洞才会被打包为 Docker 镜像，确保**可复现即正确**。

#### 2. 补丁定位器：自动化精准定位修复提交
利用已实现的可复现能力，ARVO 对 OSS-Fuzz 给出的修复提交范围（通常为多日的数十个提交）执行**提交二分搜索（commit bisect）**。  
- 关键难点在于二分过程中每一步的主项目版本都需要配合相应的依赖版本才能编译通过。ARVO 依据提交时间戳自动选择最兼容的依赖版本，构建可行的中间版本，进而能二分定位出真正消除崩溃的第一个修复提交。  
- 若二分路径上无法构建的提交过多，则降级为线性搜索。

整体框架以上游（如 OSS-Fuzz 的 Issue 数据）为输入，输出包含漏洞与修复版本的预构建 Docker 镜像、触发输入、确认的补丁，以及可重新编译的环境。

### 实验设计
- **数据集与来源**：基于 OSS-Fuzz 的 10,440 个问题，通过标签筛选出 8,921 个安全相关、可复现且已标记为已验证的漏洞作为起点。最终 ARVO 成功复现 6,138 个漏洞，横跨 311 个项目。  
- **基准对比对象**：
  - 官方 OSS-Fuzz 复现器（用于衡量复现率提升）。
  - OSS-Fuzz-OSV 数据集（用于衡量补丁定位准确性）。
- **核心实验**：
  1. **复现率对比实验**：从 OSS-Fuzz 随机抽 100 个漏洞，分别用官方复现器和 ARVO 复现，比较成功率（81% vs 37%），并按项目组件数分组分析。
  2. **补丁准确性评估**：将 ARVO 与 OSS-Fuzz-OSV 给出的补丁进行交叉对比（2,219 个重叠案例），按一致/部分一致/不一致分组，从每组随机抽 100 例人工验证补丁正确性。
  3. **消融研究**：在 100 个样本上依次禁用**全依赖版本控制**、**资源修复**、**基础环境控制**及它们的组合，量化各模块对复现率的贡献。
  4. **漏洞回迁（Backporting）案例**：在 34 个项目上尝试将历史漏洞自动插入到最近的代码版本，验证触发情况和生成模糊测试基准的可行性。
  5. **上游数据质量核查**：通过复现失败案例分析上游误报，确认了 1,519 个虚假漏洞和超过 300 个至今未修复漏洞。

### 资源与算力
- 文明确提及：**重建整个 ARVO 数据集（含补丁定位）耗时约 4 周，使用一台 192 核 CPU、256 GB RAM 的服务器**。整个过程主要为编译密集型任务，未提及 GPU 使用。提供 Docker 镜像后，下游用户使用单条命令即可复现，但自行从零构建仍需大量计算资源。

### 实验数量与充分性
实验设计全面且充分，覆盖了方法自身验证、外部对比、模块贡献度和实际应用验证：
- **复现对比**：1 组（100 个随机样本），提供了组件数分段统计。
- **消融实验**：7 种配置（单特征及组合抑制），在同一 100 样本集上进行。
- **补丁质量评估**：对 3 个类别各 100 个案例的人工检查，并计算了加权总体准确率。
- **大规模补丁统计**：对 6,138 个漏洞的补丁改动文件数、行数等进行了分布统计。
- **应用案例**：34 个项目的漏洞回迁实验。
- **上游纠错**：额外识别了 1,519 个虚假漏洞及 300+ 未修复漏洞，从另一个视角证明复现带来的数据清洗价值。

实验比较对象明确（OSS-Fuzz 官方、OSS-Fuzz-OSV），通过随机抽样和人工验证保证了客观性，消融实验清晰展示了各模块的必要性。

### 论文的主要结论与发现
- **复现率大幅领先**：ARVO 将大规模漏洞复现成功率从 OSS-Fuzz 官方的 **37% 提升至 81%**（提升 44 个百分点），且成功覆盖大量多依赖复杂项目。
- **补丁信息更精确**：ARVO 定位的补丁总体准确率达 **89.4%**，优于 OSS-Fuzz-OSV 的 82.9%，尤其在两者结果不一致时，ARVO 准确率（63%）是对方（32%）的近两倍。
- **有效推动上游修复**：通过复现和验证，ARVO 发现并报告了超 **300 个仍可在最新版本触发的漏洞**，以及超 1,500 个虚假漏洞记录，直接提升了上游 OSS-Fuzz 的数据质量。
- **解锁下游应用**：
  - 补丁分析：量化统计显示 90% 的安全补丁影响 4 个以内文件，63% 的单文件补丁，中位数仅增 6 行删 2 行。
  - 漏洞回迁：实现了跨数十个项目的自动化历史漏洞注入，为模糊测试基准构建提供了可扩展方案。
- **方法论可迁移**：在 Linux 内核漏洞（syzb bot 报告）上的原型实验取得了 78% 的复现率，证明核心思路不局限于 OSS-Fuzz。

### 优点
1. **首次实现大规模+全复现+真实漏洞的三位一体**，打破了长期以来安全数据集“三难”困境。
2. **关键技术深入且系统**：针对性解决了依赖版本控制、缺失资源修复等实践中被长期忽略的编译问题，且方案通用、可维护。
3. **自动化的补丁定位**：利用复现完成提交二分，无需依赖不可靠的提交信息关键字匹配，更精确。
4. **良好的工程化设计**：以 Docker 镜像形式交付，附带编译环境，极大降低了使用门槛，促进了跨领域研究（如机器学习）。
5. **对生态的反馈与闭环**：不仅产出了数据集，还反向纠正上游错误，并已被集成进 OSS-Fuzz 和 DARPA 的 AI 网络安全挑战赛（AIxCC）中，影响力显著。

### 不足与局限
- **高度依赖上游数据**：如果上游（如 OSS-Fuzz）的元数据（如提交范围、标签）存在偏差，可能直接导致补丁定位错误或漏报。
- **漏洞类型偏向**：仅限于有确定性 PoC 并能导致程序崩溃的内存安全漏洞，难以覆盖竞态条件、逻辑漏洞等需要复杂触发或没有明显崩溃信号的漏洞类型。
- **PoC 验证粗粒度**：仅检查崩溃与否，不强制执行完全一致的崩溃类型和地址，可能导致极少数漏洞复现虽“崩溃”但并非原始漏洞。
- **资源消耗巨大**：从零构建数据集耗时数周，依赖 192 核服务器；补丁定位时的连续编译和二分搜索时间成本高。
- **补丁质量局限**：二分逻辑可能将一系列修复提交中的首提交定为 fix，该提交可能不完整或并非最优。约 10% 的补丁被识别为只抑制了崩溃现象而没修复根因，导致少量误报。
- **回迁能力受限**：自动漏洞回迁成功率受补丁逆向应用冲突、版本跨度大导致的编译失败等影响，仍有较大优化空间。
