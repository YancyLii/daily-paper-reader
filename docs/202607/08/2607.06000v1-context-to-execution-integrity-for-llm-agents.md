## 摘要
语言模型智能体读取攻击者可写的上下文以解决任务。工具执行需要对受保护的汇字段、由汇解释的有效载荷以及调用事件进行独立的权限检查。上下文至执行完整性（CXI）是针对此设定的执行边界系统。策略标记受保护的汇字段，类型化释放将窄化的验证值从可写上下文带到特定目的地，不透明数据槽将证据保存为数据，确定性门仅在字段权限、精确效果授权和调用权限都绑定到同一个操作清单时才允许调用。我们在开放权重的字段投影运行、AgentDojo实时情景、代码智能体精确效果基准测试、清单绑定账本故障、提案压力控制以及托管/API兼容性追踪上评估了CXI。AgentDojo覆盖了720个实时情景和1739次LLM调用；代码智能体基准测试覆盖了400个仓库情景，具有精确效果授权和租赁绑定执行，产生了231次安全任务完成，且未观察到字段、效果或调用逃逸。会计报告将解析器结果、授权结果和任务质量结果与准入完整性结果一起报告。在评估的汇中，只有当字段、效果和调用权限都绑定到同一操作清单时，CXI才允许执行。

## Abstract
Language-model agents read attacker-writable context to solve tasks. Tool execution needs a separate authority check for protected sink fields, sink-interpreted payloads, and the invocation event. Context-to-Execution Integrity (CXI) is an execution-boundary system for this setting. Policies mark protected sink fields, typed releases carry narrow validated values from writable context to specific destinations, opaque data slots keep evidence as data, and a deterministic gate admits a call only after field authority, exact-effect authorization, and invocation authority all bind to the same action manifest.   We evaluate CXI on open-weight field-projection runs, AgentDojo live episodes, a code-agent exact-effect benchmark, manifest-bound ledger faults, proposal-pressure controls, and hosted/API compatibility traces. AgentDojo covers 720 live episodes and 1,739 LLM calls; the code-agent benchmark covers 400 repository episodes with exact-effect authorization and lease-bound execution, yielding 231 safe task completions and zero observed field, effect, or invocation escapes. The accounting reports parser outcomes, authorization outcomes, and task-quality outcomes together with the admission-integrity result. Across the evaluated sinks, CXI admits execution only when field, effect, and invocation authority bind to the same action manifest.

## 1. 论文的核心问题与整体含义
- **研究动机**：LLM 智能体在解决任务时常需从外部读取攻击者可写的上下文（如邮件、issue、CI 日志、检索文档等），并将其放入模型上下文中。当智能体依据这些内容生成并执行工具调用时，若缺乏对关键字段、被汇解释的载荷及调用事件的独立权限校验，攻击者便能实现“权限洗钱”：一段本应作为证据的文本获得了选择、授权或参数化特权副作用的能力（例如，用 CI 日志直接指定要执行的操作或审批状态）。
- **整体含义**：本文提出 **上下文至执行完整性（CXI）** 系统，将模型上下文到真实工具执行的最后一步设计为一个受控的执行边界。CXI 强制执行一条原则：只有当受保护的汇字段值、汇解释后的精确效果以及调用事件本身的权限全部绑定到同一个规范操作清单时，该调用才被允许执行。由此，可写上下文仅能通过严格、窄化的授权通道（类型化释放、不透明数据）转化为执行权限。

## 2. 方法论
- **核心思想**：在工具调用的最后一个环节（执行边界）加入一个确定性门控，将权限检查分解为三个必须同时满足的维度：字段权限、效果权限和调用权限，三者必须指向同一个规范化的操作清单（Action Manifest）。
- **关键技术细节与边界对象**：
  - **五类边界状态**：  
    - **T（Trusted state，可信状态）**：任务目标、策略、注册表、审批等可信输入。  
    - **W（Writable context，可写上下文）**：攻击者可写或可影响的输入。  
    - **D（Typed release，类型化释放）**：经可信验证器验证后，从 W 中提取的、仅授权用于特定目标字段的窄化值。  
    - **O（Opaque data，不透明数据）**：仅作为证据保留的 W 文本，在当前边界无任何执行权限。  
    - **X（Protected sink field，受保护汇字段）**：直接选择、授权或参数化外部副作用的汇字段。
  - **三大权限机制**：
    1. **字段权限**：受保护汇字段只能由可信原子或与该字段精确匹配的类型化释放（D）授权；原始的 W 不可直接进入 X。每个 D 都具有严格的类型、源作用域、目标作用域和验证器绑定。
    2. **精确效果授权**：对于补丁、文件体、SQL 等效应承载载荷，适配器在可信快照、主体、策略 epoch 等上下文下计算载荷将被汇解释的精确效果，并生成效果承诺。执行前核验执行效果与授权效果一致，防止载荷重放、基础状态变更等攻击。
    3. **调用权限**：调用事件（包括选择执行、重试、批量、调度等）需要消费一个与当前操作清单绑定的一次性调用能力（Invocation Capability）。该能力在可线性化的账本中被消耗，返回一个执行租约。
  - **准入流水线**：  
    1. 将候选动作规范化为操作清单。  
    2. 对每个受保护字段检查字段局部权限和来源。  
    3. 对每个汇解释载荷检查精确效果承诺。  
    4. 使用清单绑定的调用能力授权调用事件。  
    5. 消费该能力，获得执行租约。  
    6. 仅通过被中介的汇执行。  
    整个门控策略是失败即关闭的：任何权限缺失或证据不匹配都会导致拒绝执行。
  - **来源追踪**：CXI 追踪影响（Influence）的来源而非字节的原始出处。每个受保护字段和调用事件都保存保守的影响集合（Π），推理、摘要、记忆等操作保持 W 的影响，直到被 D 授权。

## 6. 主要结论与发现
- **准入完整性得以实现**：在所有评估的汇上，CXI 仅在字段、效果和调用权限全部绑定到同一个操作清单时才允许执行。所有运行中均未观察到字段、效果或调用逃逸。
- **权限机制不可互换**：通过剥离实验证明，仅模式校验、部分来源跟踪或单独的调用账本均无法阻止特定的权限洗钱攻击，CXI 的字段门、效果授权和清单绑定能力消费必须组合使用。
- **提案压力与执行分离**：模型在不安全提示下仍会大量提出未授权行动（如 196/300 的对抗性字段提案），但这些提案在执行边界被 CXI 全部拦截，未转化为实际副作用。
- **效用与安全可以平衡**：通过类型化释放和不透明数据槽，CXI 能在阻断未授权权限的同时，保留必要的值传递和证据记录功能。100 个工作流审计显示边界效用无错误阻塞。
- **任务质量仍需独立保证**：安全任务完成（231 次）的成功受解析器、效验器完备性和任务本身的正确性影响，CXI 只保证执行的动作权限到位，不保证语义正确。

## 7. 优点
- **精细化权限分解**：将工具调用的安全拆解为字段值、编译效果和调用事件三个正交维度，每个维度有独立的授权对象和检验逻辑，比粗粒度的“全部信任”或“一概拒绝”更精细。
- **明确的执行边界设计**：通过清单绑定的确定性门控，将上下文信息到外部副作用的转化点变成可审计、可强制要求的硬边界，符合最小权限原则。
- **证据驱动与审计友好**：CXI 为每个准入决策生成可验证的证据记录（字段证据、效果承诺、能力消费记录），支持事后审计，并允许不同部署模型（开放权重 vs. 托管 API）使用不同强度的证据。
- **实用的效用通道**：设计了类型化释放和不透明数据两种机制，既保持了有监督的数值流动，又避免了可写文本获得意外权限。
- **评估严格分离指标**：实验设计上将任务成功与安全逃逸、解析失败、授权失败等分开报告，使得安全性质可以被独立验证，避免了“以效用混淆安全”的问题。

## 8. 不足与局限
- **依赖策略正确性与完全中介**：CXI 的保证建立在策略正确标注字段、所有副作用路径都被门控（完全中介）的前提下。如果策略错误或存在未被中介的旁路（如直接文件 API），保证即失效。策略编写需要人工审核且容易出错。
- **托管环境下的证据不完整**：对于托管 API，CXI 无法获取注意力掩码、KV 缓存等内部状态，只能依赖主机端可见的构造证据，无法证明提供商内部的强制情况。远程证明信任模型未解决。
- **不覆盖任务语义正确性**：即使一个动作具有完整权限，其产生的补丁、命令或文本仍可能是错误的或有害的。CXI 不评估任务质量。
- **不透明数据重入风险**：不透明数据不授予权限，但当它进入下游智能体或流程时，会在新边界被视为可写上下文，存在二次利用风险，需每个边界独立防护。
- **效验器完整性分离**：精确效果授权要求适配器效验器完备地捕获所有可能的副作用，这对复杂载荷（如任意代码）可能很困难。效验器自身的正确性是独立的信任假设。
- **实验表面有限**：评估基于固定的人工漏洞场景和基准，在真实世界中更复杂的攻击模式和工具链上可能未全覆盖。
- **成本开销**：虽然掩码构建延迟很小，但托管 API 的往返延迟显著（9~15 秒/案例），且策略编写和维护负担较重。
