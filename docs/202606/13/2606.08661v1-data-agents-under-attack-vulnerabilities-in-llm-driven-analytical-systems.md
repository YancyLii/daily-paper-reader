## 摘要
数据智能体将LLM驱动的推理与关系数据访问、可执行分析工具和多步骤工作流编排相集成，使其在企业分析中越来越核心。这种集成在数据资源、数据库执行和智能体推理方面引入了新的安全漏洞，将数据库安全性和通用LLM智能体安全性的关注点重新组合成故障模式，而这两类研究领域单独都无法捕捉这些模式。为解决这一缺口，我们提出了对数据智能体的系统安全研究。我们的贡献有三方面。首先，我们开发了一个分层漏洞框架，该框架在解释层、执行层和策略层识别了八种数据智能体特有的风险。其次，我们引入了一种按攻击者目标、策略和技术组织的攻击分类法，涵盖三个目标、七种策略和十四种技术，并配合一个基于真实数据库模式的LLM驱动载荷生成管道。第三，我们在六个系统上评估了这些攻击，包括四个开源数据智能体和两个生产云分析服务。我们的实验揭示了当前系统中存在重大安全漏洞，并得出了四个关键要点。

## Abstract
Data agents integrate LLM-driven reasoning with relational data access, executable analytical tools, and multi-step workflow orchestration, making them increasingly central to enterprise analytics. This integration introduces new security vulnerabilities across data resources, database execution, and agent reasoning, recombining concerns from database security and general-purpose LLM-agent security into failure modes that neither line of work captures on its own. To address this gap, we present a systematic security study of data agents. Our contributions are threefold. First, we develop a layered vulnerability framework that identifies eight data agent-specific risks across interpretation, execution, and policy layers. Second, we introduce an attack taxonomy organized by adversary goal, tactic, and technique, covering three goals, seven tactics, and fourteen techniques, and pair it with an LLM-driven payload generation pipeline grounded in real database schemas. Third, we evaluate these attacks on six systems, including four open-source data agents and two production cloud analytics services. Our experiments reveal substantial security vulnerabilities across current systems and yield four key takeaways.

### 1. 核心问题与整体含义

- **研究背景**：数据智能体（Data Agents）通过将大语言模型（LLM）的推理能力与关系数据库访问、可执行分析工具（如SQL、Python）及多步工作流编排相结合，正在成为企业数据分析的核心组件。开源框架（如DataInterpreter、DB‑GPT、DeepAnalyze、LAMBDA）和商业平台（如Databricks Genie、BigQuery）已广泛采用这一范式。
- **安全缺口**：这种集成形成了新的攻击面——数据资源侧（大规模、结构化、含敏感信息的关系数据）、数据库执行侧（高成本查询、无界链式执行）和智能体推理侧（多步上下文、策略遗忘、组合泄露）。现有安全研究将“数据库安全”与“通用LLM智能体安全”相互割裂：前者聚焦SQL注入、访问控制，后者关注提示注入、文档投毒，二者均无法单独刻画数据智能体跨层交互带来的独特失效模式。
- **研究动机**：围绕三个研究问题展开：RQ1（新攻击面暴露了哪些漏洞？）、RQ2（如何系统地将漏洞转化为可利用的攻击技术？）、RQ3（未来数据智能体应如何设计防御？）。目标是通过系统安全研究填补这一空白。

### 2. 方法论

- **分层漏洞框架**：基于数据智能体的系统模型 \(\Sigma = \langle (\mathcal{M}, \mathcal{S}), \mathcal{T}, \mathcal{D}, \Pi, \mathcal{E} \rangle\)（LLM与状态、工具集、数据资产、策略、执行环境），在三个层次识别出**八项漏洞（V1‑V8）**：
  - **解释层**：V1（隐式信任偏差——无优先级规则时的证据冲突选择）；V2（缺乏数据来源验证——将投毒内容当作合法指令）。
  - **执行层**：V3（不受控的查询成本——LLM计划无成本感知）；V4（跨引擎语义不一致——SQL/Python结果差异导致无限调和）；V5（无界多步查询链——无停止条件的深度探索）。
  - **策略层**：V6（上下文压力下的策略遗忘——长会话中策略被挤出上下文窗口）；V7（过度授权的数据库连接——共享服务账户绕过用户级权限）；V8（缺乏组合泄露控制——单独合法的查询结果组合后泄露敏感信息）。
- **攻击分类法**：构建“目标‑策略‑技术”三层层级结构，覆盖**3类攻击目标**（Hijack：越权操作；Mislead：误导输出；Drain：资源耗尽）、**7种策略**和**14项技术**（表3）。每项技术明确对应所利用的漏洞和攻击组件。例如：
  - Hijack：T1.1/T1.2（执行注入）、T2.1/T2.2（推理操纵）、T3.1/T3.2（分析重构）。
  - Mislead：T4.1/T4.2（分析上下文投毒）、T5.1/T5.2（信任偏差利用）。
  - Drain：T6.1/T6.2（执行成本放大）、T7.1/T7.2（搜索空间诱捕）。
- **攻击载荷生成管道**：采用检索增强生成（RAG）方式，将攻击技术说明（知识库）与实际数据库模式（表名、列名、数据类型）结合。每项技术生成**5种机制不同的模板**，每种模板在**5个企业数据库**上实例化，最终产生每技术25个、总计**350个模式锚定的可执行载荷**。

### 3. 实验设计

- **数据集**：采用 **DAComp‑DA**，包含**100个SQLite数据库实例**，覆盖金融、电子商务、数字营销等领域。每个实例原本为良性分析工作负载，实验中按攻击者模型在可写字段、上传文件或多轮提示中植入攻击载荷。
- **被评估系统（共6个）**：
  - 开源：DataInterpreter、DB‑GPT、DeepAnalyze、LAMBDA（底层LLM多为DeepSeek‑V3.2，DeepAnalyze为自家蒸馏模型DeepAnalyze‑8B）。
  - 闭源商业系统：Databricks Genie Code、BigQuery Conversational Analytical Agent（以黑盒方式通过标准接口评估）。
- **评估指标**（按攻击目标度量）：
  - **Hijack → Blast Radius (BR)**：五级序数指标（BR0‑BR4），衡量越权影响的传播范围（从无影响、请求级、会话级、容器级到主机/网络级）。
  - **Mislead → Relative Error (RE)**：\(RE = |\hat{v} - v^*|/v^* \times 100\%\)，比较投毒输出与攻击者目标值。
  - **Drain → Resource Amplification Ratio (RAR)**：\(RAR_m = \hat{c}_m/c_m^{\text{ref}}\)，在token消耗和端到端时间两个维度上测量与良性工作负载的代价放大比。
  - 统一成功率 **Attack Success Rate (ASR)**：为上述指标设定操作上可解释的阈值（BR≥1，RE≤10%，overall RAR≥5），将连续度量转化为二元成功判定。
- **对比分析**：不仅对比不同系统之间的攻击成功率与攻击量级，还进行了**成功率阈值敏感性分析**（变化RE和RAR阈值）和**基座LLM敏感性分析**（在DB‑GPT框架下替换为Sonnet‑4.6、GPT‑5.2、Gemini‑3‑flash等后端），以及评估攻击在闭源商业系统上的泛化能力。

### 4. 资源与算力

- 开源系统实验运行于一台配备 **AMD Ryzen Threadripper PRO 9975WX** 处理器、**576 GB RAM** 和 **1张RTX Pro6000 GPU** 的服务器上。

- 总计生成**350个攻击载荷**，在4个开源系统上各执行全部350个载荷（共1400次运行）；BigQuery因仅支持SQL，测试275个载荷；敏感度分析在DB‑GPT上用4种LLM后端各跑350个载荷（1400次）。全部实验组合合计 **3,075次测试**。
- **充分性**：覆盖面广（多个系统、多种技术、多种后端），包含消融式的阈值变化和LLM替换，以及向商业产品的泛化验证。每个指标有明确的成功判定脚本，开源系统的完整执行记录和载荷模板已公开，保证了可复现性。实验设计客观且充分，可支撑结论的稳健性。

### 6. 主要结论与发现

- **普遍暴露**：所有被评估系统均存在可利用漏洞，没有系统能完全阻断所有攻击技术；每个漏洞至少出现在3个系统中。信任偏差（V1）是最突出的漏洞，多数系统在面对冲突证据时缺乏显式的证据优先级。
- **成功的技术分布**：Hijack中最有效的是代码注入（T1.2，当缺乏沙箱隔离时），Mislead中是数据类型偏差利用（T5.1），Drain中是低价值分支展开（T7.1，大幅增加token消耗）。
- **攻击后果量级**：
  - Hijack：成功攻击大多停留在会话级（BR2），但少数可达到主机级（BR4，如持久化任务）。
  - Mislead：DeepAnalyze在RE≤10%阈值下71%的案例高度接近于零误差，极易被误导。
  - Drain：LAMBDA在T6.1上时间放大率平均46.2倍，部分案例因无查询级取消而耗尽资源。
- **敏感性**：改变ASR阈值不影响定性结论；替换基座LLM仅重新分配风险但无法根除漏洞：某些后端抑制部分技术，但总体攻击面依然存在。
- **四个关键设计要点（Takeaways）**：
  1. **关系数据库扩大了数据智能体的攻击面**：表、元数据、查询结果均可能被解释为指令。
  2. **资源耗尽（Drain）是主要失效模式**：看似合法的分析请求可导致过度计算。
  3. **多步推理使数据泄露更容易**：单独合法的查询组合后可重构受保护记录，需要会话级泄露控制。
  4. **数据智能体没有统一的模态偏好**：SQL证据不一定被优先信赖，冲突解决依赖隐式LLM偏向，应引入显式的证据溯源与优先级管理。

### 7. 优点

- **系统性**：首次将数据智能体作为一个整体进行安全分析，提出了跨层漏洞框架和结构化的攻击分类法，连接了数据库安全与LLM安全。
- **实战导向**：基于真实数据库模式生成可执行载荷，囊括了大量工业系统（含闭源商业产品），使结论贴近生产环境。
- **评估严谨**：多维度指标（BR、RE、RAR、ASR）量化攻击后果，并进行了阈值和基座LLM的敏感性分析，增强了结论的鲁棒性。
- **可复现性**：公开了载荷模板、脚本、完整执行记录，便于社区验证和后续研究。
- **建设性输出**：提炼的四个设计要点直接为未来安全数据智能体的架构提供了系统化指导。

### 8. 不足与局限

- **攻击模型限定**：假设攻击者仅能控制输入的提示和上传数据，不能修改模型权重或DBMS本身，可能未考虑内部威胁的更高级形式。
- **数据库和系统的代表性**：虽然测试了6个系统，但数据集均为SQLite单机场景，未覆盖分布式数据仓库场景下可能的额外攻击面（如跨分片查询混淆）。
- **防御措施未展开验证**：论文停留在攻击评估阶段，对提出的防御建议（如查询成本估算、组合泄露控制）未进行实现和实证验证，其可行性与有效性需进一步研究。
- **商业系统评估受限**：商业平台为黑盒测试，无法获得内部跟踪数据，因此仅报告了ASR，缺少BR、RE、RAR的精确度量，且评估窗口为2026年5月，系统可能已更新。
- **载荷生成的多样性**：每项技术仅生成5种模板，可能未穷尽所有攻击路径；部分技术通过单一表格字段投毒，可能在实际应用中较难触发，但实验仍证明了原理可行性。
