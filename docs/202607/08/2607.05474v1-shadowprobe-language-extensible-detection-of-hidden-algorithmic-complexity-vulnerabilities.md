## 摘要
算法复杂性漏洞（ACVs）在对抗性输入触发最坏情况执行行为时出现，导致严重的性能降级或拒绝服务状况。一个关键但探索不足的来源是影子复杂性：隐藏在看似无害的标准库API内部的非平凡计算成本。由于这些成本在调用点不可见，攻击者可以利用它们引发意外的超线性运行时行为。现有的ACV检测器通常依赖于模糊测试、符号执行或混合分析，但它们通常是特定于语言的，需要大量手动工作来构建测试框架，并依赖于繁重的运行时插桩。我们提出了ShadowProbe，一个可扩展且可跨语言扩展的框架，通过轻量级静态分析、执行上下文的自动重建以及大语言模型（LLM）辅助的测试生成来发现ACV。ShadowProbe使用结构化的多阶段流水线：它根据影子复杂性信号静态筛选候选函数，从项目级符号重构最小可执行上下文，并合成大小受控的输入以探测最坏情况行为。然后它利用执行时间测量和稳健的统计增长推断验证候选者，将真正的算法爆炸与运行时噪声（如垃圾回收和JIT编译效应）区分开来。我们在WISE基准上评估了ShadowProbe，它相比现有方法持续提高了分析效率。我们进一步将其应用于大规模系统，包括CPython、JDK、Zig、Rustc和vLLM，发现了许多以前未知的ACV，其中许多已得到维护者的确认和部分修复。这些结果表明，ShadowProbe能够识别多种真实世界代码库中隐藏的算法风险。

## Abstract
Algorithmic Complexity Vulnerabilities (ACVs) arise when adversarial inputs trigger worst-case execution behavior, causing severe performance degradation or Denial-of-Service conditions. A key but underexplored source is shadow complexity: non-trivial computational costs hidden inside seemingly benign standard library APIs. Because these costs are invisible at call sites, attackers can exploit them to induce unexpected superlinear runtime behavior. Existing ACV detectors often rely on fuzzing, symbolic execution, or hybrid analysis, but they are usually language-specific, require substantial manual effort to construct harnesses, and depend on heavy runtime instrumentation.   We present ShadowProbe, a scalable and language-extensible framework for discovering ACVs through lightweight static analysis, automated reconstruction of execution contexts, and Large Language Model (LLM) assisted test generation. ShadowProbe uses a structured multi-stage pipeline: it statically screens for candidate functions guided by shadow-complexity signals, reconstructs minimal executable contexts from project-level symbols, and synthesizes size-controlled inputs to probe worst-case behavior. It then validates candidates using execution-time measurements and robust statistical growth inference, separating true algorithmic blowups from runtime noise such as garbage collection and JIT compilation effects.   We evaluate ShadowProbe on the WISE benchmark, where it consistently improves analysis efficiency over existing approaches. We further apply it to large-scale systems including CPython, the JDK, Zig, Rustc, and vLLM, uncovering many previously unknown ACVs, many of which have been confirmed and partially remediated by maintainers. These results show that ShadowProbe can identify hidden algorithmic risks across diverse real-world codebases.

  - 影子复杂度（如不可变字符串切片拷贝、线性扫描、正则回溯）经循环或递归放大后，可在攻击者构造的输入下导致**拒绝服务（DoS）**，但这在代码审计中极难通过阅读调用点直接识别。
- **整体含义**：ShadowProbe 旨在提供一种**可扩展、跨语言、自动化**的 ACV 发现框架，通过轻量级静态筛选、执行上下文自动重建、LLM 辅助输入生成和测量驱动的验证，系统性地暴露隐藏在库 API 调用背后的最坏情况时间复杂度行为。

## 2. 论文提出的方法论

ShadowProbe 是一个结构化的四阶段流水线，核心思想是**用静态分析快速圈定候选，用 LLM 构造可执行的最坏情况输入，再用实测时间增长推断复杂度**，同时避免依赖语言特定插桩。

### 2.1 静态分析（Static Analysis）
- 目标：高效筛选可能包含 ACV 的函数，降低动态验证的预算。
- 信号：
  - **嵌套循环与递归**（传统信号）。
  - **影子复杂度 sinks**：预定义的库 API 族（见表 I），其内部成本随输入规模增长但调用点不可见，例如 `substr`、`split`、`contains`、正则匹配、`map` 等。
- 匹配策略：当循环/递归受输入控制，且循环体内调用了上述 sinks，则函数被标记为候选。

### 2.2 上下文恢复（Context Recovery）
- 目标：为候选函数自动构建可编译的依赖上下文，避免人工编写 harness。
- 过程：
  1. 构建项目级符号索引（记录定义、引用、源码范围）。
  2. 从目标函数出发，展开**有界依赖闭包**：解析参数/返回类型、所用符号的定义，并保留必要的封闭声明。
  3. 产出可渲染的代码片段，供后续阶段生成可执行的 Candidate。

### 2.3 最坏情况输入构造（Worst‑Case Input Construction）
- 利用 **LLM** 生成**大小参数化**的输入生成器 `G(n)`，构造执行 harness（包含目标函数的一次调用与时间测量）。
- 关键步骤：
  - LLM 根据目标签名、恢复上下文和 harness 契约，生成多套独立的生成策略。
  - 对合成生成器进行**轻量级尺寸检查**，确认生成的输入规模确实随 `n` 线性增长（Θ(n)）。
  - 在探测执行上测量不同生成器下的运行时间增长，**保留增长最强**的生成器作为最终 Candidate。

### 2.4 动态验证（Dynamic Validation）
- 对已接受的 Candidate，在递增输入大小下执行，收集 `(n, t)` 数据点。
- 采用**自适应对数采样**（Algorithm 1）：在可行区间内优先保持对数均匀采样，提高对幂律增长的拟合效率。
- 每个输入大小执行 **3 次**，取代表值，丢弃超时或失败执行。
- 通过**曲线拟合与增长推断**将复杂度分为低阶（O(n log n) 以下）、多项式、指数、未知四类，同时通过统计方法**分离 GC、JIT 等运行时噪声**。

## 6. 论文的主要结论与发现

- **有效性**：ShadowProbe 在 WISE 上正确分类所有基准的复杂度，分析速度比 SPF‑WCA 快 4.7×，比 Badger 快 8.1×。
- **组件贡献**：
  - **上下文恢复**和**多生成策略**显著提升召回率（FNR 从 47.8% 降至 29.0%，召回从 52.2% 提升至 71.0%）。

- **真实世界发现**：
  - 在 CPython、JDK、Zig、Rustc、vLLM 中**新发现 162 个 ACV**，84 个已获确认，41 个已修复。
  - 漏洞模式主要是**不可变性引起的拷贝放大**（如字符串切片重复拷贝）、**线性成本的列表操作在循环中重复调用**、以及**无记忆化的递归**。
  - 展示了影子复杂度在解析、邮件处理、编译器等成熟模块中长期潜伏，单一底层缺陷可通过多个公开 API 暴露。

## 7. 优点

- **语言可扩展**：核心流水线语言无关，仅需少量语言特定的解析和符号提取（实现 Python、Java、Zig、Rust 合计约 3940 行）。
- **自动化程度高**：自动恢复上下文、自动生成输入、自动验证，大幅减少手工编写 harness 的工作。
- **利用 LLM 但不依赖其推理结果**：LLM 仅用于生成测试输入，最终通过时间测量和统计验证来得出结论，避免模型幻觉直接输出漏洞报告。
- **鲁棒的动态验证**：自适应对数采样和统计推断能区分算法复杂度与运行时噪声（GC/JIT），结果具有较强的可复现性。
- **实用价值**：在多个广泛部署的基础设施项目中发现了真正未知的漏洞，并推动了实质性修复，展示了工具的工业可用性。

## 8. 不足与局限

- **检测范围受限**：静态筛选依赖于循环/递归与预定义的影子复杂度 sinks，可能漏掉无明确循环结构或 sinks 未被收录的 ACV（例如某些递归或隐式迭代样式）。
- **LLM 依赖**：输入生成质量受 LLM 能力影响，对罕见、复杂数据依赖的路径可能生成不成功，需多轮修复；不同模型的性能有波动。
- **测量噪声**：虽然采用统计方法，但墙钟测量仍可能受系统负载、特定运行时行为的影响，极端噪声下仍可能误判。
- **工程拓展成本**：扩展到新语言需实现解析和符号提取，尽管作者声称成本可控，但并非即插即用。
- **真实世界案例确认不完全**：已报告 162 个 ACV，仍有约一半未确认，部分可能被维护者视为性能问题而非安全漏洞，这影响对精确率的评估。
- **无对抗性安全保障**：只检测输入能触发超线性增长的可能性，未评估输入在实际部署中是否真正可控或可攻击，需维护者结合上下文最终判定影响。
